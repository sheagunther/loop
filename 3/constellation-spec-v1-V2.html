Section 1
System Identity

The Butcher Constellation is the order management system for Deer Hill Butchers, a seasonal wild game processing operation in Windham, Maine. Rick Holte (owner, primary operator) and Christine Holte (co-operator, packaging and customer communications) process approximately 100 animals per season — primarily deer brought in by local hunters during the October-through-January hunting season.

The system manages the complete order lifecycle: intake (customer self-service or Rick at the counter), deposit collection, cut selection, pricing, hanging weight recording, stage tracking through nine processing stages, balance collection at pickup, storage fee charging for uncollected orders, no-show abandonment, customer notifications, daily condition scanning, reporting, and configuration management.

The architecture is a Loop MMT constellation: 16 typed loops on four scoped buses, event-driven, closure-enforced capability boundaries. All data flows as typed packets on buses. No loop can acquire capabilities not declared in its factory signature. The closure wall is the security model. The bus assignment table is the routing authority. The routing tables declare every legal data path. The Packet Contract Registry declares every field in every packet.

DD-002 — filter:orderValidation: KEEP (16 Loops)
The TDD (v6, Q3) deferred the decision on whether filter:orderValidation should be absorbed into working:orderIntake. The filter has 16 check loops (4 structural checks across intake and cuts flows). Fix by Design analysis: the filter is a structural enforcement of the validation boundary between untrusted external input and internal processing. The routing table enforces that INTAKE_RAW goes to the filter (§1 #1), and INTAKE_VALIDATED can only come from the filter (§1 #2). Absorbing the filter replaces a bus routing guarantee with a code convention — the Working Loop would need to self-discipline its own validation. Fix by Design says don't do that. The number of checks is irrelevant to the structural analysis. Reversing this decision later is cheap: change the source field on two routing entries. The topology remains 16 loops.
Signal Loop Dependency Gates
Both Signal Loops receive external service access through Dependency Gates — stub-replaceable interfaces injected via the services factory capability. signal:payments wraps the Stripe API; signal:notifications wraps Twilio (SMS) and an email provider. Each gate provides: the service interface, a circuit breaker (detects repeated failures, emits SERVICE_DEGRADED), and service restoration detection (emits SERVICE_RESTORED when connectivity returns). The gates also support Translator patterns for mapping between the constellation's packet format and the external API's request/response format.

Priority Levels
Three priority levels declared in the routing tables: CRITICAL (payment failures when customer is present — BALANCE_FAILED, STORAGE_FEE_FAILED at counter), STANDARD (normal business events and commands — the default), and LOW (informational events with no time sensitivity). Priority affects dashboard display ordering and notification urgency, not bus delivery order. All packets are delivered in emission order within a bus.
Cross-Bus Pattern
SIGNAL bus traffic flows one direction: Workflow → Signal Loop. Results flow back on DATA, not SIGNAL. This means the Workflow Loop subscribes on DATA for payment and notification results, not on SIGNAL. The Signal Loop publishes results to DATA so the dashboard and other subscribers can observe them. This is standard Loop MMT — the SIGNAL bus carries commands to external services, DATA carries the results back into the constellation.
Section 4
Pipeline Declarations

14 pipelines across two Workflow Loops. Each pipeline declares its trigger, step sequence with emit/expect pairs, and per-step failure strategy (Abort, Skip, or Retry with max 3 attempts and exponential backoff). Absorbed from Pipeline Spec — workflow:orderLifecycle v2 and Pipeline Spec — workflow:notifications v2, which are superseded by this section.

workflow:orderLifecycle — 8 Pipelines, 43 Steps

Owns the order state machine. Coordinates multi-step operations through a consistent pattern: fetch → validate → write → confirm → emit semantic event → audit. Every vault write is confirmed before any semantic business event is emitted. Payment failures never block business operations.

PIPELINE_ORDER_INTAKE

Trigger: INTAKE_VALIDATED (§1 #2) or INTAKE_DIRECT (§1 #4). Creates a new order from external or internal intake. 8 steps: fetch customer → conditional create customer (fire-and-forget, Skip) → write order (Abort on fail) → conditional charge deposit (Skip — deposit waiver check; failure does not block) → audit → conditional request pricing (only if cuts present) → write pricing (Retry) → check Ready to Butcher gate.

Key decisions: Dual trigger — same pipeline, different audit trail (external vs. internal origin). Deposit before pricing is deliberate (flat $75, no dependency on pricing engine). CUSTOMER_CREATE_FAILED (§1 #11) was added during Session 10 renumbering, resolving the OP-009 pattern violation. Customer creation remains fire-and-forget by design (Skip on failure), but the failure path now has a routing entry.

PIPELINE_LATE_CUTS

Trigger: CUTS_VALIDATED (§1 #28) or CUTS_DIRECT (§1 #30). Customer completes cut selections after partial intake. 5 steps: write cuts (Abort) → request pricing (Skip) → write pricing (Retry) → check RTB gate → audit.

PIPELINE_STAGE_ADVANCE

Trigger: STAGE_ADVANCE_COMMAND (§2 #31). Manual stage advance by Rick or Christine. 5 steps: fetch order (Abort) → validate transition (Abort — emits STAGE_ADVANCE_REJECTED §2 #34 with reason) → write stage change (Retry) → emit ORDER_STAGE_CHANGED (Skip) → audit. Gate conditions apply only to Ready to Butcher (hanging weight + pricing). Closed cannot be targeted manually — auto-triggered only.

PIPELINE_WEIGHT_RECORDING

Trigger: WEIGHT_RECORDED (§2 #35). Rick records hanging weight. 4 steps: fetch order (Abort) → write weight (Retry) → audit (unconditional) → check RTB gate (may auto-advance). If gate opens, two distinct audit entries: weight recording and stage transition.

PIPELINE_CANCELLATION

Trigger: CANCEL_ORDER (§2 #36). Only valid before butchering begins (Checked In, Pending Customer Input, Hanging, Ready to Butcher). 5 steps: fetch order (Abort) → validate eligibility (Abort — emits CANCEL_REJECTED §2 #38) → write cancellation (Retry) → emit ORDER_CANCELLED (Skip) → audit. Full deposit forfeit, no refund processing.

PIPELINE_PICKUP_CLOSE

Trigger: ORDER_STAGE_CHANGED (§2 #33) — self-subscribe with match { newStage: 'Picked Up' }. Auto-charges remaining balance and closes order. 6 steps: fetch order (Abort) → conditional charge balance (Skip — if balance ≤ 0, skip) → audit charge → write Closed (Retry) → emit ORDER_STAGE_CHANGED with newStage=Closed (Skip) → audit close. Self-subscribe is safe: match pattern filters for Picked Up only, emit produces Closed — no recursion.

Key decision: Charge completes before close — not succeeds. Both BALANCE_CHARGED and BALANCE_FAILED are completion. The close proceeds regardless. The thank-you notification uses the charge outcome from the payload.

PIPELINE_STORAGE_FEE

Trigger: STORAGE_FEE_DUE (§5 #67). Daily scan identifies overdue storage fee. 5 steps: fetch order (Abort) → verify fee due via lastStorageFeeChargedAt dedup (Abort if already charged — expected behavior) → charge storage fee (Skip) → conditional write charge timestamp (Retry — only if charge succeeded; if charge failed, timestamp must NOT update or next scan will skip this order) → audit.

Dedup is structural: the order record's lastStorageFeeChargedAt timestamp is the single source of truth. The scan checks it, the pipeline writes it on success only.

PIPELINE_ABANDONMENT

Trigger: NO_SHOW_DEADLINE_REACHED (§5 #68). Daily scan identifies order past no-show deadline (default 30 days after Ready for Pickup). 5 steps: fetch order (Abort) → conditional charge no-show balance (Skip — if balance ≤ 0, skip; failure does not prevent abandonment) → write Abandoned (Retry) → emit ORDER_ABANDONED (Skip) → audit. Terminal before recurring: daily scan checks no-show before storage fees per order.

workflow:orderLifecycle — Design Principles

Vault confirms writes, workflow declares meaning. Every pipeline that writes to the vault waits for ORDER_UPDATED before emitting the semantic business event. The vault is the authority; the workflow is the narrator.

Payment failures never block business operations. Deposit failure doesn't block order creation. Balance failure doesn't block pickup close. No-show charge failure doesn't block abandonment. The system flags failures; Rick handles them in person.

Ready to Butcher gate is checked in three pipelines. PIPELINE_ORDER_INTAKE (step 8), PIPELINE_LATE_CUTS (step 4), PIPELINE_WEIGHT_RECORDING (step 4). Two inputs: weight + pricing. Either can arrive first. The gate checks both conditions every time.

Retry limit: 3 attempts with exponential backoff. Declared in every Retry step. Retries exhaust → escalate to operator. Retry count and backoff strategy are specification, not implementation detail.

workflow:notifications — 6 Pipelines, 26 Steps

Owns all external customer notifications. Fire-and-forget sending — delivery results arrive asynchronously via NOTIFICATION_DELIVERED or NOTIFICATION_FAILED and are handled by dedicated pipelines. Retry logic is centralized, not duplicated across sending pipelines.

PIPELINE_AUTO_NOTIFICATION

Trigger: Match set — ORDER_CREATED (partial intake only, status=Pending Customer Input), ORDER_STAGE_CHANGED (Ready for Pickup and Closed only), ORDER_CANCELLED, ORDER_ABANDONED. 6 steps: fetch customer (Abort) → check consent (Abort — no opted-in channel = suppressed notification, audited) → fetch order (Skip — trigger payload has enough for basic message) → fetch template (Skip — hardcoded fallback exists) → assemble and send via SEND_NOTIFICATION (fire-and-forget, Skip) → audit.

Five business events, one assembly pipeline. Trigger type determines template and payload fields; mechanical flow is identical. Cancellation and abandonment notices are configurable — template can be set to "disabled" in vault:config.

PIPELINE_MANUAL_MESSAGE

Trigger: SEND_MANUAL_MESSAGE (§4 #60). Operator sends custom message from dashboard. 3 steps: fetch customer (Abort) → assemble and send (Skip) → audit. No consent check — operator intent overrides (TCPA enforcement remains at signal:notifications level).

PIPELINE_MARKETING_OUTREACH

Trigger: MARKETING_OUTREACH_DUE (§5 #72). Pre-season automated outreach. 4 steps: fetch campaign config (Abort — no partial sends from unknown template) → fetch eligible customers batch with filter marketingOptIn=true, hasCompletedOrder=true (Abort) → send batch (per-customer: fetch customer for current info, check consent, assemble, send; individual failures isolated, Skip) → audit batch completion.

Dedup is in compute:dailyScan, not here. Campaign executedAt timestamp prevents re-triggering. Mid-batch failure + re-trigger = some customers get duplicate marketing message. Acceptable tradeoff for a small butcher shop.

PIPELINE_INCOMPLETE_REMINDER

Trigger: INCOMPLETE_REMINDER_DUE (§5 #70). Orders in Pending Customer Input past reminder interval. 7 steps: fetch order (Abort) → check dedup via lastReminderSentAt (Abort — expected behavior if already sent recently) → fetch customer (Abort) → check consent (Abort — suppression audited) → fetch template (Skip — fallback available) → assemble and send with self-service link (Skip) → audit.

PIPELINE_DELIVERY_RETRY

Trigger: NOTIFICATION_FAILED (§4 #58). 4 steps: check retry budget (3 paths: retry same channel, fallback to alternate channel, or escalate permanent failure) → retry send (Skip) OR escalate permanent failure via NOTIFICATION_PERMANENTLY_FAILED §4 #59 (Retry — Rick must see this action item) → audit attempt.

Retry schedule: 1 min → 5 min → 15 min → 1 hour. Four attempts per channel, then fallback. Both channels exhausted → permanent failure. Non-retryable failures (invalid phone, deactivated email) bypass retry budget entirely.

PIPELINE_DELIVERY_CONFIRM

Trigger: NOTIFICATION_DELIVERED (§4 #57). 2 steps: audit delivery (Skip) → conditional write lastReminderSentAt to order record (Retry — only for reminder notifications; dedup timestamp is critical).

workflow:notifications — Design Principles

Fire-and-forget send, separate delivery pipelines. Sending pipelines (1–4) emit SEND_NOTIFICATION and move on. Delivery results are handled by dedicated pipelines (5–6). Sending is fast; retry logic is centralized.

Consent check is in the workflow, not the signal loop. Auto-notifications and reminders check consent before sending. Manual messages do not — operator intent overrides. Marketing uses marketingOptIn filter in batch pull. Signal Loop enforces TCPA at delivery level regardless — legal backstop.

Template fallback is fix by design. Every notification type has a hardcoded fallback template. Config mistakes cannot produce blank messages. The fallback exists in code, not in config.

Notifications never block business operations. A permanent notification failure does not reverse a stage transition, cancel an order, or block a payment. Rick or Christine can re-send manually or call the customer.

Self-Service Link Auth Model

Customer self-service intake uses bearer token auth, not session auth. The selfServiceLinkId in the order record is a UUID v4 token embedded in the URL. Possession equals access. Section 8
Data Gate Policies

Three gate types per the Standard: Transit Gate (buses — what happens to a packet after delivery), Retention Gate (vaults — data lifecycle), Workflow Gate (transaction logs — crash recovery retention). Absorbed from Data Gate Policies V2, which is superseded by this section.

Transit Gate — Per Bus

Three policies: consume (destroy after delivery — the default), archive (retain copy in circular Archive Buffer), hold (keep until acknowledged — not used in V1). Policies declared per functional category.

DATA bus (15 loops): Default consume. Three exception categories archived: failure events (11 types including payment failures — ORDER_CREATE_FAILED, ORDER_UPDATE_FAILED, DEPOSIT_FAILED, BALANCE_FAILED, STORAGE_FEE_FAILED, NOSHOW_BALANCE_FAILED, CONFIG_UPDATE_FAILED, CUSTOMER_UPDATE_FAILED, SNAPSHOT_STORE_FAILED, NOTIFICATION_PERMANENTLY_FAILED, PAYMENT_ALERT), service degradation (4 types — *_DEGRADED, *_RESTORED), rejection events (3 types — STAGE_ADVANCE_REJECTED, CANCEL_REJECTED, WEBHOOK_REJECTED). ~43 packet types consumed, ~18 archived.

VAULT bus (10 loops): Pull requests consume. Write commands archive — every mutation request is retained in the archive buffer. At ~100 orders/season with ~8 writes per lifecycle, ~800 write packets per season. This is the single most valuable diagnostic tool in the constellation.

SIGNAL bus (4 loops): All traffic archive, no exceptions. Every external request (Stripe charges, Twilio messages) is retained. Essential for dispute resolution and payment reconciliation. Volume is low — a few hundred per season.

SYNC bus: Default consume. No V1 traffic.

Background Sweep

Retention Gate transitions are executed by a daily background sweep. Runs after the daily snapshot completes (triggered by RETENTION_SWEEP_TRIGGERED from the Clock, after SNAPSHOT_STORED confirms the day's snapshot). The sweep is idempotent. Every transition is logged to the audit trail. Purge operations are logged before execution. At ~100 orders/season, the full sweep processes a few hundred records — no batching needed.

Handling Rules

Encrypt at rest: All PII fields in vault records are encrypted at the database level. Supabase provides column-level encryption for designated fields.

Mask in logs: PII fields are masked (replaced with category tag + field hash) in application logs. Audit records in vault:snapshots retain full values (they are the authorized PII store, not a log leak).

Purge on deletion: When an order or customer record is purged per Retention Gate policy, all PII fields are destroyed. Audit records documenting the purge persist (they record that a purge happened, not the purged data).

Exclude from snapshots (selective): Daily snapshots include PII (they are point-in-time records for tax and audit purposes). On-demand reports mask PII unless the requesting operator has owner role.

Retention alignment: PII retention follows the Retention Gate policies in Section 8. TCPA consent records (CONSENT category) must survive the customer record's active→archive transition — the 7-year archive retention for vault:customers already exceeds the 5-year TCPA minimum.

Append-only audit constraint: vault:snapshots is append-only. PII that enters via WRITE_AUDIT never leaves during the active retention window. Deleting PII from a vault record does not remove it from existing snapshots. This is a known architectural constraint — append-only audit trails and data deletion requests are structurally at odds. V1 accepts this constraint; full cross-domain erasure is a V2 enhancement.

Section 10
Deferred Decision Resolutions

Decisions deferred during Phase 0 that are resolved at assembly time. Each resolution states the decision, the rationale, and any downstream effects.

DD-002 — filter:orderValidation Absorption

Decision: KEEP. 16 loops. The filter remains as a structural enforcement of the validation boundary between untrusted external input and internal processing. Fix by Design: the routing table enforces that INTAKE_RAW goes to the filter (§1 #1) and INTAKE_VALIDATED can only come from the filter (§1 #2). Absorbing the filter replaces a bus routing guarantee with a code convention. The number of structural checks (4 checks, 16 loops through intake + cuts paths) is irrelevant — the architectural value is the boundary enforcement, not the check count. Reversing later is cheap: change the source field on two routing entries. Creating the filter later requires the operator to recognize when the code convention has become unreliable — harder to detect than a structural violation.

Downstream effects: None. Topology remains 16 loops. No routing table changes required.

RETENTION_SWEEP_TRIGGERED Routing Entry

Decision: Declare as a Clock-emitted schedule. The retention sweep is triggered by the Clock after the daily snapshot window. RETENTION_SWEEP_TRIGGERED follows the same path as other Clock events: Clock → working:orderIntake → DATA. Consumer is determined at Phase 1 implementation — either a dedicated sweep handler in an existing Compute Loop, or the Vault Loops process the sweep directly. The routing entry pattern is: working:orderIntake emits RETENTION_SWEEP_TRIGGERED onto DATA, consumed by the sweep processor.

Downstream effects: One new routing entry needed. The RETENTION_SWEEP_TRIGGERED packet type needs a PCR entry. The Clock schedule is declared in §5. These are Phase 1 tasks — the Constellation Spec declares the decision, the affected reference documents are updated before implementation begins.

The PCR reuse entry for WRITE_ORDER_UPDATE already accounts for this — no PCR change needed.

Routing Summary

Compact reference: 95 entries across 6 routing table sections. One row per entry. No subscriber lists, no priority, no mode, no prose — those belong to the authoritative routing tables. This table exists so a reader can see the full packet flow at a glance without opening six separate files. If this table and a routing table disagree, the routing table wins.

Documented Duplication
This is the one place where the Fix by Design no-duplication principle is deliberately relaxed. The routing summary duplicates entry → packet type → source → destination from the authoritative routing tables. The tradeoff: without it, the Constellation Spec cannot convey how the system works without loading six additional files. The summary is structurally limited (no subscriber lists, no priority, no prose, no field contracts) to minimize the surface area for drift. The risk is accepted because: (a) the summary is compact and mechanically verifiable, (b) the routing tables are stable (no changes since Session 10 renumbering), and (c) the routing tables are declared as authoritative.
