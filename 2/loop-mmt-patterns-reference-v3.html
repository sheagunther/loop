About This Document

The complete implementation guide for all thirty Loop MMT™ patterns. Each pattern includes: what it is, when to use it, how it works, a complete code example following Loop MMT conventions, implementation constraints, and related patterns. This document provides enough information for an AI to implement any pattern in a constellation build.

Patterns are organized into six categories: Core (architectural patterns that constrain constellation design), Data Flow, Workflow, Resilience, Infrastructure, and Scale. Each code example uses the Loop MMT factory pattern, bus API, packet contracts, and prefix taxonomy defined in the Coding Standards.

On pattern provenance. The thirty patterns fall into three groups. The majority are derived from Loop 2.1 hardware — twenty-four have direct L21 analogs, and the closing note in this document discusses that lineage. A smaller set are patterns where Loop MMT and established software engineering independently arrived at the same shape: the Projection converges with the CQRS read model, Split-Merge with Fork-Join, Replay with Event Sourcing. These convergences are noted inline and reflect that the underlying problems are real and recurring — not that any source copied from another. Finally, three patterns are adopted directly by name from prior software engineering practice: Debounce (from electronics and the JavaScript/UI ecosystem), Dead Letter (from Enterprise Integration Patterns and modern message brokers), and Outbox (from distributed systems reliability literature). Loop MMT does not claim to have invented these. It adopts them because they solve real problems well and fit cleanly within the architecture.

1. The Echo
L21: The operator observes the ALU result before routing it further — the computation is confirmed before it is used.
What It Is

A loop never trusts that an operation succeeded until the responsible loop confirms it by emitting a completion packet. The UI does not show "Order Confirmed" when the user taps Submit — it shows it when the Vault emits ORDER_CREATED. The difference is the difference between hope and proof.

The echo packet can carry metadata about the confirmation — persisted: true vs. persisted: buffered — so downstream loops and the UI know the real status. The Workflow Loop is built entirely on Echoes: emit a step request, wait for the echo, then proceed.

When to Use It

Every time a loop requests an action from another loop. Every Vault write. Every external service call. Every workflow step. If the action matters, wait for the echo. If you are tempted to show success before the echo arrives, you are showing hope, not proof.

How It Works

The requesting loop emits a command packet (e.g., CREATE_ORDER). The responsible loop processes it. On success, the responsible loop emits a confirmation packet (e.g., ORDER_CREATED) with the result in the payload. On failure, it emits an error packet. The requesting loop (or the Workflow engine, or the Presentation Loop) subscribes to both the success and error types.

Code Example
// ── The Echo: Vault Loop confirms a write ──

// Vault Loop — processes the command, emits the echo
function createVaultOrders({ bus, db }) {
  bus.vault.on('CREATE_ORDER', async (packet) => {
    try {
      const record = await db.from('orders').insert(packet.payload);
      bus.vault.emit({
        type: 'ORDER_CREATED',
        marker: 'live',
        payload: { ...record, persisted: true },
        source: 'vault:orders',
      });
    } catch (err) {
      bus.data.emit({
        type: 'LOOP_ERROR',
        payload: { loop: 'vault:orders', operation: 'CREATE_ORDER',
                   error: err.message },
        source: 'vault:orders',
      });
    }
  });
}

// Presentation Loop — waits for the echo, not the request
function createPresentationOrders({ bus, document }) {
  function h_onSubmitOrder(formData) {
    // Show "processing..." — NOT "order confirmed"
    v_renderOrderStatus('processing');

    bus.data.emit({
      type: 'INTAKE_RAW',
      payload: { subtype: 'create_order', ...formData },
      source: 'presentation:orders',
    });
  }

  // The echo — now we can show confirmation
  bus.vault.on('ORDER_CREATED', (packet) => {
    v_renderOrderStatus('confirmed', packet.payload);
  });

  bus.data.on('LOOP_ERROR', (packet) => {
    if (packet.payload.operation === 'CREATE_ORDER') {
      v_renderOrderStatus('error', packet.payload.error);
    }
  });
}
Constraints

Never show a success state before the echo arrives. Never assume the echo will arrive — always handle the error case. The echo must carry enough information for the subscriber to act on it without pulling additional data (include the created record, not just the ID).

Related: Marker (echos carry markers), Projection (subscribes to echoes to update derived views)
2. The Marker
L21: The marker bit (bit 16) distinguishes a live word from empty space. A word with marker 0 is vacant; marker 1 is live data.
What It Is

Every packet carries a liveness indicator. marker: 'live' is real data. marker: 'empty' is a deliberate null signal — "I looked and there is nothing here." marker: 'tombstone' is a deletion notice — "this record existed and was deliberately removed."

The distinction matters because "no data returned" is ambiguous (query failed? genuinely empty?), while a marked empty response is unambiguous. Pull requests must always return a marked packet, never silence.

When to Use It

Every pull response. Every Vault read. Any situation where "no result" is a valid, meaningful answer that downstream loops need to distinguish from "the operation failed." If a customer lookup returns nothing, the downstream loop needs to know: did the customer not exist (empty), was the record deleted (tombstone), or did the database fail (error)?

How It Works

The Vault Loop checks the result of the database query. If records are returned, it emits with marker: 'live'. If the query succeeded but returned no rows, it emits with marker: 'empty'. If a record was found but its status is archived/purged, it emits with marker: 'tombstone'. The consuming loop inspects the marker before processing the payload.

Code Example
// ── The Marker: Vault Loop marks every response ──

function createVaultCustomers({ bus, db }) {
  bus.vault.on('FETCH_CUSTOMER', async (packet) => {
    try {
      const { data, error } = await db
        .from('customers')
        .select('*')
        .eq('id', packet.payload.customerId)
        .single();

      if (error && error.code === 'PGRST116') {
        // No rows returned — genuinely empty
        bus.vault.emit({
          type: 'CUSTOMER_FETCHED',
          marker: 'empty',
          payload: { customerId: packet.payload.customerId },
          source: 'vault:customers',
        });
        return;
      }
      if (error) throw error;

      if (data.status === 'archived') {
        bus.vault.emit({
          type: 'CUSTOMER_FETCHED',
          marker: 'tombstone',
          payload: { customerId: data.id, archivedAt: data.archived_at },
          source: 'vault:customers',
        });
        return;
      }

      bus.vault.emit({
        type: 'CUSTOMER_FETCHED',
        marker: 'live',
        payload: data,
        source: 'vault:customers',
      });
    } catch (err) {
      bus.data.emit({
        type: 'LOOP_ERROR',
        payload: { loop: 'vault:customers', operation: 'FETCH_CUSTOMER',
                   error: err.message },
        source: 'vault:customers',
      });
    }
  });
}

// ── Consumer checks the marker ──
bus.vault.on('CUSTOMER_FETCHED', (packet) => {
  switch (packet.marker) {
    case 'live':
      f_processCustomer(packet.payload);
      break;
    case 'empty':
      f_handleCustomerNotFound(packet.payload.customerId);
      break;
    case 'tombstone':
      f_handleDeletedCustomer(packet.payload.customerId);
      break;
  }
});
Constraints

A pull response must never be silence — always emit a marked packet, even when there's nothing to return. The empty marker is not an error — it's a valid result that means "I successfully looked and found nothing." The tombstone marker is not an error either — it means the record existed and was deliberately removed, which may require different handling than "never existed."

Related: Echo (echos carry markers), Retention Gate (produces tombstones)
3. The Hydration
L21: The operator loads a value from memory into the ALU loop, enriching the workspace before computation.
What It Is

A Filter Loop that receives a sparse packet (just an ID), pulls full data from a Vault, and attaches it to the packet before passing it along. The enriched packet now carries everything downstream loops need without them having to make their own Vault pulls.

When to Use It

When a Working Loop receives an intake that contains just a reference (a customer ID, an order ID) and downstream Compute or Filter Loops need the full record to do their work. Rather than every downstream loop independently pulling the same data, one Hydration Filter enriches the packet once.

How It Works

Implemented as a Filter Loop. Receives bus only. Subscribes to the sparse packet type. Makes a pull request to the Vault Bus. Attaches the response to the packet. Emits the enriched packet under a new type (or the same type with an enrichment flag). The pull goes through the Vault Bus — this is a data retrieval, not a business computation.

Code Example
// ── The Hydration: enrich an order packet with customer data ──

function createFilterCustomerHydration({ bus }) {
  bus.data.on('ORDER_INTAKE_VALIDATED', async (packet) => {
    // Pull full customer data from the Vault
    const customerResponse = await bus.vault.pull({
      type: 'FETCH_CUSTOMER',
      payload: { customerId: packet.payload.customerId },
    });

    if (customerResponse.marker !== 'live') {
      bus.data.emit({
        type: 'HYDRATION_FAILED',
        payload: {
          reason: 'customer_not_found',
          customerId: packet.payload.customerId,
          originalPacket: packet.type,
        },
        source: 'filter:customerHydration',
      });
      return;
    }

    // Emit enriched packet — downstream loops get everything
    bus.data.emit({
      type: 'ORDER_HYDRATED',
      marker: 'live',
      payload: {
        ...packet.payload,
        customer: customerResponse.payload,
      },
      source: 'filter:customerHydration',
    });
  });
}
Constraints

Hydration is always done by a Filter Loop — it enriches, it does not transform. The pull goes through the Vault Bus. If the hydration pull returns empty or tombstone, the Filter must handle it — either emit a failure packet or emit the original packet with a flag indicating hydration was incomplete. Never silently drop a packet because hydration failed.

Related: Marker (hydration checks the marker on the pull response), Cascade (hydration is often the first step in a filter cascade)
4. The Cascade
L21: PM1 → PM2 → TG1 → TG2. A four-stage conditional pipeline on the Big Loop where each stage filters the output of the previous one, all in a single rotation.
What It Is

Chaining Filter Loops on the bus. Packet enters Filter A, passes or rejects. Passes go to Filter B. Passes go to Filter C. Each filter is independent, has its own contract, and can be added or removed without changing the others. The sequence is defined in the routing table, not in the filters. Adding a new filter to the cascade means updating the routing table — no existing code changes.

When to Use It

When incoming data needs multiple independent checks or transformations: validation, then authorization, then hydration, then rate-checking. Each step is a separate concern. The Cascade is the composition mechanism — you build the pipeline by composing independent filters, not by building one monolithic filter that does everything.

How It Works

Each Filter Loop subscribes to a specific input type and emits a specific output type. The routing table chains them: Filter A's output type is Filter B's input type. The filters themselves do not know about each other. The routing table is the composition layer. To add a new step: write the Filter, add routing entries, done. To remove a step: remove the routing entries, wire the predecessor's output to the successor's input.

Code Example
// ── The Cascade: three independent filters chained by routing ──

// Filter 1: Schema validation
function createFilterOrderValidation({ bus }) {
  bus.data.on('ORDER_INTAKE_RAW', (packet) => {
    const errors = f_validateOrderSchema(packet.payload);
    if (errors.length > 0) {
      bus.data.emit({
        type: 'VALIDATION_REJECTED',
        payload: { errors, originalPacket: packet },
        source: 'filter:orderValidation',
      });
      return;
    }
    bus.data.emit({
      type: 'ORDER_INTAKE_VALIDATED',
      marker: 'live',
      payload: packet.payload,
      source: 'filter:orderValidation',
    });
  });
}

// Filter 2: Authorization check
function createFilterOrderAuth({ bus }) {
  bus.data.on('ORDER_INTAKE_VALIDATED', (packet) => {
    if (!f_checkPermission(packet.operatorId, 'create_order')) {
      bus.data.emit({
        type: 'AUTH_REJECTED',
        payload: { action: 'create_order', operatorId: packet.operatorId },
        source: 'filter:orderAuth',
      });
      return;
    }
    bus.data.emit({
      type: 'ORDER_INTAKE_AUTHORIZED',
      marker: 'live',
      payload: packet.payload,
      source: 'filter:orderAuth',
    });
  });
}

// Filter 3: Customer hydration
function createFilterCustomerHydration({ bus }) {
  bus.data.on('ORDER_INTAKE_AUTHORIZED', async (packet) => {
    const customer = await bus.vault.pull({
      type: 'FETCH_CUSTOMER',
      payload: { customerId: packet.payload.customerId },
    });
    if (customer.marker !== 'live') { /* handle missing */ return; }
    bus.data.emit({
      type: 'ORDER_HYDRATED',
      marker: 'live',
      payload: { ...packet.payload, customer: customer.payload },
      source: 'filter:customerHydration',
    });
  });
}

// ── Routing table entries that define the cascade ──
// ORDER_INTAKE_RAW      → filter:orderValidation
// ORDER_INTAKE_VALIDATED → filter:orderAuth
// ORDER_INTAKE_AUTHORIZED→ filter:customerHydration
// ORDER_HYDRATED        → compute:pricing (next stage)
Constraints

Filters in a cascade must not depend on each other's internals. Each filter gets a packet, does its job, emits the result. If you find yourself needing Filter B to know what Filter A decided, that information should travel in the packet payload or as a stamp — not through a side channel. The routing table order is the only coupling.

Related: Reorder (change cascade order by changing routing), Stamp (filters can stamp packets for downstream use), Hydration (commonly a cascade step)
5. The Projection
L21: The operator reads from multiple sources (ALU result, comparator flags, counter value) to form a complete picture before making a routing decision.
What It Is

A Compute Loop subscribes to change events from multiple Vaults and maintains a derived, denormalized view. When the Order Vault emits ORDER_UPDATED, the Projection Loop pulls related customer, status, and media data, assembles the complete view, and emits PROJECTION_UPDATED. The Presentation Loop subscribes to the projection, not to the individual Vaults — one subscription, one packet, all the data needed to render.

The projection is explicitly a derived cache, not a source of truth. It can be rebuilt from the Vaults at any time.

When to Use It

When the Presentation Loop needs data from multiple Vaults to render a view. Rather than the UI making five separate pull requests and assembling the data itself, the Projection Loop does the assembly once and emits a complete view packet. Also useful when multiple Compute Loops need the same assembled data — they all subscribe to the projection rather than each assembling independently.

How It Works

Implemented as a Compute Loop. Subscribes to change events from relevant Vaults (ORDER_CREATED, ORDER_UPDATED, CUSTOMER_UPDATED). On any change, pulls current data from all relevant sources, assembles the combined view, and emits a projection packet. The Presentation Loop subscribes to the projection type. The projection can be rebuilt from scratch by pulling all source data — useful after a crash or on startup.

Code Example
// ── The Projection: assembled order view for the UI ──

function createComputeOrderProjection({ bus }) {
  // Subscribe to every event that could change the projection
  const v_triggerTypes = [
    'ORDER_CREATED', 'ORDER_UPDATED', 'ORDER_ARCHIVED',
    'CUSTOMER_UPDATED', 'STATUS_CHANGE',
  ];

  for (const type of v_triggerTypes) {
    bus.vault.on(type, (packet) => {
      f_rebuildProjection(packet.payload.orderId || packet.payload.id);
    });
  }

  async function f_rebuildProjection(orderId) {
    // Pull current state from all sources
    const order = await bus.vault.pull({
      type: 'FETCH_ORDER', payload: { orderId },
    });
    if (order.marker !== 'live') return;

    const customer = await bus.vault.pull({
      type: 'FETCH_CUSTOMER',
      payload: { customerId: order.payload.customerId },
    });

    const statusHistory = await bus.vault.pull({
      type: 'FETCH_STATUS_HISTORY',
      payload: { orderId },
    });

    // Assemble the complete view
    bus.data.emit({
      type: 'ORDER_PROJECTION_UPDATED',
      marker: 'live',
      payload: {
        order: order.payload,
        customer: customer.marker === 'live' ? customer.payload : null,
        statusHistory: statusHistory.payload || [],
        projectedAt: Date.now(),
      },
      source: 'compute:orderProjection',
    });
  }
}
Constraints

A projection is a derived cache, not a source of truth. Never write to a Vault based on projection data — always go back to the source Vaults. If the projection diverges from the Vaults (due to a missed event or timing), it must be rebuildable from scratch. Include a projectedAt timestamp so consumers know how fresh the data is. A projection that subscribes to too many event types or pulls from too many Vaults may become a performance bottleneck — scope projections to what the consumer actually needs.

Related: Fanout (projections subscribe to fanned-out events), Echo (projections rebuild on echoes from Vaults)
Convergent Pattern
The Projection is the read model in CQRS (Command Query Responsibility Segregation), described independently by Greg Young and Udi Dahan around 2010. Loop MMT arrived at the same shape through the L21 model — the operator reading from multiple sources to form a complete picture before routing — before recognizing the convergence. The problems are the same; the derivation paths differ.
6. The Bridge
L21: When a Pattern Matcher ejects a word, it goes to a one-word bridge buffer before being available to a bus. The bridge holds one word; a second match overwrites if the bridge hasn't drained.
What It Is

A buffering mechanism between a loop's output and the bus. The loop produces output faster than the bus can carry it. The bridge smooths the flow. Three overflow strategies, declared in the loop's spec: overwrite (latest wins, like the L21 PM bridge), queue (FIFO buffer), backpressure (the loop pauses until the bridge drains).

When to Use It

When a loop produces bursts of output that could overwhelm the bus or downstream consumers. Common with Inlet-pattern Working Loops receiving high-frequency streams (GPS positions, sensor data), or Compute Loops processing batch results. The strategy depends on the data semantics: position updates use overwrite (only the latest matters), order processing uses queue (every order matters), and payment processing uses backpressure (never lose a payment, never process faster than confirmation).

How It Works

The bridge sits inside the loop, between the processing logic and the bus emit call. The loop writes to the bridge instead of directly to the bus. The bridge drains to the bus according to the declared strategy. Bridge depth and overflow strategy are declared in the Constellation Spec, not hardcoded.

Code Example
// ── The Bridge: three strategies ──

function f_createBridge(strategy, maxDepth) {
  const v_buffer = [];
  let v_draining = false;

  return {
    // Loop writes here instead of directly to bus
    write(packet) {
      if (strategy === 'overwrite') {
        // Latest wins — replace whatever is in the buffer
        v_buffer.length = 0;
        v_buffer.push(packet);
      } else if (strategy === 'queue') {
        if (v_buffer.length >= maxDepth) {
          // Queue full — emit BRIDGE_OVERFLOW, drop oldest
          v_buffer.shift();
        }
        v_buffer.push(packet);
      } else if (strategy === 'backpressure') {
        if (v_buffer.length >= maxDepth) {
          // Caller must wait — return a promise that resolves
          // when space is available
          return new Promise((resolve) => {
            const v_interval = setInterval(() => {
              if (v_buffer.length < maxDepth) {
                clearInterval(v_interval);
                v_buffer.push(packet);
                resolve();
              }
            }, 10);
          });
        }
        v_buffer.push(packet);
      }
    },

    // Bus drains from here
    drain(emitFn) {
      if (v_draining || v_buffer.length === 0) return;
      v_draining = true;
      const packet = v_buffer.shift();
      emitFn(packet);
      v_draining = false;
    },

    get pending() { return v_buffer.length; },
  };
}

// Usage in a loop factory
function createInletGpsFleet({ bus }) {
  const v_bridge = f_createBridge('overwrite', 1);

  // High-frequency GPS data arrives here
  v_websocket.onmessage = (raw) => {
    const packet = f_translateGpsPayload(raw);
    v_bridge.write(packet);
  };

  // Drain to bus on a controlled interval
  setInterval(() => {
    v_bridge.drain((packet) => bus.data.emit(packet));
  }, 100); // 10 Hz max to bus, regardless of input rate
}
Constraints

Bridge strategy is a design decision, not an implementation detail — declare it in the Constellation Spec. overwrite loses data deliberately (appropriate when only the latest value matters). queue can lose data at the overflow boundary. backpressure never loses data but can stall the producing loop. Choose based on the cost of losing a packet vs. the cost of stalling. The Debounce pattern is related but different — Debounce manages event frequency at the semantic level; Bridge manages throughput at the transport level.

Related: Inlet (uses Bridge for back-pressure), Debounce (semantic-level frequency management vs. transport-level)
7. The Translator
No direct L21 analog. Derived from the dependency isolation principle.
What It Is

The data-scrubbing half of a loop that adopts an external dependency speaking a foreign data model. The Translator receives raw external data, validates it against the external provider's schema, maps it to the internal packet contract, and emits clean packets that are indistinguishable from any other packet on the bus. The translation logic is isolated inside the loop's closure — no other loop knows or cares that the data originated from a foreign system.

The Dependency Gate handles what happens when the external system is unreachable. The Translator handles what happens when it responds.

When to Use It

Every loop that talks to an adopted dependency: payment processors, auth providers, notification services, external APIs. The external system speaks its own language (Stripe webhook format, Supabase auth token structure, Twilio response shape). The Translator converts that language into clean internal packets before the data touches the bus.

How It Works

Implemented inside the loop that owns the dependency. The loop receives the raw external response, validates it against the known external schema (catch malformed responses early), maps fields to the internal contract, and emits a standard packet. The Constellation Spec should declare both the Dependency Gate (for resilience) and the Translator pattern (for data scrubbing).

Code Example
// ── The Translator: Stripe webhook → clean internal packet ──

function createSignalPayments({ bus, services }) {
  const { stripe, gate_stripe } = services;

  // Stripe webhook arrives
  async function h_onStripeWebhook(rawEvent) {
    // Step 1: Validate the raw external data
    const v_verified = stripe.webhooks.constructEvent(
      rawEvent.body, rawEvent.headers['stripe-signature'],
      v_webhookSecret
    );
    if (!v_verified) {
      bus.data.emit({
        type: 'LOOP_ERROR',
        payload: { loop: 'signal:payments',
                   error: 'Stripe signature verification failed' },
        source: 'signal:payments',
      });
      return;
    }

    // Step 2: Translate — map Stripe's schema to our contract
    // Stripe says "payment_intent.succeeded" with their field names.
    // We say "PAYMENT_CONFIRMED" with ours.
    if (v_verified.type === 'payment_intent.succeeded') {
      const v_stripeData = v_verified.data.object;
      bus.data.emit({
        type: 'PAYMENT_CONFIRMED',
        marker: 'live',
        payload: {
          paymentId:    v_stripeData.id,
          orderId:      v_stripeData.metadata.order_id,
          amountCents:  v_stripeData.amount,
          currency:     v_stripeData.currency,
          method:       v_stripeData.payment_method_types[0],
          confirmedAt:  new Date(v_stripeData.created * 1000).toISOString(),
        },
        source: 'signal:payments',
      });
    }

    if (v_verified.type === 'payment_intent.payment_failed') {
      const v_stripeData = v_verified.data.object;
      bus.data.emit({
        type: 'PAYMENT_FAILED',
        marker: 'live',
        payload: {
          paymentId:   v_stripeData.id,
          orderId:     v_stripeData.metadata.order_id,
          failureCode: v_stripeData.last_payment_error?.code || 'unknown',
          failureMsg:  v_stripeData.last_payment_error?.message || '',
        },
        source: 'signal:payments',
      });
    }
  }
}
Constraints

The translation must be complete — no Stripe field names, no Twilio response shapes, no external-provider-isms leaking onto the bus. If any downstream loop can tell the data came from an external source by looking at field names, the translation is incomplete. Validate the raw external data before translating — external APIs change without warning, and catching a malformed response at the Translator is cheaper than catching it downstream.

Related: Dependency Gate (handles unreachability; Translator handles response translation), Batch (Translator may handle batch webhook events)
8. The Batch
L21: Memory batch write sends all 16 slots sequentially. Destructive read clears after reading.
What It Is

Operating on multiple records atomically. A Workflow pipeline where the first step is a bulk pull request and subsequent steps operate on the entire result set. Vault Loops support batch insert, batch update, and batch archive as declared capabilities. The read-and-consume variant (fetch records and mark them as in-progress atomically) is how retry queues work — the Signal Loop fetches pending notifications and claims them in one operation, so no other instance re-sends them.

When to Use It

Any operation that needs to process a set of records together: sending all pending notifications, archiving all completed orders older than 90 days, reconciling all transactions from yesterday, downsampling a batch of sensor readings. Also the foundation of the read-and-consume pattern for work queues.

How It Works

The Workflow pipeline's first step is a bulk pull to the Vault. The Vault returns a set of records (marked live). Subsequent steps operate on the set. For read-and-consume: the pull atomically marks the records as in_progress so concurrent readers don't pick them up. On completion, the records are marked done. On failure, they return to pending after a timeout.

29. The Outbox
No direct L21 analog. Derived from the reliability requirement that a data write and its corresponding bus event must be atomic.
What It Is

A reliability pattern that makes Vault writes and their corresponding bus events atomic. The Vault Loop writes the business record and an outbox entry in the same database transaction. A separate sweep reads the outbox and emits to the bus. The outbox entry is deleted after the emit is confirmed. If the process crashes between the write and the emit, the outbox entry survives and the sweep picks it up on restart. No event is ever lost.

Without the Outbox, a Vault Loop that writes to the database and then emits an echo packet has a failure window: if the process crashes between the write and the emit, the record exists but no downstream loop ever learns about it.

When to Use It

Every Vault Loop where the echo matters for downstream processing. If a record is written but the ORDER_CREATED event is lost, the Projection never updates, the notification never sends, the analytics never count it. For business-critical data (orders, payments, customer records), the Outbox eliminates the failure window.

How It Works

The database has an outbox table. When the Vault Loop processes a write, it inserts both the business record and an outbox entry in the same transaction. The outbox entry contains the packet type, payload, and a published flag (initially false). A sweep function reads unpublished entries, emits them to the bus, and marks them published. The sweep is idempotent: if it crashes after emitting but before marking, the next sweep re-emits and downstream idempotency handling deduplicates.

Code Example
// ── The Outbox: atomic write + event ──

function createVaultOrders({ bus, db }) {
  bus.vault.on('CREATE_ORDER', async (packet) => {
    try {
      // Single transaction: business write + outbox entry
      const { data, error } = await db.rpc('create_order_with_outbox', {
        order_data: packet.payload,
        outbox_event: {
          packet_type: 'ORDER_CREATED',
          payload: packet.payload,
          source: 'vault:orders',
        },
      });
      if (error) throw error;

      // Immediately try to publish (best case: no delay)
      f_publishOutbox(bus, db);
    } catch (err) {
      bus.data.emit({
        type: 'LOOP_ERROR',
        payload: { loop: 'vault:orders', operation: 'CREATE_ORDER',
                   error: err.message },
        source: 'vault:orders',
      });
    }
  });

  async function f_publishOutbox(bus, db) {
    const { data: v_pending } = await db
      .from('outbox')
      .select('*')
      .eq('published', false)
      .order('created_at', { ascending: true })
      .limit(50);

    if (!v_pending || v_pending.length === 0) return;

    for (const entry of v_pending) {
      bus.vault.emit({
        type: entry.packet_type, marker: 'live',
        payload: entry.payload, source: entry.source,
      });
      await db.from('outbox')
        .update({ published: true,
                  published_at: new Date().toISOString() })
        .eq('id', entry.id);
    }
  }

  // Sweep on startup (catch missed during downtime)
  f_publishOutbox(bus, db);
  // Sweep periodically
  setInterval(() => f_publishOutbox(bus, db), 5000);
}

// Postgres RPC — atomic business write + outbox insert:
// CREATE FUNCTION create_order_with_outbox(
//   order_data jsonb, outbox_event jsonb
// ) RETURNS jsonb AS $$
// BEGIN
//   INSERT INTO orders (...) VALUES (...) RETURNING ...;
//   INSERT INTO outbox (packet_type, payload, source, published)
//     VALUES (..., false);
//   RETURN result;
// END; $$ LANGUAGE plpgsql;
Constraints

The business write and outbox insert must be in the same database transaction. If they are separate operations, the failure window still exists. The sweep must be idempotent. The outbox table needs a Retention Gate: published entries should be purged after a retention period. The sweep should have a batch limit.

Related: Echo (Outbox guarantees the echo is eventually emitted), Dead Letter (items that fail the outbox sweep), State Reincarnation (solves the same problem for workflow steps; Outbox solves it for non-workflow writes)
Adopted Pattern
The Outbox pattern is established practice in distributed systems and microservices architecture, associated primarily with Chris Richardson's work on microservices patterns and the Transactional Outbox solution to the dual-write problem. The pattern's L21 origin note is accurate — the pattern has no direct hardware analog — and it was adopted into Loop MMT because it solves a real reliability problem that emerges from the architecture's own constraints. The concept and the name come from distributed systems literature.

9. The Flag
L21: The comparator produces six flags (GT, LT, EQ, GTE, LTE, NEQ) alongside the ALU result. The flags inform — they do not trigger automatic action.
What It Is

When a Compute Loop produces a result, it also produces diagnostic flags alongside the primary payload. Flags are metadata that describe properties of the result: loyaltyDiscountApplied: true, priceExceedsAverage: true, weightInTopDecile: true. Flags travel in the packet. Any downstream loop can read them. The producing loop doesn't know who reads the flags.

When to Use It

When a computation produces information that downstream loops might want to act on but that isn't part of the core result. The Pricing Engine knows the loyalty discount was applied — the UI wants to display that, the Audit Trail wants to log it, the Sentinel wants to track it. Rather than each downstream loop re-deriving the information, the Flag carries it.

How It Works

Implemented inside a Compute Loop. After computing the primary result, the loop evaluates flag conditions and includes them in the emitted packet's payload under a flags object. Downstream loops read packet.payload.flags and branch or display accordingly.

Code Example
// ── The Flag: Pricing Engine sets diagnostic flags ──

function createComputePricing({ bus }) {
  bus.data.on('ORDER_HYDRATED', (packet) => {
    const v_basePrice = f_calculateBasePrice(packet.payload);
    const v_discount = f_calculateDiscount(packet.payload.customer);
    const v_finalPrice = v_basePrice - v_discount;
    const v_averagePrice = f_getRunningAverage();

    bus.data.emit({
      type: 'PRICE_CALCULATED',
      marker: 'live',
      payload: {
        orderId: packet.payload.orderId,
        basePrice: v_basePrice,
        discount: v_discount,
        totalPrice: v_finalPrice,
        flags: {
          loyaltyDiscountApplied: v_discount > 0,
          priceExceedsAverage: v_finalPrice > v_averagePrice,
          rushOrder: packet.payload.rushProcessing === true,
          bulkOrder: packet.payload.items.length > 10,
        },
      },
      source: 'compute:pricing',
    });
  });
}
Constraints

Flags inform — they do not trigger automatic action. A flag should never cause a branching decision in the producing loop itself; that would be business logic, not a flag. Flags are for downstream consumers. Keep the flags object flat (boolean or simple values). If a "flag" needs a complex payload, it's probably a separate packet, not a flag.

Related: Stamp (adds metadata at the Filter level; Flag adds metadata at the Compute level), Counter (can increment on flag values)
10. The Scratch
L21: Four Working Scratch registers hold intermediate values during multi-step computation without the 41-tick cost of a bus transfer.
What It Is

A loop holds transient working state during a single operation — running totals during a reconciliation batch, intermediate values during a price calculation. Scratch state is never persisted to a Vault, never exposed on the bus, and is cleared when the transaction completes. This is a declared exception to the Two-Place Rule.

When to Use It

When a computation needs intermediate values that have no meaning outside the current operation. A reconciliation loop summing a batch of transactions. A pricing engine tracking discount tiers across line items. An aggregation loop computing averages. The intermediate values are meaningless if the operation fails — they'd be recomputed from inputs on retry.

How It Works

Simple JavaScript variables inside the loop's processing function. The key discipline: scratch state is scoped to a single function invocation or a single event handler execution. It does not persist across separate bus events. If the loop crashes, scratch state is gone, and that's fine because it would be recomputed.

Code Example
// ── The Scratch: reconciliation with running totals ──

function createComputeReconciliation({ bus }) {
  bus.data.on('RECONCILE_REQUEST', async (packet) => {
    const v_transactions = packet.payload.transactions;

    // Scratch state — exists only during this computation
    let v_totalCredits = 0;
    let v_totalDebits = 0;
    let v_discrepancies = [];
    let v_processedCount = 0;

    for (const txn of v_transactions) {
      if (txn.type === 'credit') v_totalCredits += txn.amount;
      if (txn.type === 'debit') v_totalDebits += txn.amount;
      if (Math.abs(txn.reportedBalance - txn.calculatedBalance) > 0.01) {
        v_discrepancies.push({
          txnId: txn.id,
          reported: txn.reportedBalance,
          calculated: txn.calculatedBalance,
        });
      }
      v_processedCount++;
    }
    // Scratch state used, now emit the result
    bus.data.emit({
      type: 'RECONCILIATION_COMPLETE',
      marker: 'live',
      payload: {
        totalCredits: v_totalCredits,
        totalDebits: v_totalDebits,
        net: v_totalCredits - v_totalDebits,
        discrepancies: v_discrepancies,
        processedCount: v_processedCount,
      },
      source: 'compute:reconciliation',
    });
    // Scratch state goes out of scope here — gone
  });
}
Constraints

Scratch state must never leak into loop-level variables that persist across events. If you find yourself storing scratch state in a closure variable that survives after the event handler returns, it is no longer scratch — it is ghost state. The test: if the loop restarted right now, would this variable need to be recovered? If yes, it's not scratch — persist it.

Related: Two-Place Rule (Scratch is the declared exception), Flag (results of scratch computation can become flags)
11. The Clamp
L21: A Threshold Gate in clamp mode rewrites values in-place to a configured bound without ejecting them. TG with mode ≥ 1000, clamp 1000 silently clips every value above 1000.
What It Is

A Filter Loop that modifies specific fields within declared bounds. A weight normalizer that caps at a safety maximum. A price floor that ensures no order goes below a minimum. A date normalizer that rounds timestamps. The key distinction from a Compute Loop: a Clamp enforces bounds, not business logic.

When to Use It

When data can arrive with values outside an acceptable range and the correct behavior is to clip them rather than reject them. A weight entry of 99,999 lbs is clearly a typo — clamp to the declared maximum. A negative price is impossible — clamp to zero. Distinct from validation (which rejects bad data). A Clamp fixes bad data within declared limits.

How It Works

Implemented as a Filter Loop. The bounds are declared in the Constellation Spec or config, not hardcoded. The loop reads the bounds, checks each relevant field, and clamps or rejects based on the declared onViolation policy. Emits a stamped flag indicating clamping occurred.

Code Example
// ── The Clamp: weight bounds enforcement ──

const CLAMP_RULES = {
  weight: { min: 0.1, max: 2000, onViolation: 'clamp' },
  price:  { min: 0,   max: 99999, onViolation: 'reject' },
};

function createFilterWeightClamp({ bus }) {
  bus.data.on('ORDER_HYDRATED', (packet) => {
    const v_payload = { ...packet.payload };
    let v_clamped = false;

    for (const [field, rule] of Object.entries(CLAMP_RULES)) {
      const v_value = v_payload[field];
      if (v_value === undefined) continue;

      if (v_value < rule.min || v_value > rule.max) {
        if (rule.onViolation === 'reject') {
          bus.data.emit({
            type: 'VALIDATION_REJECTED',
            payload: { field, value: v_value, min: rule.min, max: rule.max,
                       reason: 'out_of_bounds' },
            source: 'filter:weightClamp',
          });
          return; // Stop processing — rejected
        }
        // Clamp: silently adjust to nearest bound
        v_payload[field] = Math.max(rule.min, Math.min(rule.max, v_value));
        v_clamped = true;
      }
    }

    bus.data.emit({
      type: 'ORDER_BOUNDS_CHECKED',
      marker: 'live',
      payload: {
        ...v_payload,
        flags: { ...(v_payload.flags || {}), boundsClamped: v_clamped },
      },
      source: 'filter:weightClamp',
    });
  });
}
Constraints

Bounds must come from the Constellation Spec or Config Vault, not from hardcoded values. The Clamp should stamp a flag when it adjusts a value so downstream loops and the audit trail know the original value was out of bounds. Never clamp silently without a flag — the operator needs to know when data arrives out of range.

Related: Stamp (Clamp often stamps to indicate clamping occurred), Cascade (Clamp is often a step in a filter cascade)
12. The Stamp
L21: Pattern Matcher rewrite mode matches a word and rewrites specific bits without ejecting it — in-place annotation of data as it circulates.
What It Is

A Filter Loop that matches a condition and adds metadata to the packet without removing it from the pipeline. Not the same as Hydration (which pulls external data). The Stamp adds information derived from the packet itself: "this order is above average weight," "this customer is VIP tier," "this order was entered during the Tuesday rush." Stamps are additive — they do not modify existing fields, they add new ones.

When to Use It

When downstream loops or the Presentation Loop need to know something about a packet that can be derived from its contents but shouldn't be re-derived by every consumer. A VIP flag on a customer order. A "bulk" tag on a large order. A "rush hour" annotation on an intake. The Stamp computes it once and carries it forward.

How It Works

Implemented as a Filter Loop. Evaluates conditions against the packet payload. Adds new fields (under a stamps namespace or as individual flags) without modifying existing fields. Emits the annotated packet.

Code Example
// ── The Stamp: annotate orders with business classifications ──

function createFilterOrderStamp({ bus }) {
  bus.data.on('ORDER_BOUNDS_CHECKED', (packet) => {
    const v_stamps = {};

    // Weight classification
    if (packet.payload.weight > 500) {
      v_stamps.heavyOrder = true;
    }

    // Time-of-day classification
    const v_hour = new Date().getHours();
    if (v_hour >= 6 && v_hour <= 9) {
      v_stamps.morningRush = true;
    }

    // Customer tier (from hydrated customer data)
    if (packet.payload.customer &&
        packet.payload.customer.totalOrders > 50) {
      v_stamps.vipCustomer = true;
    }

    // Emit with stamps added — original fields untouched
    bus.data.emit({
      type: 'ORDER_STAMPED',
      marker: 'live',
      payload: {
        ...packet.payload,
        stamps: { ...(packet.payload.stamps || {}), ...v_stamps },
      },
      source: 'filter:orderStamp',
    });
  });
}
Constraints

Stamps are additive only — never modify existing fields. Use a stamps namespace to keep stamp fields separate from core data fields. A stamp that requires a Vault pull is not a stamp — it's a Hydration. Stamps derive from data already in the packet.

Related: Flag (produced by Compute Loops; Stamps produced by Filter Loops), Hydration (pulls external data; Stamp derives from packet data), Sentinel (can monitor stamp patterns)
13. The Reorder
L21: The operator can switch whether Pattern Matchers run before Threshold Gates or vice versa on the Big Loop. Filter order is configurable at runtime.
What It Is

Cascade filter order is a routing concern, not a filter concern. The individual filters do not change. The routing table determines the sequence. Want authorization before validation? Change the routing. Want sanitization before auth? Insert a new step. The filters are independent modules — the routing table is the composition layer.

When to Use It

When you need to change the order of filter processing without modifying any filter code. During development: "we realized auth should happen before validation." In production: "we need to insert a rate limiter before everything else." Because each filter is independent and the cascade is defined by routing, reordering is a routing table change — zero code changes.

How It Works

Not a code pattern inside a loop — it's a routing table pattern. The implementation is: change the packet type wiring between filters. Filter A's output type becomes Filter C's input, Filter C's output becomes Filter B's input. The filters themselves are unchanged.

Code Example
// ── The Reorder: changing filter sequence via routing table ──

// BEFORE: Validate → Auth → Hydrate
const ROUTES_BEFORE = [
  { type: 'ORDER_INTAKE_RAW',        source: 'working:intake',
    dest: 'filter:validation',         bus: 'DATA' },
  { type: 'ORDER_INTAKE_VALIDATED',   source: 'filter:validation',
    dest: 'filter:auth',               bus: 'DATA' },
  { type: 'ORDER_INTAKE_AUTHORIZED',  source: 'filter:auth',
    dest: 'filter:hydration',          bus: 'DATA' },
];

// AFTER: Auth → Validate → Hydrate
// Only the routing table changes. No filter code changes.
const ROUTES_AFTER = [
  { type: 'ORDER_INTAKE_RAW',        source: 'working:intake',
    dest: 'filter:auth',               bus: 'DATA' },
  { type: 'ORDER_INTAKE_AUTHORIZED',  source: 'filter:auth',
    dest: 'filter:validation',          bus: 'DATA' },
  { type: 'ORDER_INTAKE_VALIDATED',   source: 'filter:validation',
    dest: 'filter:hydration',          bus: 'DATA' },
];

// The filter factories are IDENTICAL in both configurations.
// Only the routing table entries changed.
Constraints

This only works if each filter is truly independent — no filter should assume what came before it or what comes after it. If Filter B depends on a stamp that Filter A adds, reordering them breaks the dependency. Keep filters independent. If an ordering dependency exists, document it in the routing table comments.

Related: Cascade (Reorder changes cascade sequence), Stamp (ordering dependencies can arise when one filter stamps and another reads the stamp)
14. The Counter
L21: Five counters with configurable triggers. The Working counter can increment on "word passes read head," "bus transfer received," or "inject delivered." Triggers are operator-configurable at runtime.
What It Is

Every loop maintains counters — packets processed, errors encountered, average processing time, specific event counts. These feed health packets for the Observatory. The L21 insight: triggers are configurable. Not just "packets processed" but "orders above $300" or "cancellations from repeat customers." Counter triggers are declared as conditions in the Config Vault, updateable at runtime.

When to Use It

Every loop needs counters for health monitoring. Use configurable counters when the Operator needs visibility into specific business conditions that may change over time — tracking orders above a threshold, monitoring cancellation rates by customer tier, counting API calls by provider. The Sentinel pattern is built on top of Counter infrastructure.

How It Works

Implemented inside each loop factory. A counter manager receives the loop's trigger configuration from the Config Vault. On each processed event, it evaluates the triggers and increments matching counters. Periodically emits a health packet with current counter values.

Code Example
// ── The Counter: configurable triggers inside a loop ──

function f_createCounterManager(loopName) {
  const v_counters = {};
  let v_triggers = [];

  return {
    loadTriggers(triggerConfig) {
      // Triggers from Config Vault — updateable at runtime
      // Example: [{ name: 'high_value_orders', field: 'totalPrice',
      //             condition: 'gt', threshold: 300 }]
      v_triggers = triggerConfig;
      for (const t of v_triggers) {
        v_counters[t.name] = v_counters[t.name] || 0;
      }
    },

    // Called after every processed packet
    evaluate(packet) {
      v_counters['_total'] = (v_counters['_total'] || 0) + 1;

      for (const t of v_triggers) {
        const v_value = packet.payload?.[t.field];
        if (v_value === undefined) continue;
        const v_match =
          (t.condition === 'gt'  && v_value > t.threshold)  ||
          (t.condition === 'lt'  && v_value < t.threshold)  ||
          (t.condition === 'eq'  && v_value === t.threshold) ||
          (t.condition === 'gte' && v_value >= t.threshold);
        if (v_match) v_counters[t.name]++;
      }
    },

    // Emitted as part of health packet
    getSnapshot() {
      return { loop: loopName, counters: { ...v_counters },
               timestamp: Date.now() };
    },
  };
}
Constraints

Counter triggers must be declared in the Config Vault, not hardcoded. The Operator should be able to add, remove, or change trigger conditions without a code deployment. Keep counter names descriptive and stable — the Sentinel and Observatory dashboard will reference them. Reset counters on a declared schedule (per health interval), not on every packet.

Related: Sentinel (built on counter infrastructure), Config Vault (triggers stored there), Heartbeat Absence (absence of expected counter increments is itself a signal)
15. The Fanout
L21: The operator can route ALU output to multiple bus destinations simultaneously. Multiple buses can be active, each carrying the same result to a different loop.
What It Is

One event triggers multiple independent subscribers. When an ORDER_CREATED packet fires, the inventory loop, the notification loop, the analytics loop, and the projection loop all receive it independently. None knows about the others. None depends on the others. The routing table is the fanout declaration — each subscriber is a separate entry.

When to Use It

Any time a single event is relevant to multiple independent consumers. Order creation triggers inventory update, customer notification, analytics tracking, and UI update — all independently. A status change triggers parent evaluation, audit logging, and notification. The Fanout is implicit in the bus model but worth naming as a conscious design decision because adding a new subscriber is a routing table entry, not a code change.

How It Works

Not a code pattern inside a loop — it's a routing table pattern. The producing loop emits one packet. The routing table declares multiple subscribers for that packet type. The bus delivers a copy to each subscriber. The subscribers process independently and cannot affect each other.

Code Example
// ── The Fanout: ORDER_CREATED reaches four independent consumers ──

// Routing table entries — one producer, four consumers
const FANOUT_ORDER_CREATED = [
  { type: 'ORDER_CREATED', source: 'vault:orders',
    dest: 'compute:orderProjection',   bus: 'VAULT' },
  { type: 'ORDER_CREATED', source: 'vault:orders',
    dest: 'signal:customerNotify',     bus: 'VAULT' },
  { type: 'ORDER_CREATED', source: 'vault:orders',
    dest: 'compute:analytics',          bus: 'VAULT' },
  { type: 'ORDER_CREATED', source: 'vault:orders',
    dest: 'filter:inventoryCheck',     bus: 'VAULT' },
];

// The Vault Loop emits once — it doesn't know about the fanout
function createVaultOrders({ bus, db }) {
  // ... on successful insert:
  bus.vault.emit({
    type: 'ORDER_CREATED',
    marker: 'live',
    payload: record,
    source: 'vault:orders',
  });
  // That's it. The bus handles delivery to all four subscribers.
  // Adding a fifth subscriber = one routing table entry.
  // The Vault Loop code: unchanged.
}
Constraints

The producing loop must not know about or depend on its subscribers. If you find the producer checking whether a specific subscriber received the packet, the fanout discipline is broken. Each subscriber handles its own errors independently — a failure in the notification loop does not affect the analytics loop. If a subscriber's failure should stop the overall operation, it's not a fanout — it's a workflow pipeline.

Related: Projection (subscribes to fanned-out events), Echo (the echo itself is often fanned out to multiple consumers)
16. The Debounce
L21: The operator waits for a loop to stabilize — watching values circulate and settle — before reading the result and routing it onward.
What It Is

A Filter Loop that coalesces rapid-fire events, forwarding only the last event after a declared quiet period. A user types in a search field; you don't fire a query on every keystroke — you wait until they pause and then forward the final input. A status field gets updated three times in two seconds; downstream loops only need the final value.

When to Use It

When the same event fires rapidly and downstream consumers only need the final value, not every intermediate state. Search-as-you-type. Slider adjustments. Status fields that bounce through intermediate states before settling. Config changes that trigger rebuilds — debounce so only the final config fires the rebuild. Distinct from Bridge (which manages throughput at the transport level). Debounce manages event frequency at the semantic level.

How It Works

Implemented as a Filter Loop. Holds the most recent event in scratch state with a timer. Each new event resets the timer. When the timer expires (quiet period elapsed), the Filter forwards the held event. The debounce interval is declared per packet type.

Code Example
// ── The Debounce: coalesce rapid search input ──

function createFilterSearchDebounce({ bus }) {
  const DEBOUNCE_MS = 300;
  const v_pending = new Map(); // keyed by debounce group

  bus.data.on('SEARCH_INPUT_RAW', (packet) => {
    const v_key = packet.payload.fieldId || 'default';

    // Clear existing timer for this key
    if (v_pending.has(v_key)) {
      clearTimeout(v_pending.get(v_key).timer);
    }

    // Hold the latest event and start a new timer
    const v_timer = setTimeout(() => {
      // Quiet period elapsed — forward the held event
      bus.data.emit({
        type: 'SEARCH_INPUT_DEBOUNCED',
        marker: 'live',
        payload: packet.payload,
        source: 'filter:searchDebounce',
      });
      v_pending.delete(v_key);
    }, DEBOUNCE_MS);

    v_pending.set(v_key, { packet, timer: v_timer });
  });
}

// ── Status change debounce (different interval) ──

function createFilterStatusDebounce({ bus }) {
  const DEBOUNCE_MS = 1000; // Status changes settle slower
  const v_pending = new Map();

  bus.data.on('STATUS_CHANGE_RAW', (packet) => {
    const v_key = packet.payload.recordId;

    if (v_pending.has(v_key)) {
      clearTimeout(v_pending.get(v_key).timer);
    }

    const v_timer = setTimeout(() => {
      bus.data.emit({
        type: 'STATUS_CHANGE',
        marker: 'live',
        payload: packet.payload,
        source: 'filter:statusDebounce',
      });
      v_pending.delete(v_key);
    }, DEBOUNCE_MS);

    v_pending.set(v_key, { packet, timer: v_timer });
  });
}
Constraints

The v_pending map is scratch state that persists across events — this is the one pattern where scratch state legitimately lives longer than a single event handler. If the loop crashes, pending debounced events are lost. For critical events (payment status changes), do not debounce — use the Bridge pattern instead. Debounce intervals should come from the Config Vault, not hardcoded values. Key the debounce by the appropriate field (record ID, field ID, user session) to avoid unrelated events resetting each other's timers.

Related: Bridge (transport-level throughput management vs. semantic-level frequency), Scratch (debounce pending state is a special case of scratch)
Adopted Pattern
Debounce is a well-established technique originating in electronics (suppressing mechanical switch bounce) and widely adopted by the JavaScript and UI ecosystem. The name and concept are used without modification. The L21 analog is genuine — the operator deliberately pauses before routing a settled input — but the pattern was already named and in broad use before Loop MMT was conceived. Loop MMT adopts it rather than claims it.
Category 3
Workflow Patterns

Patterns for multi-step orchestration within the Workflow Loop's pipeline spec system.

17. The Split-Merge
L21: Multiple buses can transfer data simultaneously between different loops. The operator configures parallel data paths.
What It Is

Extends pipeline specs to support parallel step groups. The Workflow Loop dispatches all steps in a group simultaneously, waits for completion, then proceeds with merged results. merge: 'all' waits for every branch. merge: 'any' proceeds on the first response (useful for redundant fallback paths). onBranchFail: 'abort' kills the group. onBranchFail: 'skip' lets surviving branches complete.

When to Use It

When multiple independent operations can run concurrently: compute price while checking inventory while verifying payment method. When you need redundant paths: try primary API, try fallback API, use whichever responds first. When the total latency of sequential steps is unacceptable but the steps are independent.

How It Works

Declared in the pipeline spec using a type: 'split' step. The Workflow engine dispatches all branches simultaneously, tracks completions, and merges results according to the declared merge strategy. Subsequent steps access branch results via needs: ['parallel_step.*'].

Code Example
// ── The Split-Merge: parallel order preparation ──

const PIPELINE_CREATE_ORDER = {
  name: 'create_order',
  trigger: { type: 'ORDER_VALIDATED', bus: 'DATA' },
  steps: [
    {
      name: 'parallel_prep',
      type: 'split',
      branches: [
        {
          name: 'price',
          emit: { type: 'COMPUTE_PRICE', bus: 'DATA' },
          expect: 'PRICE_CALCULATED',
          timeout: 5000,
          onFail: 'abort',
        },
        {
          name: 'inventory',
          emit: { type: 'CHECK_INVENTORY', bus: 'VAULT' },
          expect: 'INVENTORY_CONFIRMED',
          timeout: 3000,
          onFail: 'abort',
        },
        {
          name: 'payment_method',
          emit: { type: 'VERIFY_PAYMENT_METHOD', bus: 'SIGNAL' },
          expect: 'PAYMENT_METHOD_VERIFIED',
          timeout: 10000,
          onFail: 'abort',
        },
      ],
      merge: 'all',          // Wait for all three
      onBranchFail: 'abort', // If any fails, abort the group
    },
    {
      name: 'persist_order',
      emit: { type: 'CREATE_ORDER', bus: 'VAULT' },
      needs: ['parallel_prep.*'], // Access all branch results
      expect: 'ORDER_CREATED',
      timeout: 5000,
      onFail: 'escalate',
    },
  ],
};
Constraints

Each branch must be truly independent — if branch B depends on branch A's result, they cannot run in parallel. Set per-branch timeouts, not just a group timeout — a slow inventory check shouldn't be masked by a fast price calculation. The merge: 'any' strategy is powerful for redundant fallback paths but be careful: the "losing" branch may still complete and its side effects (if any) must be handled.

Related: Batch (parallel processing of a set), Cascade Update (may use Split-Merge for parallel child evaluations)
Convergent Pattern
The Split-Merge is the Fork-Join pattern in parallel computing — dispatch independent tasks simultaneously, wait for results, then continue. The term "fork-join" dates to concurrent programming literature of the 1960s and is the execution model behind Java's ForkJoinPool, OpenMP, and similar constructs. Loop MMT uses different terminology because the implementation lives inside a pipeline spec rather than as a threading primitive, but the underlying shape is not new.
18. The Cascade Update
L21: Closing a gate on the Big Loop affects data that then passes through PM and TG inspection — a change in one part of the pipeline affects evaluation downstream.
What It Is

A Workflow pattern for hierarchical state propagation. When a child record changes state, the pipeline evaluates whether the parent should also change. Two pipeline spec features enable this: conditional steps (condition field — execute only if prior results satisfy a predicate) and recursive pipelines (recursion block with maxDepth guard).

When to Use It

When records have parent-child relationships and state changes should propagate through the hierarchy. All tasks in a project are done → project status should change to done. A parent record is archived → all children should be archived. An order's line items all ship → order status should change to shipped. Works both upward (child completes → parent evaluates) and downward (parent archived → children archived).

How It Works

The pipeline's first step fetches siblings (for upward cascades) or children (for downward cascades). A conditional step evaluates whether the parent/children should transition. If the condition is met, the transition fires and the pipeline can recursively trigger itself for the next level of the hierarchy, up to a declared maxDepth.

Code Example
// ── The Cascade Update: task completion propagates to project ──

const PIPELINE_CASCADE_TASK = {
  name: 'cascade_task_completion',
  trigger: {
    type: 'STATUS_CHANGE',
    match: { newStatus: 'done', recordType: 'task' },
  },
  steps: [
    {
      name: 'fetch_siblings',
      emit: { type: 'FETCH_TASKS', bus: 'VAULT' },
      queryParams: (trigger) => ({
        parentId: trigger.payload.parentId,
      }),
      expect: 'TASKS_FETCHED',
      onFail: 'abort',
    },
    {
      name: 'evaluate_parent',
      emit: { type: 'EVALUATE_COMPLETION', bus: 'DATA' },
      needs: ['fetch_siblings.payload'],
      expect: 'EVALUATION_RESULT',
      onFail: 'abort',
    },
    {
      name: 'transition_parent',
      emit: { type: 'STATUS_CHANGE', bus: 'DATA' },
      needs: ['evaluate_parent.payload'],
      // Only fire if ALL siblings are done
      condition: (results) =>
        results.evaluate_parent.payload.allComplete === true,
      expect: 'STATUS_VALIDATED',
      onFail: 'abort',
    },
  ],
  recursion: {
    enabled: true,
    maxDepth: 10,
    trigger: 'transition_parent.success',
  },
};
Constraints

Always set a maxDepth guard — without it, a circular hierarchy or a very deep tree causes infinite recursion. If the hierarchy exceeds the depth limit, the cascade stops and emits CASCADE_DEPTH_EXCEEDED for the operator. The conditional step's predicate must be deterministic — the same inputs must always produce the same decision. Downward cascades combine with the Batch pattern: "archive all children of this parent" is a batch status update.

Related: Batch (downward cascades are batch operations), Split-Merge (parallel child evaluation)
Category 4
Resilience Patterns

Patterns for monitoring, detecting anomalies, and recovering from failures.

19. The Sentinel
L21: The operator monitors running match counts on Pattern Matchers and counter values to detect unusual patterns over time — not just individual values, but trends.
What It Is

A Filter Loop that subscribes to the archive buffer (or to live traffic via passive observation) and applies pattern detection over a sliding window. Is the cancellation rate unusually high? More than five LOOP_ERROR events in an hour? One operator creating orders at triple the normal rate? The Sentinel emits ANOMALY_DETECTED on the Signal Bus. It is a watchdog, not a gatekeeper — it does not block traffic.

When to Use It

When you need to detect patterns that emerge over time, not individual bad values. A single error is handled by the Loop Recovery Protocol. Five errors in a row from the same loop is an anomaly worth investigating. A sudden spike in cancellations. An operator account with unusual activity. The Sentinel watches trends; the Counter provides the data; the Operator Alert loop delivers the notification.

How It Works

Implemented as a Filter Loop using passive observation (reads traffic without being a routing destination). Maintains a sliding window of recent events. Evaluates anomaly rules on each new event. Emits ANOMALY_DETECTED when a rule fires. Rules are declared in the Config Vault and updateable at runtime.

Code Example
// ── The Sentinel: anomaly detection with sliding window ──

function createFilterSentinel({ bus }) {
  const WINDOW_MS = 60 * 60 * 1000; // 1 hour window
  const v_events = []; // sliding window

  // Anomaly rules — from Config Vault, updateable at runtime
  let v_rules = [
    { name: 'error_spike',  type: 'LOOP_ERROR',
      threshold: 5,         window: WINDOW_MS },
    { name: 'cancel_spike', type: 'ORDER_CANCELLED',
      threshold: 10,        window: WINDOW_MS },
  ];

  // Passive observation — sees all traffic, not a routing destination
  bus.data.observe((packet) => {
    v_events.push({ type: packet.type, timestamp: Date.now(),
                    source: packet.source });

    // Prune events outside the window
    const v_cutoff = Date.now() - WINDOW_MS;
    while (v_events.length > 0 && v_events[0].timestamp < v_cutoff) {
      v_events.shift();
    }

    // Evaluate rules
    for (const rule of v_rules) {
      const v_matches = v_events.filter(
        (e) => e.type === rule.type &&
               e.timestamp > Date.now() - rule.window
      );
      if (v_matches.length >= rule.threshold) {
        bus.signal.emit({
          type: 'ANOMALY_DETECTED',
          payload: {
            rule: rule.name,
            count: v_matches.length,
            threshold: rule.threshold,
            window: rule.window,
            samples: v_matches.slice(-3), // Last 3 for context
          },
          source: 'filter:sentinel',
        });
      }
    }
  });

  // Listen for rule updates from Config Vault
  bus.data.on('CONFIG_UPDATED', (packet) => {
    if (packet.payload.key === 'sentinel_rules') {
      v_rules = packet.payload.value;
    }
  });
}
Constraints

The Sentinel is a watchdog, not a gatekeeper — it emits alerts but does not block traffic. If you need to block traffic based on an anomaly, that's a rate limiter or a circuit breaker, not a Sentinel. Rules must come from the Config Vault so the Operator can adjust thresholds without a deployment. Be careful with the sliding window size — too large and memory grows; too small and patterns are missed. The Sentinel uses passive observation (bus.observe) not subscription (bus.on) — it should not appear as a routing destination.

Related: Counter (provides the data the Sentinel watches), Heartbeat Absence (absence detection vs. anomaly detection), Operator Alert (delivers Sentinel findings)
20. The Heartbeat Absence
L21: If a loop stops circulating data (gate stuck, clock stopped), the operator notices because the read head stops displaying new values. The absence of expected activity is itself a signal.
What It Is

The Observatory (or Operator Alert loop) maintains a table of expected periodic events and their tolerance windows. If the daily reconciliation usually runs at 11pm and it is 11:05 with no RECONCILE_REQUEST packet, the Clock may be dead or the schedule table failed to load. A MISSING_EVENT alert fires. Most monitoring alerts on presence (an error occurred). This alerts on absence (an expected event did not occur).

When to Use It

When the system has expected periodic events — scheduled jobs, heartbeats, health checks — and the absence of those events is itself a problem. The Clock should emit heartbeats every 30 seconds. The daily reconciliation should run at 11pm. The health check should fire every 5 minutes. If any of these stop happening, something is broken — and no error event will be emitted because the thing that would emit the event is the thing that's broken.

How It Works

Implemented in the Operator Alert loop or as a dedicated absence monitor. Maintains a table of expected events with their schedules and tolerance windows. On each tick (driven by its own timer, not by the Clock — because the Clock might be the thing that died), it checks whether each expected event has been seen within its tolerance window.

Code Example
// ── Heartbeat Absence: detect missing expected events ──

function createFilterAbsenceMonitor({ bus }) {
  const v_expectedEvents = [
    { type: 'HEALTH_HEARTBEAT', source: 'clock',
      intervalMs: 30000, toleranceMs: 10000 },
    { type: 'RECONCILE_REQUEST', source: 'clock',
      intervalMs: 24 * 60 * 60 * 1000, toleranceMs: 5 * 60 * 1000 },
  ];

  const v_lastSeen = new Map();

  // Track when expected events arrive
  bus.data.observe((packet) => {
    for (const expected of v_expectedEvents) {
      if (packet.type === expected.type) {
        v_lastSeen.set(expected.type, Date.now());
      }
    }
  });

  // Check on our OWN timer — not the Clock
  // (because the Clock might be dead)
  setInterval(() => {
    const v_now = Date.now();
    for (const expected of v_expectedEvents) {
      const v_last = v_lastSeen.get(expected.type) || 0;
      const v_elapsed = v_now - v_last;
      const v_deadline = expected.intervalMs + expected.toleranceMs;

      if (v_elapsed > v_deadline) {
        bus.signal.emit({
          type: 'MISSING_EVENT',
          payload: {
            expectedType: expected.type,
            expectedSource: expected.source,
            lastSeenMs: v_last,
            elapsedMs: v_elapsed,
            deadlineMs: v_deadline,
          },
          source: 'filter:absenceMonitor',
        });
      }
    }
  }, 15000); // Check every 15 seconds
}
Constraints

The absence monitor must run on its own timer, not on the Clock — because the Clock dying is one of the primary failure modes it needs to detect. Set tolerance windows that are meaningful: a heartbeat expected every 30 seconds with a 10-second tolerance fires at 40 seconds of silence. Avoid false alarms by calibrating tolerances to real-world variance. The expected event table should be configurable via the Config Vault so new scheduled events are automatically monitored.

Related: Sentinel (detects anomalous presence; Heartbeat Absence detects anomalous absence), Counter (absence of expected counter increments), Clock (often the monitored source)
21. The Replay
L21: The .loop session recording file captures every operator action and machine event. A .loop file can be read to reconstruct exactly what happened, in order, at any point in the session.
What It Is

The Transit Gate's archive buffer stores recent bus traffic. The Replay pattern reads the buffer and re-emits packets in order — either to the live system (reprocess a failed batch) or to a test harness (reproduce a bug). Operator reports: "the price was wrong on the 3pm order." Pull the archive buffer, find the COMPUTE_PRICE packet, inspect its inputs, replay it through the Pricing Engine in isolation. No guessing. The exact packets are in the buffer.

When to Use It

Post-incident debugging: what happened, exactly, with the exact data? Reprocessing a failed batch after fixing the bug. Regression testing: replay yesterday's traffic through today's code and compare outputs. Any situation where you need to answer "what exactly happened" with certainty rather than reconstruction.

How It Works

Three operations: extract (pull packets from the archive buffer by time range and type), replay (re-emit the packets to a target loop or test harness in order), and compare (diff the replay output against the original output). For isolated replay, the target loop is wired to a Phantom bus so replayed packets don't affect the live system.

Code Example
// ── The Replay: extract, replay, compare ──

// Extract packets from the archive buffer
function f_extractFromArchive(bus, options) {
  const { packetType, startTime, endTime, source } = options;
  return bus.data.getArchive({
    type: packetType,
    after: startTime,
    before: endTime,
    source: source || undefined,
  });
}

// Replay through a specific loop in isolation
async function f_replayThroughLoop(packets, loopFactory, capabilities) {
  // Create an isolated bus for replay — not the live bus
  const v_replayBus = f_createBus('replay');
  const v_results = [];

  // Capture outputs
  v_replayBus.data.observe((packet) => {
    v_results.push(packet);
  });

  // Create the loop with the replay bus
  const v_loop = loopFactory({ bus: v_replayBus, ...capabilities });

  // Feed the extracted packets in order
  for (const packet of packets) {
    v_replayBus.data.emit(packet);
    // Allow async processing to complete
    await new Promise((r) => setTimeout(r, 10));
  }

  return v_results;
}

// Compare replay output against original
function f_compareResults(original, replayed) {
  const v_diffs = [];
  for (let i = 0; i < Math.max(original.length, replayed.length); i++) {
    const o = original[i];
    const r = replayed[i];
    if (!o || !r || JSON.stringify(o.payload) !== JSON.stringify(r.payload)) {
      v_diffs.push({ index: i, original: o, replayed: r });
    }
  }
  return v_diffs;
}

// Usage: "The price was wrong on the 3pm order"
// Step 1: Extract
const v_packets = f_extractFromArchive(bus, {
  packetType: 'ORDER_HYDRATED',
  startTime: '2026-03-30T15:00:00',
  endTime: '2026-03-30T15:05:00',
});
// Step 2: Replay through the pricing engine
const v_replayResults = await f_replayThroughLoop(
  v_packets, createComputePricing, {}
);
// Step 3: Compare
const v_diffs = f_compareResults(v_originalOutputs, v_replayResults);
Constraints

Replay must use an isolated bus — never replay into the live system unless you explicitly intend to reprocess (and even then, use idempotency keys). The archive buffer has a finite size; packets older than the buffer window are gone. For mission-critical replay capability, increase the archive buffer depth or persist it to a Vault. Replay timing may not match original timing — async operations may resolve in different order. Compare payloads, not timing.

Related: Phantom (provides controlled loop behavior for replay testing), Archive Buffer (the data source for replay)
Convergent Pattern
Replay is a core concept in Event Sourcing, developed by Greg Young and the CQRS/ES community. The ability to reconstruct state by replaying a log of events is foundational to that approach. Loop MMT arrived at the same shape through the L21 session recording — the .loop file capturing every operator action for reconstruction — and implements it at the bus level rather than as an application-level event store. The problems addressed are the same: debug by inspecting the exact sequence of events, not reconstructed memory.
28. The Dead Letter
L21: The Pattern Matcher eject buffer when no bus is configured to drain it — data that matched but has nowhere to go sits in the bridge until the operator deals with it.
What It Is

A dedicated holding area for items that have exhausted all processing attempts. When a Signal Loop fails to send a notification after all retries, when a workflow step fails and the Operator resolves the escalation with "skip it," when a packet fails contract validation repeatedly — where does the failed item go? The Dead Letter queue holds it. It is not discarded. It is not retried automatically. It waits for the Operator to investigate and decide: fix and reprocess, or discard deliberately.

When to Use It

Any constellation with Signal Loops that deliver to external services (notifications, webhooks, email). Any constellation with Batch operations where partial failure is possible. Any operation where the cost of silently dropping a failed item is unacceptable but the cost of indefinite retries is also unacceptable. The Dead Letter queue is the third option between "retry forever" and "silently lose it."

How It Works

Implemented as a Vault Loop (or a dedicated table within an existing Vault). When an operation exhausts its retry limit, the failed item is written to the dead letter table with the original payload, error details, retry count, and timestamp. The Clock periodically sweeps the dead letter table and emits a health packet with the count. The Operator Alert loop fires when the count exceeds a threshold. The Operator reviews dead letters, fixes the root cause, and either reprocesses them (re-emit to the original pipeline) or archives them deliberately.

Code Example
// ── The Dead Letter: catch failed items, hold for operator ──

function f_sendToDeadLetter(bus, failedItem) {
  bus.vault.emit({
    type: 'DEAD_LETTER_WRITE',
    marker: 'live',
    payload: {
      id: f_generateId(),
      originalType: failedItem.type,
      originalPayload: failedItem.payload,
      failureReason: failedItem.error,
      retryCount: failedItem.retries,
      failedAt: new Date().toISOString(),
      source: failedItem.source,
      status: 'pending_review',
    },
    source: 'system:deadLetter',
  });
}

// In a Signal Loop — after retries exhaust
function createSignalNotifications({ bus, services }) {
  bus.signal.on('SEND_NOTIFICATION', async (packet) => {
    let v_retries = 0;
    const MAX_RETRIES = 3;

    while (v_retries < MAX_RETRIES) {
      try {
        await services.sms.send(packet.payload);
        bus.signal.emit({
          type: 'NOTIFICATION_SENT', marker: 'live',
          payload: { notificationId: packet.payload.id },
          source: 'signal:notifications',
        });
        return;
      } catch (err) {
        v_retries++;
        if (v_retries < MAX_RETRIES) {
          await new Promise((r) =>
            setTimeout(r, 1000 * Math.pow(2, v_retries)));
        }
      }
    }
    // All retries exhausted — dead letter
    f_sendToDeadLetter(bus, {
      type: 'SEND_NOTIFICATION', payload: packet.payload,
      error: 'Max retries exhausted', retries: MAX_RETRIES,
      source: 'signal:notifications',
    });
  });
}

// Operator reprocesses a dead letter
bus.data.on('REPROCESS_DEAD_LETTER', async (packet) => {
  const v_item = await bus.vault.pull({
    type: 'FETCH_DEAD_LETTER',
    payload: { id: packet.payload.deadLetterId },
  });
  if (v_item.marker !== 'live') return;

  // Re-emit to original pipeline
  bus.data.emit({
    type: v_item.payload.originalType, marker: 'live',
    payload: v_item.payload.originalPayload,
    source: 'system:deadLetter:reprocess',
  });

  // Mark reprocessed
  bus.vault.emit({
    type: 'DEAD_LETTER_UPDATE',
    payload: { id: v_item.payload.id, status: 'reprocessed' },
    source: 'system:deadLetter',
  });
});
Constraints

Dead letters are not a retry mechanism — they are a holding area for investigation. The dead letter table must capture the complete original payload so reprocessing is possible. Dead letters should have a Retention Gate — pending items are never auto-purged, but reprocessed and archived items can be. The Operator must be alerted when dead letters accumulate; a queue that grows silently defeats its purpose.

Related: Batch (partial failures produce dead letters), Replay (reprocessing is a targeted replay), Heartbeat Absence (absence of dead letter sweeps is itself a problem)
Adopted Pattern
The Dead Letter Queue is a named pattern from Enterprise Integration Patterns (Gregor Hohpe and Bobby Woolf, 2003) and is standard infrastructure in modern message brokers — RabbitMQ, AWS SQS, Azure Service Bus, and others all implement it under this name. Loop MMT adopts the pattern unchanged. The L21 analog (the PM eject buffer with nowhere to drain) is a genuine independent convergence, but the name and the concept belong to messaging systems literature. Credit where it is due.
Category 5
Infrastructure Patterns

Patterns for infrastructure components that support the constellation.

22. The Inlet
L21: External Bus E receiving data from outside the machine.
What It Is

A recognized sub-pattern of the Working Loop for continuous external data streams — WebSocket, Server-Sent Events, MQTT, or polling intervals on REST endpoints. Not a new loop type. It is a Working Loop with additional infrastructure for stream management: connection lifecycle, back-pressure handling, and stream health telemetry.

When to Use It

When the constellation receives continuous data from an external source: GPS fleet tracking, market data feeds, IoT sensor streams, real-time chat messages, live sports scores. Any data source that pushes to the constellation rather than waiting to be pulled.

How It Works

Implemented as a Working Loop with three additional concerns. Connection lifecycle: manages establishment, authentication, reconnection with exponential backoff, and graceful shutdown — all configured in the Constellation Spec. Back-pressure: when the stream is faster than the system can process, a declared strategy applies (buffer, sample, latest, or drop). Stream health: emits health packets with connection state, ingest rate, back-pressure state, and drop count.

Code Example
// ── The Inlet: GPS fleet tracking stream ──

const INLET_CONFIG = {
  name:       'inlet:gps',
  source:     { type: 'websocket', url: 'wss://fleet.example.com/positions' },
  auth:       { type: 'bearer', tokenFrom: 'config:fleet_api_key' },
  reconnect:  { backoff: 'exponential', maxDelay: 30000, maxRetries: 100 },
  backpressure: { strategy: 'latest', keyField: 'payload.vehicleId' },
  rateLimit:  { maxPerSecond: 100 },
};

function createWorkingInletGps({ bus }) {
  let v_ws = null;
  let v_retryCount = 0;
  let v_connected = false;
  const v_bridge = f_createBridge(
    INLET_CONFIG.backpressure.strategy, 100
  );

  function f_connect() {
    v_ws = new WebSocket(INLET_CONFIG.source.url);
    v_ws.onopen = () => {
      v_connected = true;
      v_retryCount = 0;
      bus.data.emit({
        type: 'HEALTH_HEARTBEAT',
        payload: { loop: 'inlet:gps', status: 'connected' },
        source: 'working:inletGps',
      });
    };

    v_ws.onmessage = (event) => {
      const v_raw = JSON.parse(event.data);
      // Translate external format to internal packet
      v_bridge.write({
        type: 'INTAKE_RAW',
        payload: {
          subtype: 'gps_position',
          vehicleId: v_raw.vid,
          lat: v_raw.lt,
          lng: v_raw.ln,
          speed: v_raw.spd,
          timestamp: v_raw.ts,
        },
        source: 'working:inletGps',
        operatorId: 'system:inlet:gps',
      });
    };

    v_ws.onclose = () => {
      v_connected = false;
      f_reconnect();
    };
  }

  function f_reconnect() {
    if (v_retryCount >= INLET_CONFIG.reconnect.maxRetries) {
      bus.signal.emit({
        type: 'DEPENDENCY_OFFLINE',
        payload: { source: 'inlet:gps', reason: 'max_retries_exceeded' },
        source: 'working:inletGps',
      });
      return;
    }
    const v_delay = Math.min(
      1000 * Math.pow(2, v_retryCount),
      INLET_CONFIG.reconnect.maxDelay
    );
    v_retryCount++;
    setTimeout(f_connect, v_delay);
  }

  // Drain bridge to bus at controlled rate
  setInterval(() => {
    v_bridge.drain((packet) => bus.data.emit(packet));
  }, 1000 / INLET_CONFIG.rateLimit.maxPerSecond);

  f_connect();
}
Constraints

Inlet-injected packets must pass through the Auth Gate with a system-level identity (operatorId: 'system:inlet:gps') — they are distinguishable from human actions in the audit trail. The reconnection logic must use exponential backoff with a maximum delay and a maximum retry count. After max retries, emit DEPENDENCY_OFFLINE and stop attempting. The back-pressure strategy is a design decision declared in the Constellation Spec — latest is right for position updates (only the current position matters), buffer is right for events that must not be lost.

Related: Bridge (Inlet uses Bridge for back-pressure), Translator (Inlet performs translation as part of ingestion), Heartbeat Absence (monitors Inlet health)
23. Document Generation
No direct L21 analog.
What It Is

A two-loop pattern for producing formatted output. A Compute Loop takes business data and produces formatted content (HTML template populated with order data, a PDF layout, a printable tag). A Signal Loop handles the delivery channel — print, download, email attachment. The Compute Loop produces; the Signal Loop delivers. Neither crosses into the other's responsibility.

When to Use It

When the application needs to produce formatted output: receipts, reports, printable labels, export files, email bodies. The formatting logic (how to lay out an order receipt) is a Compute concern. The delivery logic (how to send it as an email attachment) is a Signal concern. Separating them means adding a new delivery channel requires only a new Signal Loop subscription, not changes to formatting logic.

How It Works

The Workflow pipeline triggers the document generation. The Compute Loop receives the business data, applies a template, and emits a DOCUMENT_GENERATED packet with the formatted content. The Signal Loop subscribes and handles delivery. Multiple Signal Loops can subscribe for different channels (print, email, download) — this is a Fanout.

Code Example
// ── Document Generation: order receipt ──

// Compute Loop — produces the formatted content
function createComputeReceiptGenerator({ bus }) {
  bus.data.on('GENERATE_RECEIPT', async (packet) => {
    const v_order = packet.payload;

    // Apply template — produces HTML string
    const v_html = f_applyTemplate('receipt', {
      orderNumber:  v_order.orderId,
      customerName: v_order.customer.name,
      items:        v_order.items,
      totalPrice:   v_order.totalPrice,
      date:         new Date().toLocaleDateString(),
    });

    bus.data.emit({
      type: 'DOCUMENT_GENERATED',
      marker: 'live',
      payload: {
        documentType: 'receipt',
        format: 'html',
        content: v_html,
        metadata: { orderId: v_order.orderId },
      },
      source: 'compute:receiptGenerator',
    });
  });
}

// Signal Loop — handles print delivery
function createSignalPrint({ bus, services }) {
  bus.signal.on('DOCUMENT_GENERATED', (packet) => {
    if (packet.payload.documentType === 'receipt') {
      services.printer.print(packet.payload.content);
    }
  });
}

// Signal Loop — handles email delivery (separate loop, same event)
function createSignalEmail({ bus, services }) {
  bus.signal.on('DOCUMENT_GENERATED', (packet) => {
    if (packet.payload.documentType === 'receipt') {
      services.email.send({
        to: packet.payload.metadata.customerEmail,
        subject: 'Your Receipt',
        html: packet.payload.content,
      });
    }
  });
}
Constraints

The Compute Loop must not know about or reference delivery channels. If you find the receipt generator checking whether to email or print, the separation is broken. Templates should be stored in the Config Vault so they can be updated without code changes. The DOCUMENT_GENERATED packet should carry the content in a format-agnostic way (HTML is a good default) — individual Signal Loops convert to their channel's format if needed.

Related: Fanout (multiple delivery channels subscribe to the same document event), Translator (delivery Signal Loops may translate content format for their channel)
30. The Snapshot
L21: The operator pauses to read the full state of a loop — every word, every register — capturing a frozen picture of the machine at a specific moment.
What It Is

A point-in-time frozen copy of a Vault's state for reporting, backup, or migration. Different from a Projection (which is a live derived view that updates continuously). A Snapshot is a frozen copy at a specific moment — "what did the order book look like at end-of-day Thursday?" The output is immutable after it is taken.

When to Use It

End-of-period reporting: daily close, monthly reconciliation, season summary. Data migration preparation: freeze current state before migrating to a new schema. Disaster recovery checkpoints. Audit compliance: provable state at a specific regulatory deadline. Any situation where the question is "what was the exact state at time T?"

How It Works

Triggered by the Clock or by Operator request. A Compute Loop pulls the full current state from the target Vault, assembles it into a frozen dataset with timestamp and metadata, and emits a SNAPSHOT_COMPLETE packet. A Signal Loop or Vault Loop handles storage. The Vault data continues to change after the snapshot; the snapshot itself is immutable.

Code Example
// ── The Snapshot: end-of-day order freeze ──

const PIPELINE_DAILY_SNAPSHOT = {
  name: 'daily_order_snapshot',
  trigger: { type: 'CLOCK_TICK',
             match: { subtype: 'daily_snapshot' } },
  steps: [
    {
      name: 'fetch_all_orders',
      emit: { type: 'FETCH_ORDERS', bus: 'VAULT' },
      queryParams: { status: ['pending', 'in_progress', 'completed'],
                     dateRange: 'today' },
      expect: 'ORDERS_FETCHED',
      onFail: 'escalate',
    },
    {
      name: 'build_snapshot',
      emit: { type: 'BUILD_SNAPSHOT', bus: 'DATA' },
      needs: ['fetch_all_orders.payload'],
      expect: 'SNAPSHOT_BUILT',
      onFail: 'abort',
    },
    {
      name: 'persist_snapshot',
      emit: { type: 'STORE_SNAPSHOT', bus: 'VAULT' },
      needs: ['build_snapshot.payload'],
      expect: 'SNAPSHOT_STORED',
      onFail: 'retry', retryLimit: 3,
    },
  ],
};

// Compute Loop — assembles the frozen dataset
function createComputeSnapshotBuilder({ bus }) {
  bus.data.on('BUILD_SNAPSHOT', (packet) => {
    const v_orders = packet.payload;
    const v_snapshot = {
      snapshotId: `SNAP-${new Date().toISOString().slice(0, 10)}`,
      takenAt: new Date().toISOString(),
      type: 'daily_orders',
      recordCount: v_orders.length,
      summary: {
        totalOrders: v_orders.length,
        totalRevenue: v_orders.reduce(
          (sum, o) => sum + (o.totalPrice || 0), 0),
        byStatus: f_countByField(v_orders, 'status'),
      },
      records: v_orders, // The full frozen dataset
      metadata: {
        sourceVault: 'vault:orders',
        version: 1,
      },
    };

    bus.data.emit({
      type: 'SNAPSHOT_BUILT', marker: 'live',
      payload: v_snapshot,
      source: 'compute:snapshotBuilder',
    });
  });
}

// Vault Loop — persists the snapshot (separate table)
function createVaultSnapshots({ bus, db }) {
  bus.vault.on('STORE_SNAPSHOT', async (packet) => {
    const { error } = await db.from('snapshots').insert({
      snapshot_id: packet.payload.snapshotId,
      taken_at: packet.payload.takenAt,
      type: packet.payload.type,
      record_count: packet.payload.recordCount,
      summary: packet.payload.summary,
      records: packet.payload.records,
      metadata: packet.payload.metadata,
    });
    if (error) throw error;

    bus.vault.emit({
      type: 'SNAPSHOT_STORED', marker: 'live',
      payload: { snapshotId: packet.payload.snapshotId,
                 recordCount: packet.payload.recordCount },
      source: 'vault:snapshots',
    });
  });
}
Constraints

Snapshots must be immutable once taken — never update a snapshot record. Store snapshots in a dedicated table, not inline with business data. Apply a Retention Gate: daily snapshots retain for 90 days, monthly for 2 years. The snapshot must record its query parameters and source Vault so it can be understood independently. For large Vaults, use pagination (Batch pattern) to avoid timeout.

Related: Projection (live derived view vs. frozen point-in-time copy), Document Generation (snapshots feed report generation), Batch (large snapshots use pagination), Retention Gate (snapshots need lifecycle management)

Scale Patterns

Patterns that address testing, multi-system coordination, and deployment scenarios.

24. Self-Routing Packets
L21: Address-read mode on the Memory loop. The word's top four bits encode the destination slot. The data routes itself.
What It Is

A packet that carries routing refinement information in its payload. The bus reads a field to determine additional routing beyond what the routing table specifies. An ORDER_UPDATED packet carries a region field. A multi-region deployment routes to the correct regional Vault based on the field, without separate packet types per region.

When to Use It

Multi-tenant or multi-region deployments where the routing table declares the general path and the payload refines it. Rather than creating ORDER_UPDATED_NORTHEAST, ORDER_UPDATED_SOUTHEAST, etc., one ORDER_UPDATED type carries a region field and the bus uses it to select the correct regional Vault instance.

How It Works

The bus infrastructure supports a routeBy declaration in the routing table. When a packet matches, the bus reads the specified field from the payload and uses it to select among the available destinations. The producing loop emits a standard packet — it does not know about the routing refinement.

Code Example
// ── Self-Routing: region-based Vault selection ──

// Routing table entry with self-routing
const ROUTE_ORDER_UPDATED = {
  type: 'ORDER_UPDATED',
  bus: 'VAULT',
  source: 'compute:orderProcessing',
  // Self-routing: read 'region' from payload to select destination
  routeBy: {
    field: 'payload.region',
    destinations: {
      'northeast': 'vault:ordersNortheast',
      'southeast': 'vault:ordersSoutheast',
      'midwest':   'vault:ordersMidwest',
      'west':      'vault:ordersWest',
    },
    fallback: 'vault:ordersDefault',
  },
};

// The producing loop doesn't know about regions
function createComputeOrderProcessing({ bus }) {
  bus.data.on('ORDER_VALIDATED', (packet) => {
    bus.vault.emit({
      type: 'ORDER_UPDATED',
      marker: 'live',
      payload: {
        orderId: packet.payload.orderId,
        region: packet.payload.region, // This field drives routing
        ...packet.payload,
      },
      source: 'compute:orderProcessing',
    });
    // The bus handles the rest — routes to the correct regional Vault
  });
}
Constraints

The producing loop must not know about or depend on the self-routing mechanism. It emits a standard packet; the bus handles refinement. Always declare a fallback destination for payloads that don't match any route. Self-routing adds a field read to every matching packet delivery — for high-throughput scenarios, verify the performance impact. The self-routing field must be present in every packet of that type; missing fields route to the fallback.

Related: Scoped Subscription (per-subscriber filtering vs. per-packet routing)
25. The Phantom
No direct L21 analog. Derived from the factory + routing table architecture.
What It Is

A test double on the bus. During integration testing, a Phantom loop replaces a real loop — it subscribes to the same packet types and emits the same responses, but with controlled, deterministic behavior. Test a Workflow pipeline by replacing the Vault with a Phantom that always succeeds (happy path), always fails (fault injection), or fails on the third call (retry testing). Phantoms are wired through the constellation initializer by swapping the factory — the routing table and contracts don't change.

When to Use It

Integration testing and scenario testing. Happy path: all Phantoms succeed. Failure injection: specific Phantoms fail to test retry, abort, compensate, and escalate strategies. Latency simulation: Phantoms add delays to test timeout behavior. State verification: Phantoms record what they received so tests can assert on the exact packets that arrived.

How It Works

A Phantom factory creates a loop that subscribes to the same types as the real loop and emits responses matching the same contracts. The factory accepts a behavior configuration that controls success/failure/delay. The constellation initializer swaps the real factory for the Phantom factory — the routing table is unchanged, the contracts are unchanged, only the behavior changes.

Code Example
// ── The Phantom: test double for a Vault Loop ──

function createPhantomVault(behavior) {
  // behavior: { mode: 'success' | 'fail' | 'failOnNth',
  //             failOn: 3, delay: 0, recorded: [] }

  return function phantomVaultFactory({ bus }) {
    let v_callCount = 0;

    bus.vault.on('CREATE_ORDER', async (packet) => {
      v_callCount++;
      behavior.recorded.push({ type: 'CREATE_ORDER', payload: packet.payload,
                               callNumber: v_callCount });

      // Simulate latency if configured
      if (behavior.delay > 0) {
        await new Promise((r) => setTimeout(r, behavior.delay));
      }

      // Determine success or failure
      const v_shouldFail =
        behavior.mode === 'fail' ||
        (behavior.mode === 'failOnNth' && v_callCount === behavior.failOn);

      if (v_shouldFail) {
        bus.data.emit({
          type: 'LOOP_ERROR',
          payload: { loop: 'vault:orders', operation: 'CREATE_ORDER',
                     error: 'Phantom: simulated failure' },
          source: 'vault:orders', // Same source as real loop
        });
        return;
      }

      bus.vault.emit({
        type: 'ORDER_CREATED',
        marker: 'live',
        payload: {
          ...packet.payload,
          id: `phantom-${v_callCount}`,
          createdAt: new Date().toISOString(),
        },
        source: 'vault:orders',
      });
    });
  };
}

// ── Usage in tests ──

// Happy path
const v_behavior = { mode: 'success', delay: 0, recorded: [] };
const v_phantomFactory = createPhantomVault(v_behavior);
// Wire into initializer instead of real createVaultOrders
initConstellation({ 'vault:orders': v_phantomFactory });

// Retry testing — fails on the 3rd call
const v_retryBehavior = { mode: 'failOnNth', failOn: 3, delay: 0,
                           recorded: [] };
const v_retryPhantom = createPhantomVault(v_retryBehavior);

// After test: check what the Phantom received
console.assert(v_behavior.recorded.length === 1);
console.assert(v_behavior.recorded[0].type === 'CREATE_ORDER');
Constraints

Phantoms must emit packets with the same source as the real loop — otherwise downstream loops that filter by source will break. The Phantom's contracts must match the real loop's contracts exactly. The recorded array is the test's primary assertion mechanism — verify that the correct packets arrived with the correct payloads. Phantoms should be created by a factory function that accepts behavior configuration, not as one-off implementations per test.

Related: Replay (Phantom provides controlled behavior for replay testing), Workflow Engine (Phantom Vaults test workflow failure strategies)
26. The Tap
L21: The operator can observe any single bus by watching its shift register display without affecting its operation.
What It Is

A lightweight, selective observer. Where passive observation sees all traffic on a bus, a Tap filters to a single packet type or source. Useful for targeted debugging: "show me only PRICE_CALCULATED packets" or "show me everything emitted by workflow:cancel." The Tap is an observer with a filter predicate. It can be attached and detached at runtime without modifying any loop or route.

When to Use It

Debugging in development or production. "I want to see what the Pricing Engine is emitting right now." "I want to watch all LOOP_ERROR events for the next 5 minutes." "I want to log every packet that touches this specific order ID." The Tap is a runtime debugging tool — attach, observe, detach. No code changes, no restart.

How It Works

The bus infrastructure supports bus.tap(predicate, callback). The predicate filters traffic to matching packets. The callback receives matching packets. The tap returns a handle that can be detached. The Observatory UI can offer a "tap this type" interaction.

Code Example
// ── The Tap: runtime debugging tool ──

// Tap all PRICE_CALCULATED packets
const v_priceTap = bus.data.tap(
  (packet) => packet.type === 'PRICE_CALCULATED',
  (packet) => {
    console.log('[TAP] Price calculated:',
      packet.payload.orderId, packet.payload.totalPrice);
  }
);

// Tap everything from a specific source
const v_sourceTap = bus.data.tap(
  (packet) => packet.source === 'workflow:cancel',
  (packet) => {
    console.log('[TAP] Cancel workflow:', packet.type, packet.payload);
  }
);

// Tap a specific order's packets across all types
const v_orderTap = bus.data.tap(
  (packet) => packet.payload?.orderId === 'ORD-2026-0042',
  (packet) => {
    console.log('[TAP] Order 0042:', packet.type, packet.source);
  }
);

// Detach when done
v_priceTap.detach();
v_sourceTap.detach();
v_orderTap.detach();

// Time-limited tap — auto-detach after 5 minutes
function f_timedTap(bus, predicate, callback, durationMs) {
  const v_tap = bus.tap(predicate, callback);
  setTimeout(() => v_tap.detach(), durationMs);
  return v_tap;
}
Constraints

Taps are for debugging, not for business logic. If you find yourself building business logic that depends on a tap, it should be a subscription instead. Taps should be detachable — a tap that can't be removed is a memory leak. In production, taps should be time-limited (auto-detach after N minutes) to prevent forgotten taps accumulating. Taps use the passive observation path — they do not appear as routing destinations and do not affect delivery or priority.

Related: Passive Observer (Tap is a filtered observer), Sentinel (Sentinel is a persistent observer with analysis; Tap is a temporary observer for debugging)
27. Inter-Constellation Communication
L21: P2P buses F and G connect two machines directly over WebRTC. The CBX protocol coordinates multi-machine challenges. Each machine operates independently; the P2P buses are the shared channel.
What It Is

Two separate Loop MMT constellations that exchange data. Not multi-device sync (that is the Sync Bus) — two different applications talking to each other. The butcher constellation talks to an accounting constellation or a supply-ordering system. Each is a separate constellation with its own loops, buses, and spec. They communicate through a dedicated inter-constellation bus carrying a defined set of packet types — the shared protocol.

When to Use It

When your application ecosystem grows beyond a single constellation. The butcher app needs to send financial data to an accounting system. The order management system needs to talk to a supply chain system. Each system is independently developed and deployed. The inter-constellation bus is the contract boundary — governed by the same packet validation that governs every other bus.

How It Works

Each constellation declares the inter-constellation bus in its spec. The bus carries only the agreed-upon packet types — the shared protocol. Each constellation has a dedicated Signal Loop that translates internal events into the shared protocol and a dedicated Working Loop that receives shared protocol packets and translates them into internal events. The Translator pattern applies at both ends.

Code Example
// ── Inter-Constellation Communication ──
// Butcher constellation → Accounting constellation

// ── Shared protocol (agreed between both constellations) ──
const INTER_CONSTELLATION_CONTRACTS = {
  'FINANCIAL_TRANSACTION': {
    fields: {
      transactionId: 'string',
      type: 'string',         // 'sale' | 'refund' | 'adjustment'
      amountCents: 'number',
      currency: 'string',
      timestamp: 'string',
      sourceSystem: 'string',
      referenceId: 'string',  // Order ID in the source system
    },
  },
};

// ── Butcher constellation: outbound adapter ──
function createSignalAccountingBridge({ bus, services }) {
  const { interBus } = services; // The shared bus

  // Listen for internal events that need to go to accounting
  bus.signal.on('PAYMENT_CONFIRMED', (packet) => {
    // Translate internal event → shared protocol
    interBus.emit({
      type: 'FINANCIAL_TRANSACTION',
      payload: {
        transactionId: f_generateId(),
        type: 'sale',
        amountCents: packet.payload.amountCents,
        currency: packet.payload.currency,
        timestamp: packet.payload.confirmedAt,
        sourceSystem: 'butcher-constellation',
        referenceId: packet.payload.orderId,
      },
    });
  });
}

// ── Accounting constellation: inbound adapter ──
function createWorkingFinancialInlet({ bus }) {
  // Receives from the shared bus
  interBus.on('FINANCIAL_TRANSACTION', (packet) => {
    // Translate shared protocol → internal event
    bus.data.emit({
      type: 'INTAKE_RAW',
      payload: {
        subtype: 'external_transaction',
        ...packet.payload,
      },
      source: 'working:financialInlet',
      operatorId: 'system:interConstellation',
    });
  });
}
Constraints

Neither constellation has access to the other's internal buses, loops, or spec. The shared protocol is the only coupling. Both constellations use the Translator pattern at the boundary — internal field names and packet types do not leak across the boundary. The shared protocol must be versioned independently of either constellation's internal contracts. Changes to the shared protocol require coordination between both systems' operators — this is the one place where independent development is constrained.

Related: Translator (used at both ends of the bridge), Inlet (the receiving end is an Inlet-like Working Loop), Dependency Gate (the shared bus is an external dependency)
On Patterns and the L21 Model
Twenty-four of the thirty patterns trace directly to a specific Loop 2.1 hardware component or operating technique. This is not forced mapping — it is evidence that the hardware model captures fundamental computational patterns that recur at every level of abstraction. A marker bit and a packet liveness indicator solve the same problem in different media. A clamp gate and a bounds-enforcing filter serve the same purpose at different scales. The machine's design, because it is minimal and observable, exposes patterns that conventional software buries under layers of abstraction. Loop MMT names them and makes them explicit.

Three patterns — Debounce, Dead Letter, and Outbox — are adopted from prior software engineering practice without claiming independent invention. The L21 hardware offers genuine analogs for two of them, but the names and concepts were already established before Loop MMT existed. Good patterns are worth adopting. Pretending otherwise would be inaccurate and unnecessary.
