About This Guide

This is the deep-dive reference for developing a Loop MMT™ project. It complements the Interaction Guide — the IG defines how sessions work, the Developer Guide goes deeper on phases, roles, and operational procedures. If the IG is the operating manual, this is the technical companion you open when you need more detail on a specific phase, role, or failure mode.

For the session cycle, handoff format, conversation types, and loading guidance, see the Interaction Guide. For the document registry and dependency graph, see Standard §17. This guide assumes those documents are available and focuses on what they do not cover: phase walkthroughs with completion criteria, deep operational guidance for each role, the Operator's routing workflow, document hygiene procedures, and failure modes unique to multi-conversation coordination.

This guide is v2 and pre-dates a completed real project — it will be updated after the Butcher Constellation run produces lived experience.

Section 1
The Foundation
The Mental Model

A Loop MMT application is a constellation — a set of independent processing loops connected by typed message buses. No loop touches another directly. They communicate only by publishing and subscribing to packets on the buses. Every data path is explicit. Every transfer is observable. Every failure is contained.

The development process mirrors this architecture. AI conversations are loops. Documents are the bus. The Operator routes between them. Just as the application's loops are isolated and communicate through declared contracts, the AI conversations are isolated and communicate through documents the Operator carries. The same discipline that makes the runtime safe makes the development process auditable.

The Core Constraint
The AI forgets everything when a conversation ends. Documents are the only thing that survives. The Operator's job — above all else — is to ensure that no important decision, architectural choice, or status update lives only in a conversation. Everything that matters goes in a document before the session closes.
The Seven Loop Types

Every loop in a constellation is one of seven types. The type determines what capabilities it receives, what buses it can access, and what it is allowed to do. Types are enforced structurally through JavaScript closures — a loop cannot access capabilities it was not given, and it has no way to acquire them at runtime.

TypeReceivesDoes
Working bus only The entry point for all external inputs (user forms, API calls, Clock, Inlet). Ingests, validates structure, emits to bus. Does not compute, store, or render.
Vault bus + db All database operations. Creates, reads, updates, archives records. The only loop type with database access. Handles schema migrations on startup.
Compute bus + config Pure calculation. Price computation, report generation, formatted document production. No database access, no side effects beyond emitting results.
Filter bus only Validation, transformation, enrichment. Takes a packet, returns a modified or rejected version. No storage, no computation beyond field manipulation.
Signal bus + notifier Outbound communications — email, SMS, print, push notifications. One conversation type: send a message to an external recipient. Never stores state, never returns data to the system.
Presentation bus + dom Renders the UI. Subscribes to state updates, manages DOM, serves the human interface. Cannot write to the database. Can only request operations by publishing to the bus.
Workflow bus + wf Orchestrates multi-step operations using declared pipeline specs. Manages idempotency, step sequencing, failure recovery. Never executes steps — it coordinates other loops to execute them.
The Closure Wall
Capability enforcement is structural, not policy-based. A Compute Loop that doesn't have db in its closure cannot query the database — there is no function call to make. This is not a convention you can forget to follow. It is an architectural constraint that holds regardless of what any individual session decides.
The Bus System

Four named buses handle all inter-loop communication. Every packet travels on exactly one bus. The bus it travels on is declared in the routing table — not decided at runtime.

BusCarries
DATAApplication data flow — intake, validation, processing, results.
VAULTPersistence operations and their responses — fetch requests, record results.
SIGNALOutbound communication events — notifications, alerts, print jobs.
SYNCCross-device and real-time coordination — system events, heartbeats, config updates.

Buses support scoped subscriptions (a Presentation Loop subscribes only to packets relevant to its view), authoritative loop designation (one loop owns a packet type — if two loops try to handle the same packet, this resolves the conflict), and extensible query types. Additional buses beyond the standard four can be declared in the Constellation Spec when a project needs them.

The Ten Practices

These are the operating principles of Loop MMT development. They are not suggestions — they encode the lessons of 320+ builds that shaped this methodology.

  1. Read before you write. Load the relevant spec, contracts, and coding standards before building anything. The AI cannot know what it was not given.
  2. Talk first, code second. Every build starts with a conversation about what is about to be built. The code phase begins only when both parties agree on exactly what will be produced.
  3. Plan in phases, deliver in builds. Work is structured into phases with defined completion criteria. Each phase produces concrete deliverables that can be verified before the next phase begins.
  4. The Verification Gantry is a fixed gate. Tests are not optional and not skippable. A deliverable is not done until it passes. There are no exceptions.
  5. Code is the persistence layer for architecture. If an architectural decision was made in a conversation but not reflected in a document or the code, it will be lost. Architecture lives in artifacts, not in chat history.
  6. The Operator owns decisions; pushback is a deliverable. The AI's job is to surface problems, not to silently comply. When the AI identifies a conflict or risk, saying so is part of the work. The Operator then decides — but the AI flags it first.
  7. Documentation is part of the product. The handoff document, the decision log, the event ledger — these are not overhead. They are what makes the project recoverable and the codebase maintainable.
  8. Verify against source, not memory. When the AI needs to check a field name, a routing entry, or a contract — it reads the document. It does not rely on what it remembers from earlier in the session. Memory drifts. Documents don't.
  9. Design around the memory boundary. Every session ends with a clean state transfer. Never rely on context that will not survive the session close.
  10. This method is not a shortcut. Loop MMT produces better software more reliably, and the structure that feels like overhead upfront is what prevents the debugging and re-implementation cycles that consume far more time later. Phase 0 takes time. It pays back many times over.
Section 2
The Five Roles

Five roles participate in a Loop MMT project. One is human. Four are AI conversations. Each holds different information, operates under different constraints, and does different work. They do not communicate with each other — only with you.

Operator — You
You make all architectural decisions. You evaluate quality. You approve spec changes. You route documents between conversations. You work the decision queue. You do not write code, and you do not maintain project documents mechanically — the Coordinator handles that. What you cannot delegate: judgment about whether something is the right thing to build, and ownership of anything that affects the architecture.
Coordinator
Maintains the event ledger, tracks module status, triages Integrator findings, surfaces decisions to the Operator, and prepares routing manifests. Manages flow. Does not write code, verify code, or make architectural decisions. The Coordinator is your project logistics layer — it tracks what is happening and prepares the information packages you need to act.
Integrator
Holds the full Constellation Spec, the contract registry, and the routing table. Verifies deliverables against the spec — does code match the contracts? Do subscriptions match the routing table? Reports mismatches with category and confidence tags. Does not write code, modify implementations, or make decisions. Manages correctness.
Auditor
Receives documents fresh each check cycle — holds no state between cycles. Runs mechanical reconciliation checks comparing documents that should agree. Produces pass/fail reports with specific discrepancies. Does not recommend corrective action, hold state, or communicate with any other role. Manages consistency across the document set.
Module Workers
Hold only their scoped loading pack. Build the code for their assigned module(s). Cannot see other modules, cannot modify the spec, cannot communicate with other worker conversations. Their context is exactly what they need and nothing more. Multiple module conversations run in parallel during Phase 2.
Why Separate Roles
Combining the Coordinator and Integrator in a single conversation would create a conflict of interest — the entity tracking schedule would also be deciding whether deliverables pass. Combining the Auditor with either would allow the checked entity to influence its own audit. Separation is structural, not organizational. Each role sees exactly what it needs and nothing that would compromise its function.

The Auditor Operating Instructions define the reconciliation check suite, report format, loading protocol, and boundary constraints. The Auditor role is available at phase gates and on-demand when the Operator needs a consistency check across the document set.

Section 3
The Session Cycle

Every interaction between the Operator and the AI follows a six-step cycle: Establish State → Talk → Build → Verify → Deliver → Handoff. This is the atomic unit of work in Loop MMT. It applies to every conversation type without exception. The content changes by phase and role. The shape does not.

For the full six-step walkthrough, see Interaction Guide §4. For the handoff document format, template, and worked examples, see the Handoff Standard. This section covers what those documents do not: the talk-first protocol in detail and the session identity system.

The Talk-First Protocol

Talk-first is not optional. It prevents the most common and most expensive failure mode: the AI implementing a reasonable interpretation of an ambiguous request, delivering it, and the Operator discovering it wasn't what they wanted. The more complex the task, the more expensive the skip.

StepWhoWhat Happens
1OperatorStates the objective. What needs to be built and why.
2AIDescribes its understanding of the problem and the scope of the work.
3OperatorCorrects, refines, or confirms. Narrows scope if needed.
4AIProposes implementation approach — strategy, not code. "I will build X by doing Y, using Z pattern."
5OperatorApproves or pushes back. Raises concerns. Asks about alternatives.
6AIIdentifies edge cases, conflicts with existing code, and open questions.
7OperatorResolves edge cases and open questions. Makes design decisions.

The talk phase scales with complexity. A simple bug fix with a known cause might be two messages. A new Workflow Loop with multiple failure strategies and cross-module dependencies might be twenty messages of design discussion. Calibrate to the task. The principle is the same at any scale: don't build until you know what you're building.

Session Identity

Every conversation carries a session ID that encodes role, date, time, and sequence. Format: ROLE-YYYYMMDD-HHMM-SEQ

PrefixRoleExample
SPECSpecification (Phase 0)SPEC-20260401-0900-A
COORDCoordinatorCOORD-20260408-0800-A
MOD-xxxModule WorkerMOD-AUTH-20260412-1530-A
INTGIntegratorINTG-20260413-0900-A
AUDITAuditorAUDIT-20260415-1400-A
BUGBug FixBUG-20260420-1100-A

Session IDs sort chronologically. Every event ledger entry carries the session ID that produced it. Cross-session references use the fully qualified form: COORD-20260408-0800-A:E-047 means event 47 from that Coordinator session. This enables full traceability — any event can be traced backward to its cause and forward to its consequences.

Use the sequence letter (A, B, C…) to distinguish multiple sessions of the same role on the same day. The first session of the day is always A.

Phase 0
Specification
No code is written. Everything is documents. The purpose is to produce a spec precise enough that code can be written against it and verified against it.

Phase 0 is a single conversation — or multiple sessions of the same conversation type — between the Operator and the AI. Load the Standard (relevant sections), the Glossary, and the Coding Standards. Describe the application. Let the AI ask questions. Don't describe architecture in your opening — describe the problem. Architecture comes from the conversation.

Phase 0 Can Run Indefinitely
If every answer opens new questions, Phase 0 never ends. When the same category of question recurs without resolution, name it as an open decision and move forward. Phase 0 produces a buildable spec — not a perfect one. Decisions that don't affect the architecture can be deferred to Phase 5.
The Ten Steps of Phase 0
1
Describe the Application
Tell the AI what the software needs to accomplish. Who uses it. What problems it solves. What the primary workflows are. Not how it should be built — what it needs to do. The AI will ask questions. Answer them. This is the talk phase at the project level.
2
Identify the Loops
Which loop types are needed and what each one does. Every external input needs a Working Loop. Every database table needs a Vault Loop. Every validation or transformation needs a Filter or Compute Loop. Every notification needs a Signal Loop. Every UI view needs a Presentation Loop. Every multi-step operation needs a Workflow Loop. The question is which ones this project needs and how many of each.
3
Define the Buses
The four standard buses (DATA, VAULT, SIGNAL, SYNC) cover most projects. Decide whether additional buses are needed. If in doubt, start with the standard four — adding buses mid-project is a spec change that cascades through loading packs.
4
Build the Routing Table
Every legal data path in the system is declared. If a movement isn't in the routing table, it doesn't happen. For each entry: the packet type, the source loop, the destination loop (or subscription), and the bus it travels on. This is where the architecture becomes concrete. Expect this step to take the most time.
5
Define the Packet Contracts
Every packet type the system can carry gets a registered contract: field names, field types, required vs. optional, validation rules, PII declarations. The Integrator will verify every module deliverable against these contracts exactly. Imprecision here becomes mismatches in Phase 2.
6
Write the Pipeline Specs
Every multi-step operation needs a pipeline spec: the steps, their order, dependencies, timeouts, and failure strategies. Failure strategies use exactly five words: abort, retry, skip, compensate, escalate. No other vocabulary. For each step in each pipeline, one of these five words is the declared failure behavior.
7
Settle Architecture-Affecting Decisions
Any business logic question that shapes a loop or a route must be answered before Phase 1. Payment models, status state machines, role definitions, concurrency strategies, retention policies. Decisions that don't affect the architecture can be deferred. Do not start Phase 1 with open architectural questions.
8
Produce the Constellation Spec
All of the above assembled into a single document. This is the master reference for the project. Every conversation from Phase 1 onward derives its context from this document. The AI drafts it; you review and approve. Approval is reflected by the version number increment.

The Phase 0 Specification conversation produces two kinds of output. The architecture — loops, buses, routes, contracts, pipelines — goes in the Constellation Spec. The reasoning behind specific design decisions — why this approach, what was rejected, what constraints drove the choice — goes in Technical Design Documents. The spec declares what the system is. The TDD records why it is that way. Both are produced during Phase 0. The spec is the authority that module conversations and the Integrator verify against. The TDD is the reasoning that future sessions load when they need to understand a design choice. If a TDD's decision affects the spec, both must be updated — a decision that lives only in a TDD is a decision the Integrator will never know about.

9
Produce the Action Plan
Based on the Constellation Spec: how many phases, how many parallel conversations in Phase 2, what each conversation builds, what the dependencies are between conversations, and the delivery criteria for each phase. This becomes the Coordinator's operational map.
10
Extract the Conversation Loading Packs
One pack per planned conversation. Each pack is a version-pinned slice of the document stack — the Coding Standards, the bus API reference, the routing entries for that conversation's scope, the contracts for packets it emits or consumes. Test each pack for context fit: can it be loaded alongside the code that will be written without exceeding context limits? If not, split the conversation.
Phase 0 Completion Criteria

Phase 0 is done when all five of the following are true. If any are missing, Phase 1 cannot start.

  • The Constellation Spec exists and is internally consistent — no routing table entry references a loop that isn't declared, no packet type appears in a contract that isn't in the routing table.
  • The Action Plan exists with phases defined and conversation assignments made.
  • The Conversation Loading Packs are extracted, version-pinned, and tested for context fit.
  • All architecture-affecting decisions are settled and recorded.
  • The Coordinator conversation is opened, loaded with the Phase 0 outputs, and has confirmed its initialization.

The Coordinator takes custody of project state from the moment it initializes. From Phase 1 onward, the Coordinator is the single source of project status truth. Not your memory. Not a chat log. The Coordinator's ledger.

Phase 1
Infrastructure
One conversation. Builds the foundation everything else depends on. No modules until infrastructure passes.

Phase 1 is a single infrastructure conversation. Unlike Phase 2 module conversations, it receives the full Constellation Spec because infrastructure is deeply interdependent — the bus system, gates, workflow engine, and initializer all need to understand the full topology. Module conversations in Phase 2 receive only their scoped loading pack. The infrastructure conversation needs the whole picture.

What Gets Built in Phase 1
  • The Bus System — DATA, VAULT, SIGNAL, SYNC buses, plus any project-specific buses declared in the spec.
  • The Dependency Gate — Wraps every external dependency (database, auth provider, external APIs). Handles buffered mode when a dependency goes offline. Governs graceful degradation.
  • The Auth Gate — Enforces the role-permission matrix declared in the Constellation Spec. Stamps every packet with the operator's role before it enters the working loops. Enforces permissions at the same architectural level as the closure wall.
  • The Data Gate — Manages data lifecycle: retention periods, archive triggers, purge schedules. Declared in the spec and enforced structurally.
  • The Workflow Engine — Supports all five failure vocabulary words (abort, retry, skip, compensate, escalate), idempotency key management, and cascade update logic.
  • The Clock — Scheduled event emitter. Reads a schedule table from the spec and injects packets at declared times. Clock-emitted packets are indistinguishable from user-initiated packets once on the bus.
  • The Constellation Initializer with Self-Verification — Verifies structural integrity before accepting any traffic: every declared loop is instantiated, every bus is active, every route has a live subscriber, every packet contract is loaded. If any check fails, the constellation does not start.
What to Load

Infrastructure is the exception to the scoped-loading rule. Load the full Constellation Spec, the full Coding Standards, the full routing table, the full packet contract registry, and the test suite framework. Infrastructure needs to support everything — it cannot be built against a partial picture.

Phase 1 Completion Criteria
  • All infrastructure components are built and individually tested.
  • The Constellation Initializer runs and passes self-verification against the full spec.
  • All gantry tests pass green.
  • The Auditor (or manual equivalent) has checked infrastructure against the spec — no routing mismatches, no contract violations, no naming violations.
  • The Coordinator has logged Phase 1 complete in the event ledger.
  • The handoff explicitly confirms readiness to begin Phase 2.
Do Not Begin Phase 2 Until Phase 1 Passes
Module conversations in Phase 2 depend on infrastructure being correct. A bus system that doesn't fully implement the routing table will cause cascading failures across every module conversation. Infrastructure errors discovered mid-Phase 2 are expensive — they require every module conversation that touched the affected infrastructure to be re-evaluated.
Phase 2
Build
Multiple conversations in parallel. Each builds a scoped set of modules. The Coordinator tracks everything. The Integrator verifies deliverables as they arrive.

Phase 2 is where most of the code gets written. Multiple module conversations run in parallel — each builds a different piece of the system — without drifting apart. That isolation without drift is what the loading pack architecture is designed to produce.

Loading Packs

Each module conversation receives exactly its loading pack and nothing more. A loading pack contains:

  • The Coding Standards (always)
  • The bus API reference
  • The routing table entries for this module's scope only — not the full routing table
  • The packet contracts for packets this module emits or consumes — not the full contract registry
  • The capability injection spec for this module's loop type(s)
  • Any relevant pipeline specs for workflows this module orchestrates

Not included: the full Constellation Spec, other modules' implementations, other modules' contracts, the full routing table. Context is finite. Loading more than a conversation needs does not help it — it slows the session and can introduce confusion.

Loading Pack Version Pinning
Each pack declares the exact version of every document it includes — not just "the Glossary" but which version. When a pinned document advances, the pack is stale. A session that opens with a stale pack will produce work built on shifted ground. The Coordinator tracks this through the ledger and flags stale packs. Always check the Coordinator's status before opening a module session.
Running Parallel Work

The Action Plan assigns conversations to specific modules. The dependency graph in the Action Plan tells you which conversations can run in parallel and which must wait for others to complete. A module that publishes a packet type that another module subscribes to may need its contract finalized before the subscriber conversation can proceed — but the implementations themselves can usually run in parallel.

Your role during Phase 2: Open module conversations in the order the Action Plan specifies. Deliver completed modules to the Integrator. Deliver Integrator findings to the Coordinator. Deliver Coordinator routing manifests to module conversations. Work the decision queue the Coordinator surfaces. You are the only communication path between all of these conversations.

The session cycle for each module:

  1. Open the conversation with its Loading Pack using the Module opening template from the Prompt Templates document.
  2. Run the talk-first protocol before any code is written.
  3. Build and verify (gantry green).
  4. Deliver the versioned module code to the Integrator conversation.
  5. The Integrator produces a verification report. Bring findings back to the Coordinator.
  6. The Coordinator triages findings, prepares routing manifests for fixes, and logs status in the ledger.
  7. Deliver routing manifests to the relevant module conversations for remediation.
  8. Write the module handoff at session close.
The Integrator Is Not Optional
Every module deliverable goes to the Integrator before it is considered accepted. The module conversation cannot see the full spec — it can only verify its own contracts. The Integrator can see conflicts between modules that neither module conversation can detect. Skipping Integrator verification means accepting deliverables that may contain cross-module contract violations that won't surface until Phase 3 integration, where they are much more expensive to fix.
Bug Fix Conversations

When the Integrator identifies a defect, or when Operator testing finds one, a Bug Fix conversation handles it. One defect per conversation by default. If multiple bugs are found in the same module, assess whether they are related before combining them — unrelated bugs in a single session multiply risk.

What to load: The module's loading pack at its current version, the bug report (from the Integrator finding ID, test failure log, or Operator observation), and the current module code. Nothing else.

Critical constraint: Fix the defect without changing the module's public interface unless the Integrator has explicitly approved a contract change. Interface changes affect every module that subscribes to or publishes this module's packets — the Coordinator must log the change and flag all downstream loading packs as stale.

Phase 2 Completion Criteria
  • All modules listed in the Action Plan are delivered and Integrator-verified.
  • No open Integrator findings remain unresolved.
  • All module gantry tests pass in isolation.
  • The Coordinator's module status index shows all modules at "Verified" status.
  • An Auditor check cycle has been run at Phase 2 close — all documents consistent.
Phase 3
Integration
Assembly. All modules come together with the infrastructure and are tested as a system.

Phase 3 uses the Integrator conversation in its assembly mode — rather than verifying individual deliverables, it now orchestrates the full assembly and end-to-end scenario testing. The Coordinator tracks integration status.

What Happens in Phase 3

Assembly. All module code is assembled with the infrastructure in a single codebase. The Constellation Initializer runs self-verification against the full assembled system. If the initializer doesn't pass, Phase 3 cannot proceed — a self-verification failure indicates a wiring error in the assembly itself, not just in a module.

Scenario testing. The workflows defined in the Constellation Spec are exercised end-to-end. Not just "does the code run" but "does the system do what was designed." Take each workflow in the spec and trace it through the assembled constellation. Verify that packets travel the routes declared in the routing table. Verify that contract fields match what the spec declares. Verify that pipeline steps execute in the declared order with the declared failure behaviors.

Failure injection. Deliberately inject failure conditions to verify the system degrades the way it was designed. If a Dependency Gate is supposed to buffer operations when the database goes offline, take the database offline and verify it buffers. If a Workflow Loop is supposed to compensate on a step failure, force that step to fail and verify compensation runs. Failure paths that aren't tested are unknown risks.

Cross-module findings. Integration testing will surface issues that module-level testing couldn't — race conditions between loops, timing edge cases in Workflow Loop pipelines, Presentation Loop state management under concurrent updates. These go to the relevant module conversations for fixes, routed through the Coordinator.

Phase 3 Completion Criteria
  • Constellation Initializer passes self-verification on the fully assembled system.
  • All primary workflows from the Constellation Spec pass end-to-end testing.
  • All declared failure modes have been injected and verified.
  • No open integration findings remain unresolved.
  • The Coordinator's event ledger shows all integration findings resolved or explicitly deferred with Operator acknowledgment.
Phase 4
Verification
The full verification gantry runs. Every test, every contract, every routing entry, every constraint — against the final integrated codebase. This gate is fixed. It does not negotiate.

Phase 4 is not another round of testing — it is the final pass of the complete verification suite against the fully integrated codebase. Everything that was checked module-by-module in Phase 2 is now checked against the whole system.

The Verification Gantry

The gantry checks:

  • All tests pass (the full test suite, not just module-level tests)
  • All packet contracts match their declared schemas exactly
  • All routing table entries have live subscribers
  • All capability constraints are satisfied (no loop uses capabilities it wasn't given)
  • All naming follows the prefix taxonomy from the Coding Standards
  • All factory signatures match the capability injection table
  • The Constellation Initializer self-verification passes on the final build

Phase 4 is done when the gantry is green. Not when it's mostly green. Not when the failing tests are "just edge cases." There are no exceptions, no skips, no deferrals. A failing gantry means Phase 4 is not done. Fix the failures and run it again.

The Gantry Is the Gate
The methodology has exactly one hard gate. This is it. The previous phases all have completion criteria and Auditor checkpoints, but they involve judgment — is the spec good enough, are the decisions settled enough, is the integration testing thorough enough? The gantry is not judgment. It either passes or it doesn't. A green gantry is the only acceptable exit condition from Phase 4.
When the Gantry Fails

Failures at this stage are either integration failures (something that worked in isolation fails in the assembled system) or regression failures (a fix in Phase 3 broke something that was passing). Route failures through the Coordinator to the appropriate Bug Fix conversation. Gantry failures at Phase 4 are expected — address them systematically, one conversation per defect.

Phase 4 Completion Criteria
  • Full gantry green — every test, every check, every constraint.
  • Final Auditor reconciliation check complete — all documents consistent with the final codebase.
  • The Decision Log is complete — every decision made during the project is recorded with rationale.
  • The Coordinator has produced a Phase 4 completion status to the event ledger.
Phase 5
Ship
UI refinement, performance work, documentation completion, cosmetic deferred items. No architectural changes. The project ends with a complete, auditable archive.

Phase 5 is deliberate finishing work, not a continuation of Phase 4. The gantry is green. The architecture is settled. Phase 5 is where the product gets the attention that was deferred to keep earlier phases moving.

What Happens in Phase 5

UI refinement. Presentation Loop polish — visual alignment, responsive behavior, accessibility improvements, error state messaging. No architectural changes: the Presentation Loop emits to and subscribes from the bus the same way it did when the gantry passed. Cosmetic changes only.

Performance work. If performance issues were observed during Phase 3 or 4 testing and deferred, address them now. Profiling, query optimization, subscription debouncing. All changes go through the full session cycle — talk-first, gantry-verified.

Documentation completion. Inline code comments, README updates, any operational documentation the system needs. The methodology's documentation is already complete (the event ledger, the decision log, the Constellation Spec). This is the application-facing documentation.

Final archive production. The Coordinator produces the project archive: the complete event ledger in its final state, all materialized views (progress dashboard, decision log, session log, dependency graph), and the final Constellation Spec. This archive is the complete, auditable record of what was built and every decision that shaped it. Store it with the codebase.

No Architectural Changes in Phase 5
Phase 5 is not the place to reconsider the routing table, add a new loop type, or change a packet contract. If a Phase 5 change requires touching the spec, it is not a Phase 5 change — it is a new development cycle. Log it, schedule it, and start Phase 0 for that work separately. The current ship stays clean.
Phase 5 Completion Criteria
  • All deferred cosmetic and performance items are resolved or explicitly accepted as known limitations.
  • Application documentation is complete.
  • The project archive is produced and stored.
  • The Coordinator has produced the final event ledger artifact and all materialized views.
  • The constellation initializer self-verification passes on the final shipped build.
Section 10
Working the Coordinator

The Coordinator is your project logistics layer. It does not make architectural decisions. It does not write code. It does not verify code. What it does: maintains the event ledger, tracks module status, triages Integrator findings, prepares routing manifests, surfaces decisions to you in a priority-ordered queue, and manages document dependencies. Every significant thing that happens in the project is an event in the Coordinator's ledger.

Opening a Fresh Coordinator

The Coordinator opens once — at the end of Phase 0, after the Constellation Spec, Action Plan, and Loading Packs are finalized. Use the COORD opening template from the Prompt Templates document. Load:

  • Coordinator Operating Instructions (current version per registry)
  • Action Plan vX (current)
  • Constellation Spec — module list, routing summary, contract registry (not the full implementation detail sections)
  • Loading Pack templates for all planned module conversations

The Coordinator confirms its session ID, initializes the event ledger, builds its initial module status index, and reports its operational parameters (auto-routing threshold, verbosity level). Do not proceed until initialization is confirmed.

Restarting the Coordinator

The Coordinator's conversation will eventually hit context limits and need to restart. When this happens:

  1. Before the current session closes, instruct the Coordinator to produce a State Export — the complete serialized state needed to reconstruct the session from scratch.
  2. Open a new COORD conversation with the Coordinator Operating Instructions and the State Export only. The State Export contains everything else. Do not reload the Action Plan or Constellation Spec separately — the State Export already incorporates the relevant summaries.
  3. The new session initializes from the State Export and confirms the reconstructed state before accepting new events.

The most recent State Export is the recovery point for any Coordinator failure. Produce one before every context-limit restart and before any operation you want to be able to roll back to.

The Event Ledger

The event ledger is append-only. Nothing in it is ever edited. If a status was recorded incorrectly, the correction is a new event — both the original and the correction remain. Every event carries:

  • A sequential ID
  • A timestamp
  • An event type
  • The session ID that produced it
  • A payload
  • A triggered-by reference tracing the causal chain

The Progress Dashboard, Decision Log, Session Log, Dependency Graph, and Routing History are all materialized views derived from the ledger. They are convenient summaries. The ledger is the source of truth. When a materialized view conflicts with the ledger, the ledger is correct.

Cross-session references use the fully qualified form: COORD-20260408-0800-A:E-047. Use these when routing a finding or decision — the Coordinator needs the exact event reference, not a description.

The Decision Queue

Decisions requiring your attention are collected into a priority-ordered queue. The Coordinator surfaces them in minimal verbosity by default — you request more context on any item. Priority levels:

PriorityWhat It Means
P1Blocking other work. Resolve before proceeding with any dependent module conversations.
P2Spec-affecting. Resolution will require a spec change and loading pack updates.
P3Cross-module. Affects more than one conversation but not the spec itself.
P4Module-scoped. Can be resolved within the relevant module conversation.

Work the queue from P1 down. A P1 decision that stays unresolved blocks the project. P4 decisions can accumulate and be batched into a module session. The Coordinator never makes a decision — it presents options and context. You decide.

The Divergence Circuit Breaker

If a single Auditor check cycle produces three or more failures, routing pauses. The Coordinator stops routing findings, spec changes, and new deliverables. Module conversations actively building continue; the Coordinator queues incoming events instead of processing them.

When the circuit breaker trips: review the Auditor report, diagnose the root cause of the failures, fix the inconsistencies in the appropriate conversations, and release the pause. The Coordinator logs the trip and the release as separate ledger events. Do not release the pause without diagnosing the failures — the threshold exists because three simultaneous failures indicates a systemic problem, not isolated defects.

The threshold (default: 3) is calibrated per project. A project with very tight contracts and frequent Auditor cycles might warrant a higher threshold. A project running in first-time territory should keep it at 3.

Section 11
Working the Integrator

The Integrator holds the full Constellation Spec and verifies deliverables against it. Its output is a structured report with findings categorized by type and tagged with confidence levels. It does not write code. It does not make decisions. It reports mismatches.

What to Load

The Integrator needs the full picture to do its job. Load:

  • Full Constellation Spec at its current version
  • The complete routing table
  • The complete packet contract registry
  • The module code being verified
  • The module's loading pack (so the Integrator can check that the module built against the right version)
Understanding Integrator Reports

Every finding carries two tags:

Category:

  • mechanical — a verifiable factual mismatch (field name doesn't match contract, subscription not in routing table, naming violates taxonomy)
  • judgment — a concern that requires interpretation (implementation approach seems to violate a principle without a clean spec reference)
  • architectural — something that affects the system beyond this module (a contract change that would break other modules, a routing gap)

Confidence:

  • high — the Integrator is certain; there is a clear spec reference for the mismatch
  • low — the Integrator is inferring; the spec is ambiguous on this point

The Coordinator auto-routes mechanical / high findings to the relevant module conversation for remediation. All other categories go to you. When in doubt, the Coordinator escalates — false escalation costs you a decision-queue message; false auto-routing costs a module conversation building against wrong instructions.

What the Integrator Does Not Do

The Integrator does not recommend corrective action. It reports mismatches. When you receive a finding, bring it to the appropriate conversation — the Coordinator tells you where. Do not ask the Integrator how to fix what it found. That decision belongs to you (for architectural findings) or to the module conversation (for mechanical findings).

Section 12
Working the Auditor

The Auditor runs intermittent reconciliation checks comparing documents that should agree. It holds no state between cycles — it receives documents fresh each time. Its job is to catch cross-document inconsistencies that accumulate between Coordinator operations: stale loading packs, undocumented spec changes, missing dependency entries, deliverables received but never verified.

Auditor Operating Instructions
The Auditor Operating Instructions define the full reconciliation check suite, the report format, loading protocol, and boundary constraints. Load them into every Auditor conversation. The check suite is precise enough that any AI session loaded with it will run the checks identically.
When to Run an Auditor Cycle
  • At every phase gate transition (Phase 1 → 2, Phase 2 → 3, Phase 3 → 4)
  • After any spec change that affects multiple modules
  • After any period where multiple simultaneous module sessions were active
  • Any time the Coordinator's counts seem inconsistent with your own observation
  • On demand, when something feels off
What the Auditor Does Not Do

After the Auditor delivers its report, do not ask it what should be done about failures. Its job ends at "here are the mismatches." Take failures to the appropriate conversation — document inconsistencies go to the Coordinator, code mismatches go to the Integrator or the relevant module conversation. The Auditor's structural isolation from the other roles is what makes its checks reliable.

Section 13
The Operator's Routing Role

You are the only communication path between all conversations. Information does not flow directly from the Coordinator to a module conversation, or from the Integrator to the Coordinator. Every document, every finding, every routing manifest travels through you.

Routing Rules

Information flows through documents, not descriptions. If something needs to move from one conversation to another, it moves as a document — a file, a spec update, a routing manifest. Not as a verbal summary you paraphrase from memory. Verbal routing degrades over multiple hops. Document routing does not.

The Coordinator prepares; you deliver. The Coordinator produces routing manifests — packages telling you exactly what to deliver, to which conversation, with what context. Review the manifest. Deliver the package. Report back to the Coordinator that delivery is complete. If you modify a package before delivery, tell the Coordinator so the ledger stays accurate.

If you route outside the manifest, report it. Sometimes you'll carry a finding verbally for a simple fix. That's acceptable for low-stakes mechanical issues. For anything structural, use the manifest. And always tell the Coordinator what you did outside the manifest — it cannot log what it does not know about.

The Constellation Spec is the single source of truth. If a decision changes the spec, the Coordinator drafts the change and you approve it before it is applied. The Coordinator then rebuilds all affected loading packs. No module conversation should ever work against a stale spec — if one does, every deliverable it produces is built on shifted ground.

A Typical Routing Day in Phase 2
  1. Open the Coordinator. Review the decision queue and routing manifests prepared since your last session.
  2. Work P1 decisions first — anything blocking other work.
  3. Deliver routing manifests to the relevant module conversations.
  4. Collect module deliverables and bring them to the Integrator.
  5. Bring Integrator reports back to the Coordinator for triage and ledger logging.
  6. The Coordinator will have prepared new manifests based on the Integrator reports. Deliver those.
  7. Close any sessions that have completed their work and write their handoffs.
  8. Report back to the Coordinator on all deliveries made today.
Section 14
Document Hygiene

With the Coordinator active, document hygiene is largely automatic — every document change is a ledger event, dependency propagation is structural, and the Document Registry updates itself. Your role is oversight: spot-checking that the Coordinator's picture matches reality.

Document Status Values
StatusMeaning
CurrentApproved, consistent with all dependencies. Safe to load.
Review RequiredA dependency has changed since this document was last reviewed. Must be reviewed or explicitly acknowledged before loading into a conversation.
DraftProduced by the AI but not yet reviewed and approved by you. A version increment = approval.
ArchivedRetired. Superseded by a newer version or no longer needed.

A Review Required document must be updated (returns to Current) or explicitly acknowledged as a known risk before loading into a session. There is no silent loading of stale documents — an AI operating on stale information cannot detect that it is stale.

Version Pinning

Every loading pack declares the exact version of every document it includes. "Load the Glossary" is not sufficient — the pack must specify which version. When a pinned document advances, the Coordinator flags the pack as stale. Open the Coordinator and request a loading pack refresh before opening any session that would use a stale pack.

Document Authorship Rule

The AI produces content. The Operator approves it. A version increment equals Operator approval. The AI does not unilaterally advance version numbers. A document the Operator has not reviewed is a Draft. This is not a formality — it is the mechanism that keeps you in the decision loop on every document that enters the project.

Spec Changes Mid-Project

Spec changes are unavoidable. Business requirements evolve. Design decisions get reversed when implementation reveals consequences. The methodology handles spec changes — it does not prevent them. The protocol:

  1. Bring the proposed change to the Coordinator.
  2. The Coordinator identifies all downstream documents affected (loading packs, module status, dependency graph).
  3. You approve the change.
  4. The Coordinator applies the change as a ledger event and flags all downstream documents as Review Required.
  5. Update or rebuild affected loading packs before any dependent module conversation proceeds.
  6. Route a notification to all module conversations currently building against the changed sections.

A spec change applied without going through this protocol means module conversations will produce deliverables against the old spec. The Integrator will catch it — but as a failed verification, not as a clean correction. Don't make the Integrator find spec drift; run the protocol and prevent it.

Section 15
Failure Modes and Prevention

For the common failure modes that affect any AI-assisted session — Pattern Blindness, Scope Creep, Assumption Carrying, Context Starvation, and Context Bloat — see Interaction Guide §9. Those are session-level failures that every conversation type can experience.

This section covers three additional failure modes specific to multi-conversation coordination. These emerge only when running parallel conversations with a Coordinator, Integrator, and multiple module workers — which is why they live here rather than in the IG.

Coordinator Drift

What it is: The Coordinator's internal state — module status index, ledger continuity, dependency graph — drifts from the actual project state as the session grows long.

How it manifests: The Coordinator reports module status that doesn't match the deliverables you've seen. Routes a finding to a module that was already resolved. Produces a summary that misses a recent decision.

Prevention: The Coordinator runs self-checks after every event. The Auditor's reconciliation cycle catches cross-document drift. Produce a State Export before approaching context limits — don't let the Coordinator run until it degrades. Spot-check the Coordinator's module counts against your own understanding periodically.

Infinite Phase 0

What it is: Phase 0 continues indefinitely because every question the spec answers opens two more questions. The spec never reaches "precise enough to build against."

How it manifests: Weeks of spec work, no code written, the spec document keeps getting revised. The AI is asking good questions that you can't answer yet.

Prevention: When the same category of question recurs without resolution, name it as an open decision and defer it. Phase 0 produces a buildable spec — not a perfect one. Decisions that don't affect the architecture can wait. A spec that is 90% settled and has three clearly-named open decisions is ready to build against. A spec that is "almost done" with no named open decisions is not.

Cross-Conversation Contamination

What it is: Information from one conversation bleeds into another through the Operator's description rather than through documents.

How it manifests: The Operator tells a module conversation "the Integrator said to change the field name to X" — without the actual Integrator report. The module conversation makes the change based on the Operator's paraphrase. The paraphrase was slightly wrong. The resulting code doesn't match the contract.

Prevention: Information travels as documents. The Integrator report goes to the module conversation intact, not as a verbal description. The Coordinator's routing manifest goes to the module conversation as a document. When in doubt, paste the source material rather than summarizing it.

Section 16
Methodology Scaling

The full methodology — Coordinator, Integrator, Auditor, multiple parallel module conversations, loading packs, event ledger — is designed for projects large enough to need it. Not every project is that large. Running the full process on a small project costs more in coordination overhead than it saves in prevented errors. Skipping roles that turn out to be necessary costs more in rework than in prevented overhead. This section provides guidance on when to scale down.

Pre-Build Estimates
These thresholds are the Operator's best current judgment. No production project has been completed with the methodology yet. The Butcher Constellation build will produce real data on where the coordination overhead becomes net-positive. Expect these numbers to be revised.
Scaling by Project Size
Project SizeLoopsRecommended Configuration
Small 1–4 loops No Coordinator. No Auditor. No parallel module conversations. Run a single conversation per phase. Handoff documents serve as the state record. The Operator tracks status manually. The Integrator is still recommended if any two loops exchange packets — cross-module verification catches mismatches that in-conversation testing cannot.
Medium 5–10 loops Coordinator recommended. Run it lightweight — the Operator may handle some routing manually and use the Coordinator primarily for ledger maintenance and status tracking. Auditor cycles at phase gates only (not mid-phase). Two to three parallel module conversations in Phase 2. The Integrator is required.
Large 11+ loops Full methodology. Coordinator mandatory — the Operator cannot track status manually at this scale without losing information. Auditor cycles at phase gates and mid-Phase 2 at a cadence matching the project's velocity. Multiple parallel module conversations with scoped loading packs. The Integrator is required and will be busy.
Role-by-Role Optionality

The Coordinator is optional below ~5 loops. Without it, the Operator maintains the event ledger manually — which in practice means handoff documents serve as the ledger and the Operator holds the status picture in their head. This works when the picture is small enough to hold. It stops working when you cannot remember which module is waiting on which decision without checking a document.

The Integrator is required any time two or more module conversations run concurrently, regardless of project size. If any two conversations emit or subscribe to packet types that the other conversation also touches, cross-module verification is necessary. A project with a single conversation that builds all loops does not need an Integrator — the conversation can see its own contracts. The moment work splits into parallel conversations, the Integrator becomes necessary.

The Auditor is optional for projects under ~8 loops if the Operator is disciplined about manual document checks at phase gates. For larger projects, the accumulated document drift between phase gates becomes difficult to catch manually, and the Auditor's mechanical check suite earns its overhead. The Auditor is always recommended at Phase 4 (final verification) regardless of project size.

Loading Packs are always recommended, even for small projects. The cost of assembling a loading pack is low. The cost of a conversation operating on incomplete or wrong context is high. Even a two-loop project benefits from a clear declaration of what each conversation needs to see.

The Butcher Constellation Decision

The Butcher Constellation is estimated at 8–12 loops. This places it in the medium range. The current plan is to run the full methodology — Coordinator, Integrator, Auditor at phase gates — with the explicit goal of measuring coordination overhead against project throughput. This is a deliberate choice to stress-test the full system on the first real project rather than to optimize for speed. The data from this run will calibrate the thresholds above.

Section 17
Context Budget

Every conversation has a finite context window. Loading documents consumes part of it. The code produced during the session consumes the rest. If the loading pack is too large, there is not enough room for the work. If the loading pack is too small, the AI fills gaps with assumptions. This section provides rough estimates for budgeting context across conversation types.

When Context Gets Tight

If a module conversation is running low on context — the AI's responses are getting shorter, it's losing track of earlier decisions, or you're approaching the window limit — take these steps in order:

  1. Write a handoff and restart. This is the primary mechanism. The handoff captures the current state. A fresh conversation with the loading pack plus the handoff picks up without loss. Most context pressure is resolved by restarting.
  2. Trim the loading pack. If the loading pack itself is consuming too much context, trim in this order: drop the Patterns Reference (consult it in a separate conversation if needed), drop the Glossary (the AI can ask about terms it's unsure of), trim the routing table to only the entries this module directly emits or consumes. Never drop the Coding Standards or the module's own contracts.
  3. Split the conversation. If a single module is too large to build in one conversation even after trimming, split it into sub-module conversations — each building a subset of the loops. This requires additional loading pack preparation and Integrator verification passes, but it keeps each conversation's context manageable.
The Coordinator Context Problem

The Coordinator accumulates state throughout the project. Unlike module conversations, which produce a deliverable and end, the Coordinator runs continuously. Its context fills with the growing ledger, status updates, and routing history. The State Export mechanism (see §10) exists specifically for this: serialize the Coordinator's state, open a fresh conversation, load the State Export, and continue. Plan for Coordinator restarts — they are not failures, they are expected maintenance. A rough rule of thumb: produce a State Export every 30–40 events or when the Coordinator's responses start losing precision.

Questions to Ask at Every Session Start
  • "What is the objective for this session?" — if not already clear
