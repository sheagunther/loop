Abstract

Loop MMT™ (Multi-Module Theory) is a software architecture and development methodology for building production applications as a solo developer leading AI coding assistants. It is derived from Loop 2.1™, a browser-based manual flow computer refined across 329 builds, and carries forward that system's core discipline: every data path is explicit, every transfer is observable, every failure is contained, and the operator is always in control.

The architecture treats software as a constellation of independent processing loops connected by multiple typed buses. Seven loop types divide all responsibilities; boundaries between them are enforced structurally through JavaScript closures, not by convention. Multi-step operations are orchestrated by Workflow Loops using declared pipeline specs with a five-word failure vocabulary and idempotency guarantees. Data Gates govern lifecycle. Dependency Gates isolate external services with graceful degradation. The system is designed for longevity: self-verifying constellations, versioned contracts, automatic schema migrations, and dependency sunset protocols.

The development methodology coordinates parallel AI conversations through six conversation types — Specification, Module, Integrator, Coordinator, Auditor, and Bug Fix — unified by an append-only event ledger. Document governance rides on the ledger: every document change is an event, every dependency is tracked, and staleness propagation is automatic. A five-layer safety architecture with a Divergence Circuit Breaker provides defense-in-depth for multi-role AI coordination.

Section 1
Origin and Principles

Loop MMT is derived from Loop 2.1 — a browser-based manual flow computer where the human operator replaces the stored program — and from the governance framework developed across 329 builds of that system. The architecture did not begin as theory. It was extracted from a working machine, tested by use, and formalized into nine principles:

1. The Operator is always in control. The human developer is the architectural authority. The AI writes code; the human directs. When the system cannot decide, it escalates — it does not guess.

2. All state is explicit. Data exists in exactly two places: on a bus or in a Vault. No ghost state.

3. Every data path is declared. The routing table is a complete map of every legal data movement. No implicit routing.

4. Boundaries are structural, not conventional. The closure wall enforces loop type constraints. The test suite verifies them.

5. Failures are contained. Each loop and each bus is its own failure domain. The system degrades gracefully.

6. The system is observable. Every bus transfer is traceable. Every workflow step is logged. Every error is an event on the bus.

7. Dependencies are adopted deliberately. Build what you can. Adopt what you must. Isolate what you adopt. Wrap it in a gate.

8. The source of truth survives the session. Architectural decisions are expressed in code, tests, and specifications — not in chat history.

9. The system outlives its builders. A constellation carries its own blueprint, can verify itself against it, and can evolve without breaking. Software rot is a structural problem with a structural solution.

The Core Insight
Build the constraints into the system. Let the system enforce them. An architectural decision baked into structure is a decision that will be respected by every future session, every future developer, and every future AI — without anyone having to remember it.
Section 2
Loop Types

Every module in a Loop MMT constellation is one of seven types. The type is not a label — it determines what the module can access. A Compute Loop receives the bus and nothing else. A Vault Loop receives the bus and the database client. The closure wall (§5) enforces these boundaries at the language level: capabilities not passed to the factory function do not exist in the loop's scope. The type system is the mechanism by which the architecture prevents responsibility creep — a Compute Loop that gradually acquires database access, a Filter Loop that starts sending notifications. Each type has exactly the capabilities it needs and structurally cannot acquire more.

Section 3
The Bus System

Multiple named buses, each independent with its own priority queue and contract validation. Push (event) and Pull (request-by-type, decoupled from provider). Packets are structured objects with type, bus, source, timestamp, payload, and hash. Contracts enforced at outbound and inbound gates. Three priority levels: CRITICAL, STANDARD, DEFERRED. The routing table declares every legal path. Buses support passive observers for the Observatory.

Query Parameters in Pull Requests

Pull requests can carry query parameters in their payload — filter conditions, sort orders, pagination, date ranges. The Vault Loop translates these into database queries. The packet contract for each FETCH_* type declares what filter parameters are supported. This is not a search engine — it is a structured query interface where the valid parameters are part of the contract.

Scoped Subscription

A standard subscription receives all packets of a type. A Scoped Subscription adds a filter predicate evaluated at delivery time. The bus evaluates the predicate against each packet before delivering — packets that don't match are never delivered to that subscriber.

// Standard — receives ALL MESSAGE_POSTED packets
bus.data.on('MESSAGE_POSTED', handleMessage);

// Scoped — receives only messages for channels I'm in
bus.data.on('MESSAGE_POSTED', handleMessage, {
  scope: { field: 'payload.channelId', in: myActiveChannelIds }
});

// Update scope at runtime — user joined a new channel
bus.data.updateScope('MESSAGE_POSTED', handleMessage, {
  field: 'payload.channelId',
  in: [...myActiveChannelIds, newChannelId]
});

Scope operators: equals (exact match), in (match against a set), not (inverse), exists (field present), predicate (arbitrary function — use sparingly, opaque to Observatory). Scopes are dynamic — updatable at runtime as user context changes (join channel, switch tenant). For multi-tenant systems, scoped subscriptions are a security mechanism: the Auth Gate stamps packets with a tenantId, and the Presentation Loop's subscription is scoped to its own tenant. Defense in depth — routing table is the first barrier, scope is the second. Passive observers are not affected by scopes — they see all traffic. L21 analog: the Pattern Matcher mask that selects which bits to inspect.

Authoritative Loop

The Constellation Spec can designate a loop as authoritative for a data domain. All mutations to that domain go through that loop in strict arrival order. No concurrent processing. The bus guarantees sequential delivery: the next packet is not delivered until the authoritative loop has emitted its response to the current one.

const LOOPS = {
  'compute:gameState': {
    type: 'compute',
    authoritative: true,
    domain: 'game_state',
    sequencing: 'strict_fifo',
  }
};

Three enforced properties: Sequential delivery — packets delivered one at a time, using the hold Transit Gate policy automatically. Global ordering — in multi-device constellations, mutations route through the server which assigns sequence numbers. Adds latency but guarantees consistency. Read replicas — other loops read via the Projection pattern but cannot write to the domain directly.

Authoritative designation overrides the concurrency strategy for that domain — there are no conflicts because mutations are sequential. Appropriate for game state, financial ledgers, and any system where mutation order is critical. Not needed for most CRUD applications where eventual consistency is acceptable.

Pull requests support an extensible vocabulary of query types beyond simple filter/sort — including range queries, geospatial lookups, full-text search, graph traversal, and aggregation. Each FETCH_* contract declares which query types its Vault Loop supports. Unsupported types return QUERY_TYPE_UNSUPPORTED. The specific query types available are declared per project in the Constellation Spec, not prescribed by the standard. The Vault Loop translates query parameters to appropriate database operations.

Section 4
Data Gates

Data Gates are lifecycle policies that govern when data is consumed, archived, or destroyed. Every piece of data in a constellation — bus packets, Vault records, workflow transaction logs — has a declared retention policy. Data does not accumulate unbounded. The L21 origin is the Gate on each loop: a selective destruction point where the operator deliberately retires consumed data by closing the gate. In Loop MMT, the destruction is automatic, policy-driven, and auditable.

Transit Gate (Buses): Three policies per packet type. consume — destroy after delivery (default). archive — retain in a circular buffer for diagnostics. hold — stay on bus until receiver acknowledges (for guaranteed delivery across crashes).

Retention Gate (Vaults): Three stages per data type. Active (working set, default queries), Archived (soft-removed, available on explicit request), Purged (permanently deleted). Transitions are declared conditions in the Constellation Spec. A background sweep runs periodically, is idempotent, and logs every transition to the audit trail.

Vault Loops can declare sub-patterns for specialized workloads (e.g., append-only time-series data with automatic downsampling). These are not new loop types — they are recognized configurations of the Vault Loop with declared behaviors. Sub-patterns are defined per project in the Constellation Spec.

Workflow Gate: Transaction log retention by outcome. Completed: 30 days active, 1 year archived. Failed/escalated: 90 days active, 2 years archived. In-progress: never gated.

Section 5
Boundary Enforcement

Every loop is constructed by a factory function that receives exactly its permitted capabilities. A Compute Loop's factory receives { bus }. A Vault Loop's factory receives { bus, db }. The closure scope is the wall. If the database client was not passed in, it does not exist — not as a restricted variable, not as a null reference. It is absent from the scope chain entirely. A Compute Loop that attempts to call the database will produce a ReferenceError, not a policy violation. The boundary is enforced by the JavaScript runtime, not by the team's discipline.

This is not access control. Access control means the variable exists and the system checks whether you are allowed to use it. The closure wall means the variable does not exist. There is nothing to check, nothing to bypass, no flag to toggle. The only way to violate the boundary is active sabotage — smuggling a reference through the bus payload, assigning to the global object — and any such violation is visible, deliberate, and obvious in code review. The architecture converts boundary violations from passive drift (the easy shortcut that accumulates silently) into active sabotage (the deliberate wrong that requires effort and leaves evidence).

The bus object itself is a filtered view. Each factory receives a bus interface that exposes only the channels that loop type is authorized to use. The test suite verifies every factory's parameter list against the Capability Injection Table on every build — if a factory signature changes to accept a capability its type does not permit, the Gantry catches it before the code integrates.

The technique has a lineage in capability-based security (Dennis and Van Horn, 1966) and the object-capability model. Loop MMT's application of it is specific to the constraint of AI-assisted development: architectural boundaries must survive complete memory loss between sessions. The AI does not take shortcuts out of laziness or deadline pressure. It takes shortcuts because it does not know the boundary exists unless the scope chain enforces it. The closure wall solves a different problem than process isolation in microservices — process isolation provides crash containment, memory isolation, and deployment independence. The closure wall prevents the gradual accumulation of unauthorized dependencies through convenience, which is the erosion pattern that kills most long-lived single-process applications. These are complementary protections, not competing ones.

The Derivation
In L21, the Gate is a physical point on each loop where data is destroyed when the gate is closed. The operator cannot route data past a closed gate — the hardware enforces the boundary. The closure wall is the software equivalent: a boundary enforced by the runtime, not by the operator's memory or the team's conventions. The principle is the same in both cases — build the constraint into the structure so that violating it is not a matter of discipline but of possibility.
Section 6
Workflows

The Workflow Loop drives multi-step operations. Pipeline specs are data structures (not procedural code) declaring steps, dependencies, timeouts, and failure strategies. The engine is generic — adding a workflow means adding a spec. Five-word failure vocabulary: Abort (stop), Retry (with backoff), Skip (proceed without this step's output), Compensate (undo earlier steps), Escalate (hand to human). Transaction logs persist after every step. State Reincarnation resumes workflows after crashes.

Idempotency

Each step in a pipeline spec can declare an idempotency key derivation — a function that produces a unique key from the step's inputs. If the step is retried (including after reincarnation), the same key is generated and sent with the request. External services that support idempotency keys (payment processors, most REST APIs) correctly deduplicate.

{
  name:           'charge_payment',
  emit:           { type: 'PROCESS_PAYMENT', bus: 'PAYMENT' },
  expect:         'PAYMENT_CONFIRMED',
  timeout:        30000,
  onFail:         'abort',
  idempotencyKey: (inputs) =>
    `pay_${inputs.orderId}_${inputs.totalPrice}_${inputs.workflowId}`,
}

The key is derived deterministically from inputs — the same order, price, and workflow always produce the same key. Included in the emitted packet as a standard field. The receiving loop passes it through to the external service. If the service doesn't support idempotency keys, the field is ignored.

Internal idempotency is implicit for all workflow steps. Before dispatching any step, the Workflow Engine checks the transaction log: has this step already completed for this workflow? If yes, skip it and use the recorded result. External idempotency (via declared keys) protects against duplicate calls to outside services. Internal idempotency (via the transaction log) protects against duplicate processing within the constellation. Together, they make any step safe to retry under any failure condition — including the worst case: crash between external API success and transaction log persistence.

Cascade Update

A workflow pattern for hierarchical state propagation. When a child record changes state, the pipeline evaluates whether the parent should also change. Two new pipeline spec features enable this:

Conditional steps. A condition field allows a step to execute only if a predicate on prior results is true. "Only transition parent if all siblings are done." If the condition is false, the step is skipped (not failed — skipped) and the workflow completes normally.

Recursive pipelines. A recursion block declares that a pipeline can trigger itself. When the parent's status changes, the same pipeline fires for the grandparent. A maxDepth guard prevents infinite recursion. If the hierarchy exceeds the depth limit, the cascade stops and emits CASCADE_DEPTH_EXCEEDED.

const PIPELINE_CASCADE = {
  name:    'cascade_task_completion',
  trigger: { type: 'STATUS_CHANGE',
             match: { newStatus: 'done', recordType: 'task' } },
  steps: [
    { name: 'fetch_siblings',
      emit: { type: 'FETCH_TASKS', bus: 'VAULT' },
      queryParams: (trigger) => ({ parentId: trigger.payload.parentId }),
      expect: 'TASKS_FETCHED', onFail: 'abort' },
    { name: 'evaluate_parent',
      emit: { type: 'EVALUATE_COMPLETION', bus: 'DATA' },
      needs: ['fetch_siblings.payload'],
      expect: 'EVALUATION_RESULT', onFail: 'abort' },
    { name: 'transition_parent',
      emit: { type: 'STATUS_CHANGE', bus: 'DATA' },
      needs: ['evaluate_parent.payload'],
      condition: (results) => results.evaluate_parent.payload.allComplete,
      expect: 'STATUS_VALIDATED', onFail: 'abort' }
  ],
  recursion: { enabled: true, maxDepth: 10,
               trigger: 'transition_parent.success' }
};

Works both upward (child completes → parent evaluates) and downward (parent archived → children archived). Downward cascades combine naturally with the Batch pattern — "archive all children of this parent" is a batch status update.

Failure Strategy Assignment

Every step in a pipeline spec declares its failure strategy using one of the five words. The mechanism is simple. The decision of which word to assign is not — especially for steps that could plausibly use more than one strategy. This guidance provides a decision framework.

Escalate when the cost of a wrong automatic decision exceeds the cost of delay. If the system choosing incorrectly would create a problem that requires human effort to undo — a double charge, a lost order, an irreversible state transition — escalate. The Operator sees the paused workflow, evaluates the situation with context the system doesn't have, and decides. Use for: payment failures above a threshold, cancellations with partial completion, any step where the compensation action is expensive or unreliable.

Retry when the failure is likely transient and the operation is safe to repeat. Network timeouts, temporary service unavailability, rate limit hits. Always declare a retry limit and backoff strategy — unbounded retries are an infinite loop with extra steps. Combine with idempotency keys for external calls. Use for: database writes during brief connectivity loss, external API calls that return 503, webhook deliveries.

Abort when continuing would produce corrupt or inconsistent state. If the prerequisite for subsequent steps has failed in a way that makes those steps meaningless — the record to update doesn't exist, the validation returned hard errors, the price calculation produced an impossible result — abort. Already-completed steps are not automatically rolled back unless they declared compensation actions. Use for: validation failures, missing prerequisite data, business rule violations.

Skip when the failed step is supplementary and the workflow's primary purpose can still be achieved without it. Sending a confirmation SMS fails but the order was successfully created — the order matters more than the notification. The failure is logged, the Operator can investigate, but the workflow proceeds. Use for: notification delivery, analytics events, non-critical enrichment steps.

Compensate when earlier completed steps must be explicitly undone because the workflow can no longer reach a valid end state. A payment was charged but inventory allocation failed — the payment must be refunded. Compensation actions are declared per step and execute in reverse order. Use sparingly: compensation is complex, can itself fail, and adds a second failure path that must be tested. Prefer designing workflows so that irreversible steps come last, after all fallible steps have succeeded.

The Ordering Principle
Where possible, order pipeline steps so that reversible operations come before irreversible ones. Validate before charging. Reserve before confirming. Check inventory before processing payment. This reduces the need for compensation — if the workflow aborts early, there is nothing to undo.

Batch operations and partial success. When a batch step processes fifty records and three fail, the step itself is treated as successful — it processed all fifty. The result packet contains both the completed records and the failed records with their error details. A subsequent conditional step handles the failures: routing them into a retry pipeline, a dead letter queue, or an escalation, depending on the failure type. The five-word vocabulary applies to the handling of each failed record in the secondary pipeline, not to the batch step that surfaced them. See also the Batch pattern in §7.

Section 7
Patterns

Patterns are named, reusable solutions to problems that arise when building with the loop + bus + closure wall architecture. This section defines the architectural patterns — patterns that affect how you design a constellation. A companion document (the Patterns Reference) catalogs additional implementation patterns — useful techniques for working within the architecture that do not constrain its design.

Architectural Patterns

These patterns define structural relationships in the architecture. If you don't know them, you will build the wrong thing.

The Echo. L21: The operator observes the ALU result before routing it further — the computation is confirmed before it is used. A loop never trusts that an operation succeeded until the responsible loop confirms it by emitting a completion packet. The UI does not show "Order Confirmed" when the user taps Submit — it shows it when the Vault emits ORDER_CREATED. The difference is the difference between hope and proof. The echo packet can carry metadata about the confirmation — persisted: true vs. persisted: buffered — so downstream loops and the UI know the real status. The Workflow Loop is built entirely on Echoes: emit a step request, wait for the echo, then proceed.

The Marker. L21: The marker bit (bit 16) distinguishes a live word from empty space. A word with marker 0 is vacant; marker 1 is live data. Every packet carries a liveness indicator. marker: 'live' is real data. marker: 'empty' is a deliberate null signal — "I looked and there is nothing here." marker: 'tombstone' is a deletion notice — "this record existed and was deliberately removed." The distinction matters because "no data returned" is ambiguous (query failed? genuinely empty?), while a marked empty response is unambiguous. Pull requests must always return a marked packet, never silence.

The Hydration. L21: The operator loads a value from memory into the ALU loop, enriching the workspace before computation. A Filter Loop that receives a sparse packet (just an ID), pulls full data from a Vault, and attaches it to the packet before passing it along. The Customer Lookup does this: receives a customer ID, fetches the full customer record, attaches it. The enriched packet now carries everything downstream loops need. Hydration is always done by a Filter Loop (it enriches, it does not transform) and the pull goes through the Vault Bus (it is a data retrieval, not a business computation).

The Cascade. L21: PM1 → PM2 → TG1 → TG2. A four-stage conditional pipeline on the Big Loop where each stage filters the output of the previous one, all in a single rotation. Chaining Filter Loops on the bus. Packet enters Filter A, passes or rejects. Passes go to Filter B. Passes go to Filter C. Each filter is independent, has its own contract, and can be added or removed without changing the others. The sequence is defined in the routing table, not in the filters. Adding a new filter to the cascade means updating the routing table — no existing code changes. The most powerful composition mechanism in the architecture.

The Projection. L21: The operator reads from multiple sources (ALU result, comparator flags, counter value) to form a complete picture before making a routing decision. A Compute Loop subscribes to change events from multiple Vaults and maintains a derived, denormalized view. When the Order Vault emits ORDER_UPDATED, the Projection Loop pulls related customer, status, and media data, assembles the complete view, and emits PROJECTION_UPDATED. The Presentation Loop subscribes to the projection, not to the individual Vaults — one subscription, one packet, all the data needed to render. The projection is explicitly a derived cache, not a source of truth. It can be rebuilt from the Vaults at any time.

The Bridge. L21: When a Pattern Matcher ejects a word, it goes to a one-word bridge buffer before being available to a bus. The bridge holds one word; a second match overwrites if the bridge hasn't drained. A buffering mechanism between a loop's output and the bus. The loop produces output faster than the bus can carry it. The bridge smooths the flow. Three overflow strategies, declared in the loop's spec: overwrite (latest wins, like the L21 PM bridge), queue (FIFO buffer, like the Dependency Gate), backpressure (the loop pauses until the bridge drains). Bridge depth and overflow strategy are declared, not hardcoded.

The Translator. When a loop adopts an external dependency that speaks a fundamentally different data model — a payment processor's webhook format, a legacy SOAP API, an OIDC provider's token structure — the loop performs two distinct responsibilities: data translation (converting the external format into clean internal packets) and resilience management (the Dependency Gate's live/buffered/draining lifecycle). These are different concerns and should be recognized as such, even though they live in the same loop. The Translator is the data-scrubbing half. It receives raw external data, validates it against the external provider's schema, maps it to the internal packet contract, and emits clean packets that are indistinguishable from any other packet on the bus. The translation logic is isolated inside the loop's closure — no other loop knows or cares that the data originated from a foreign system. The Dependency Gate handles what happens when the external system is unreachable. The Translator handles what happens when it responds. The Constellation Spec should declare both: "this loop adopts Stripe, uses a Dependency Gate for resilience, and uses the Translator pattern to convert Stripe webhooks into PAYMENT_CONFIRMED packets." This makes the data-scrubbing responsibility a design-time decision, visible in the spec, rather than an implementation detail discovered during coding.

The Batch. L21: Memory batch write sends all 16 slots sequentially. Destructive read clears after reading. Operating on multiple records atomically. A Workflow pipeline where the first step is a bulk pull request and subsequent steps operate on the entire result set. Vault Loops support batch insert, batch update, and batch archive as declared capabilities. The read-and-consume variant (fetch records and mark them as in-progress atomically) is how retry queues work — the Signal Loop fetches pending notifications and claims them in one operation, so no other instance re-sends them.

This is not forced mapping — it is evidence that the hardware model captures fundamental computational patterns that recur at every level of abstraction. The machine's design, because it is minimal and observable, exposes patterns that conventional software buries under layers of abstraction. Loop MMT names them and makes them explicit.
Section 8
Fault Tolerance

Fault tolerance in Loop MMT operates at three independent levels. Each level catches a different category of failure. Together, they provide defense-in-depth — no single failure propagates across the system unchecked.

Bus level. Contract gates validate every packet at outbound and inbound. A malformed packet — wrong type, missing field, shape mismatch — is rejected before it reaches the bus. Bad data does not travel. This is the first line of defense and it operates on every transfer, every time, with no exceptions. The contract gate does not log a warning and let the packet through. It rejects. The emitting loop receives an error. The bus stays clean.

Loop level. Each loop is its own failure domain. The closure wall prevents a failure in one loop from accessing another loop's state. The bus contracts prevent a failure from propagating bad data to downstream consumers. If a loop encounters an unrecoverable error, it follows a four-step recovery protocol: do not crash the process, emit an error event to the bus (so the Observatory and Operator Alerts can respond), preserve whatever state can be preserved, and continue operating in a degraded mode if possible. A Compute Loop that fails on one packet should not fail on the next — each packet is independent. The goal is containment: the failure stays in the loop that experienced it.

Workflow level. Multi-step operations fail in more complex ways than single-loop operations — a step can succeed and then a subsequent step can fail, leaving the workflow in a partially-completed state. The five-word failure vocabulary (§6) addresses this: each step declares its failure strategy before code is written. The Workflow Engine executes the declared strategy automatically. The transaction log records every step's outcome, enabling State Reincarnation after crashes. Workflow-level fault tolerance is not about preventing failures — it is about ensuring that every failure has a pre-declared response.

Bus isolation. Each bus is an independent failure domain with its own priority queue and its own contract registry. A malformed packet on the Data Bus does not affect the Vault Bus or the Signal Bus. A backpressure condition on one bus does not stall another. This isolation means that a single failure — even a sustained one — cannot take down the entire communication layer. The buses degrade independently.

Section 9
State and Persistence

The Two-Place Rule: Data exists on a bus or in a Vault. No third place. The Presentation Loop holds a projection, not the source of truth.

Sync Model: Eventually consistent via real-time subscriptions on the Sync Bus. Offline devices queue writes locally and drain on reconnection.

Concurrency Strategy

When two operators modify the same record on different devices simultaneously, the default resolution is last-write-wins — the second write silently overwrites the first. Both writes are captured in the audit trail, so the overwritten change is never lost. This is appropriate for most projects.

For data types where silent overwrites are unacceptable, the Constellation Spec can declare alternative strategies per record type: field-level merge (non-conflicting field changes both apply) or optimistic locking (version-checked writes that fail on conflict). Data domains governed by an Authoritative Loop (§3) override the concurrency strategy entirely — there are no conflicts because mutations are sequential.

Section 10
The Dependency Principle

Dependency-averse, not zero-dependency. Built: custom infrastructure owned entirely. Adopted: external services, justified, isolated behind one loop type, wrapped in a Dependency Gate. Forbidden: dependencies that compromise architectural guarantees (UI frameworks, state management, build tools).

Dependency Gates: Three modes. Live (pass-through with latency monitoring). Buffered (unreachable — writes buffer in IndexedDB, reads from cache). Draining (recovered — buffer drains at controlled rate). The gate is generic infrastructure. The loop's bus interface does not change across modes.

Section 11
Security and Identity

Security in Loop MMT is not a feature bolted on after the architecture is built. It is a structural property of the architecture, enforced at the same level as loop boundaries and bus contracts.

The Auth Gate

The Auth Gate is a specialized Filter Loop positioned between the Presentation Loop and the Working Loop. Every operator action passes through the Auth Gate before entering the system. The Auth Gate performs three functions:

1. Identity verification. The Auth Gate validates the session token (via the auth provider — Supabase Auth, for example) and attaches the verified operator identity to the packet. The identity is not self-reported by the UI — it is cryptographically verified by the auth provider and stamped by the Auth Gate. Every packet that enters the Working Loop carries a verified operatorId and role.

2. Role-based access control. The Auth Gate checks whether the operator's role is authorized to perform the requested action. The authorization rules are declared in the Constellation Spec as a role-permission matrix:

const ROLES = {
  owner:    { can: ['*'] },                       // all actions
  operator: { can: ['intake', 'status_change',
                     'lookup', 'reconcile_daily'] },
  viewer:   { can: ['lookup', 'reconcile_daily'] }  // read-only
};

const ACTIONS_REQUIRING = {
  cancel_order:      ['owner'],
  issue_refund:      ['owner'],
  modify_price_table:['owner'],
  reconcile_season:  ['owner'],
  intake:            ['owner', 'operator'],
  status_change:     ['owner', 'operator'],
};

An operator who does not have the required role for an action gets a AUTH_REJECTED packet back through the Data Bus. The UI shows a clear "you don't have permission" message. The audit trail logs the attempt.

3. Session management. The Auth Gate tracks active sessions and handles token refresh, expiration, and forced logout (if the owner revokes an operator's access). Session state follows the Two-Place Rule — it is either on the bus (a session validation request in transit) or in the auth provider (persisted).

Structural Enforcement
The Auth Gate enforces permissions at the same architectural level as the closure wall enforces loop capabilities. A Compute Loop cannot access the database because it does not have the db client in scope. An operator cannot cancel an order because the Auth Gate strips that action before it reaches the Working Loop. Neither constraint depends on the UI hiding a button — both are enforced by infrastructure that sits below the UI layer.

The Auth Gate supports contextual access rules beyond simple role-action checks. The role-permission matrix can include predicate-based where clauses for record-level access control ("this doctor can only see patients assigned to them") and cannot lists for field-level access control ("nurses can see patient demographics and clinical data but not prescriptions"). Predicates are evaluated at two points: the Auth Gate checks them before actions enter the Working Loop, and the Vault Loop applies them as server-side query filters so unauthorized records are never returned. Field groups and access rules are declared in the Constellation Spec. The specific schema for contextual rules is defined per project.

Input Sanitization

Every string that enters the system through the Working Loop is sanitized before it travels further. This is a responsibility of the Validation Filter Loop — the same loop that validates schema compliance also strips or escapes potentially dangerous content (HTML entities, script injection patterns, SQL-relevant characters). Sanitization is not optional and not per-feature. It is a property of the validation layer that every packet passes through.

Rate Limiting

The Working Loop enforces a declared maximum ingest rate — packets per second per operator, and packets per second globally. If the rate is exceeded, the Working Loop rejects with a RATE_LIMITED error. This prevents a runaway UI bug, a malicious actor, or a misconfigured Clock from overwhelming the bus. The rate limits are declared in the Constellation Spec and are tunable without code changes.

PII Policy

The Constellation Spec declares which fields in which packet types contain personally identifiable information (PII) — customer names, phone numbers, email addresses. The policy governs: PII fields are encrypted at rest in the Vault (the persistence layer handles this). PII fields are never logged in full in the audit trail — they are hashed or truncated (last four digits of a phone number, for example). PII fields are excluded from health packets and Observatory telemetry. The PII declaration is part of the packet contract — it sits alongside the field type and validation rules.

Infrastructure Patterns
The Inlet

The Inlet is a recognized sub-pattern of the Working Loop for continuous external data streams — WebSocket, Server-Sent Events, MQTT, or polling intervals on REST endpoints. It is not a new loop type. It is a Working Loop with additional infrastructure for stream management. Receives bus only (same capabilities as any Working Loop).

What makes it distinct: Connection lifecycle — manages persistent connection establishment, authentication with the source, reconnection on failure (with exponential backoff), and graceful shutdown. Config is declared in the Constellation Spec. Back-pressure — if the stream is faster than the system can process, a declared strategy applies: buffer (hold in local queue up to max), sample (every Nth reading), latest (most recent value per key — useful for position updates), or drop (reject when full, with counter). Stream health — emits health packets: connection state, ingest rate, back-pressure state, drop count.

const INLET_GPS_FLEET = {
  name:       'inlet:gps',
  source:     { type: 'websocket', url: 'wss://fleet.example.com/positions' },
  auth:       { type: 'bearer', tokenFrom: 'config:fleet_api_key' },
  reconnect:  { backoff: 'exponential', maxDelay: 30000 },
  backpressure: { strategy: 'latest', keyField: 'payload.vehicleId' },
  injectAs:   { type: 'INTAKE_RAW', subtype: 'gps_position' },
  bus:        'DATA',
  rateLimit:  { maxPerSecond: 100 }
};

Inlet-injected packets pass through the Auth Gate with a system-level identity (operator: 'system:inlet:gps'), same as Clock packets. Once on the bus, they are indistinguishable from any other packet. The source is external. The processing is standard. L21 analog: External Bus E receiving data from outside the machine.

The Clock

The Clock is a scheduled event emitter that injects packets into the Working Loop on declared schedules. It is not a loop type — it is infrastructure, like the bus. It reads a schedule table from the Constellation Spec and emits INTAKE_RAW packets at the declared times. The Working Loop does not know or care whether a packet came from a user tapping a button or from the Clock — it processes both identically.

const SCHEDULES = [
  { name: 'daily_reconciliation',  cron: '0 23 * * *',
    emit: { subtype: 'reconcile', scope: 'daily' } },
  { name: 'retention_sweep',       cron: '0 3 * * *',
    emit: { subtype: 'retention_sweep' } },
  { name: 'pickup_reminder_check', cron: '0 10 * * *',
    emit: { subtype: 'pickup_reminder_check' } },
  { name: 'health_heartbeat',      interval: 30000,
    emit: { subtype: 'heartbeat_tick' } },
];

The Clock supports both cron-style schedules (for time-of-day events) and interval-based schedules (for periodic heartbeats). Clock-emitted packets pass through the Auth Gate with a system-level identity (operator: 'system:clock') so they are distinguishable from human-initiated actions in the audit trail.

Configuration

Application configuration lives in a Config Vault — a Vault Loop whose sole job is storing and serving configuration data. Two kinds:

Static config: Changes rarely. Price tables, cut options, processing fees, role definitions. Loaded at constellation startup. Loops access it via pull requests (FETCH_PRICE_TABLE, FETCH_ROLE_MATRIX).

Runtime config: Can change while the system is running. Feature flags, maintenance mode toggle, temporary surcharges. Stored in the Config Vault but pushed to subscribing loops when changed via a CONFIG_UPDATED event on the Data Bus. Loops that use runtime config subscribe to this event and refresh their local copy.

The Config Vault is the single answer to "where do I change X?" Every configurable value in the system lives here. The UI provides an admin interface for the owner role to modify configuration. Changes go through the standard pipeline: Working Loop → Auth Gate (owner only) → Validation → Config Vault → CONFIG_UPDATED → all subscribing loops.

Document Generation

Generating a formatted document (PDF receipt, printable carcass tag, season report) is a two-loop pattern. A Compute Loop takes business data and produces formatted content (HTML template populated with order data). A Signal Loop handles the delivery channel — print, download, email attachment. The Compute Loop produces the document; the Signal Loop delivers it. Neither crosses into the other's responsibility.

Operator Alerts

The Notification Signal Loop sends messages to customers. A separate Operator Alert Signal Loop sends messages to the system operator — the person responsible for keeping the system running. Different audience, different trigger conditions, different urgency. When a Dependency Gate transitions to buffered mode at 2am, the customer doesn't need to know. The operator does. The Operator Alert loop subscribes to DEPENDENCY_OFFLINE, WORKFLOW_ABORTED, LOOP_ERROR (repeated), and other critical system health events, and delivers alerts via the operator's preferred channel (SMS, email, push notification).

Section 13
Longevity and Evolution

Software rots. Dependencies get deprecated. APIs change. The developer who understood the system moves on. Business rules change but old logic stays. Data accumulates in outdated formats. Security vulnerabilities emerge in code nobody touches. The environment shifts — browsers drop features, devices change, services get acquired and shut down. The knowledge of why things are the way they are evaporates, until nobody can safely change anything because nobody knows what will break.

Loop MMT is designed to resist this. Not through perfect code — there is no such thing — but through structural properties that make decay detectable, evolution safe, and understanding recoverable. A constellation built to the standard should be as comprehensible to a new developer (human or AI) reading it five years from now as it was the day it shipped. That is the design goal.

Self-Verification

A Loop MMT constellation carries its own blueprint — the Constellation Spec. The spec declares every loop, every bus, every route, every contract, every pipeline. The key insight: the spec is not just documentation. It is executable. The bus reads the routing table from it. The workflow engine reads pipeline specs from it. The contract gates read packet shapes from it. The initializer reads the loop manifest and capability table from it.

This means the system can check itself against its own blueprint at any time. Self-verification runs at two levels:

Startup verification. When the constellation initializes, it verifies structural integrity before accepting any traffic: every declared loop is instantiated, every bus is active, every route has a live subscriber, every packet contract is loaded. If any check fails, the constellation does not start — it emits a diagnostic report listing exactly what is missing or misconfigured. The implementation is straightforward (iterate the manifest, check each entity exists) and belongs in the Coding Standards, not here. The principle is what matters: a partially-wired constellation never operates.

Runtime verification. While the constellation is running, periodic integrity checks verify that the system has not drifted from its spec. The Clock emits a SELF_VERIFY event on a declared schedule (hourly by default). The initializer re-runs the startup checks against the live system. If a loop has crashed and not recovered, if a bus subscription has silently dropped, if a contract definition has been corrupted in memory — the runtime verification catches it and emits a INTEGRITY_FAILURE event on the Data Bus. The Operator Alert loop picks it up.

The Principle
Most software systems can only be verified by external tests. A Loop MMT constellation can verify itself, because it carries its own specification and can compare its running state against it. The spec is the blueprint. The initializer is the inspector. The gap between "what the system should be" and "what the system is" is measurable at any moment.
Contract Versioning

Packet contracts evolve. v2 of the order system adds a rushProcessing field that v1 didn't have. If you update all producers and consumers simultaneously, there is no problem. In practice, rolling updates — especially across multiple devices that may be running different versions — mean that v1 and v2 packets coexist on the bus during the migration window.

Loop MMT handles this through versioned contracts:

Every packet carries a version field. The contract registry holds multiple versions of each packet type. The bus validates against the version declared in the packet. A v1 packet is validated against the v1 contract. A v2 packet is validated against the v2 contract. Both pass validation. Neither is rejected for being the "wrong" version.

Consuming loops declare which versions they accept. A loop that has been updated to v2 can declare that it accepts both v1 and v2. A translation function converts v1 packets to v2 format (adding default values for new fields). When all producers have been updated to v2 and all consumers no longer need v1 support, the v1 contract is retired.

const CONTRACT_ORDER_CREATED = {
  versions: {
    1: {
      fields: { orderId: 'string', customerId: 'string',
                status: 'string', totalPrice: 'number' }
    },
    2: {
      fields: { orderId: 'string', customerId: 'string',
                status: 'string', totalPrice: 'number',
                rushProcessing: 'boolean' },
      migrateFrom: {
        1: (v1) => ({ ...v1, rushProcessing: false })
      }
    }
  },
  current: 2,
  deprecated: [1],   // still accepted, migration available
  retired: []        // no longer accepted by the bus
};

The deprecated list is a warning — packets with this version are accepted but trigger a CONTRACT_DEPRECATED event on the Signal Bus so the operator knows that old-version producers still exist. The retired list is a hard stop — packets with this version are rejected by the bus. This gives the operator a clear migration path: deploy v2, watch for deprecation warnings, fix the remaining v1 producers, then retire v1.

Schema Migration

Vault Loops handle database schema evolution through declared migration scripts. Each migration is a named, ordered, idempotent transformation. The Vault knows what version its data is at (stored as a metadata record) and what version the code expects. The gap is bridged automatically on startup.

const MIGRATIONS = {
  orders: [
    {
      version: 2,
      name: 'add_rush_processing',
      up: async (db) => {
        await db.rpc('add_column_if_not_exists', {
          table: 'orders',
          column: 'rush_processing',
          type: 'boolean',
          default_val: 'false'
        });
      },
      down: async (db) => {
        await db.rpc('drop_column_if_exists', {
          table: 'orders',
          column: 'rush_processing'
        });
      }
    }
  ]
};

Each migration has an up (apply) and down (revert) function. Migrations are idempotent — running up twice produces the same result as running it once (the if_not_exists guard). The Vault Loop runs pending migrations before accepting any bus traffic on startup. If a migration fails, the Vault Loop emits MIGRATION_FAILED and refuses to start — better to surface the problem than to operate against a mismatched schema.

This means deploying a new version of the constellation is: update the code, restart, and the Vault Loops automatically migrate the schema to match. No manual database scripts. No "did someone remember to run the migration?" The migration is part of the code, runs on startup, and is verified by the self-verification system.

Dependency Sunset Protocol

Every adopted dependency will eventually die. The service shuts down, changes its API, gets acquired, raises prices, or becomes a security liability. The Dependency Sunset Protocol is a planned transition process, not an emergency scramble.

Each dependency in the Constellation Spec declares a sunset profile:

const DEPENDENCY_SUPABASE = {
  name:       'supabase',
  type:       'persistence',
  usedBy:     ['vault:customers', 'vault:orders', 'vault:media'],
  isolatedBy: 'Dependency Gate',
  replacementStrategy: {
    alternatives: ['firebase', 'planetscale', 'local-sqlite'],
    migrationScope: 'Vault Loop internals only',
    busInterfaceChange: false,
    estimatedEffort: 'One Vault Loop rewrite per table. Bus contracts, 
                      routing table, pipeline specs, Compute Loops, 
                      Filter Loops, and Presentation Loop: zero changes.'
  }
};

The sunset profile makes the replacement cost visible before it is needed. When Supabase announces a breaking API change, the operator consults the sunset profile and knows immediately: which loops are affected (only the Vault Loops), what the alternatives are, and that the rest of the constellation is untouched. The Dependency Gate's buffered mode provides a grace period during the migration — the old service is dying but the system continues operating on its local buffer while the Vault Loop internals are rewritten for the new provider.

The sunset profile is reviewed periodically (the Clock can emit a reminder) to ensure it stays current. If a new alternative has emerged or the effort estimate has changed, the profile is updated. This is preventive maintenance — the same principle as checking the expiration date on a fire extinguisher.

The Immortal Constellation

A constellation built to the full Loop MMT standard has the following properties:

It can explain itself. The Constellation Spec, the routing table, the pipeline specs, the contract registry, the capability injection table, the retention policies, the role matrix, the schedule table, the dependency sunset profiles — together, these are a complete, readable description of what the system is and how it works. A new developer (human or AI) reading these documents can understand the system without access to the people who built it. The L21 production code paper calls this "cold-read comprehension." It is the single most important property for longevity.

It can verify itself. Startup verification and runtime integrity checks compare the running system against the spec. Drift is detected automatically. The operator does not need to manually audit the system — the system audits itself and reports discrepancies.

It can evolve safely. Contract versioning allows packet types to change without breaking existing consumers. Schema migrations run automatically on startup. The constellation version number in the spec tracks the overall system evolution. Each version bump is a deliberate, documented change — not an accumulation of invisible drift.

It can survive its dependencies. Every adopted dependency is isolated behind a loop, wrapped in a gate, and documented with a sunset profile. When a dependency dies, the blast radius is contained to one loop. The replacement path is known in advance.

It cleans up after itself. Data Gates retire consumed bus traffic, archive old records, and purge expired data on declared schedules. The system does not accumulate unbounded state. Storage costs grow linearly with business activity, not exponentially with time.

It alerts when it is sick. The Operator Alert loop monitors system health events and notifies the operator when something needs attention — before the system degrades to the point where the operator notices symptoms. Dependency gate transitions, workflow failures, integrity check failures, rate limit hits — all surface as actionable alerts.

The Standard
A well-built Loop MMT constellation does not need its original developer to keep running. It does not need a specific AI model or a specific chat history. It needs its spec, its tests, and its code — all of which are permanent, readable artifacts. The system is its own documentation. The tests are its own verification. The Data Gates are its own janitor. The dependency profiles are its own insurance policy. It is designed to be handed to a stranger and understood, maintained, and extended — indefinitely. That is what "just always works" means in practice.

The Observatory

A separate, reusable monitoring constellation that attaches to any Loop MMT application via passive bus observation. Renders live system state: bus traffic, loop health, dependency gate status, workflow timelines, and historical traces. Four internal loops: Telemetry Intake (Working), Metrics Engine (Compute), History Vault (Vault), Observatory UI (Presentation). Subscribes read-only — never emits to the application's buses. Portable across constellations. The same Observatory works with any Loop MMT application because the health packet spec is standardized.

The Observatory also surfaces Operator Alerts visually — paused workflows needing escalation, dependency gates in buffered mode, integrity check failures. Combined with the Operator Alert Signal Loop, this provides two channels: visual (for when the operator is watching) and push notification (for when they are not).

The Constellation Map

The Observatory monitors runtime traffic. The Constellation Map addresses a different problem: design-time comprehension. As a constellation grows from 8 loops to 30 or more, the routing table becomes long enough that the Operator can no longer hold the full topology in their head by reading it as a list. The Constellation Map is a generated visualization of the system's structure — loops as nodes, buses as edges, routing entries as connections — produced from the Constellation Spec.

The Constellation Map is not a live traffic monitor. It is a static architectural diagram that answers: what are all the loops, how are they connected, what packet types flow between them, and where does this new module plug in? It is generated from the spec, which means it is always consistent with the spec and never hand-drawn from memory. The Integrator conversation can use it to visualize cross-module relationships. The Operator can use it to plan Phase 2 conversation groupings. A new developer (human or AI) can use it to understand the system's shape before reading any code.

The Constellation Map is a development tool, not a runtime tool. It belongs alongside the Progress Dashboard and the Action Plan as a planning and comprehension aid. When the Constellation Spec changes, the map is regenerated. It does not require the application to be running.

The Auditor

The Observatory monitors running constellations at runtime. The Auditor is its analog for the development process at build time — a passive reconciliation role that checks cross-document consistency without the power to change anything.

The Auditor is an intermittent conversation — opened for a check cycle, loaded with the current document set, run through a reconciliation check suite, and closed after delivering its report. It does not persist between cycles. Each cycle is a fresh session with no memory of previous cycles. A stateless Auditor cannot drift.

The Auditor's checks are mechanical and binary. They compare two sources that should agree and report whether they do. Three categories: Count Reconciliation (do totals match across documents — module count in dashboard vs. spec, deliverables received vs. verified), Reference Integrity (do identifiers in one document exist in the documents they point to — contract names in loading packs vs. contract registry, module names in dashboard vs. spec), and Sequence Integrity (are ordered events consistent — deliverable timestamps flow forward through the lifecycle, spec changes follow decisions and precede propagation).

The Auditor does not recommend corrective action. It does not assess quality. It does not evaluate whether any role is performing well. It compares documents and reports mismatches. The Operator reads the report and decides what to do. This constraint is what makes the Auditor trustworthy — its output is verifiable by inspection. If the Auditor says there is a mismatch, the Operator looks at the two numbers. They either match or they do not.

The Auditor runs at three kinds of moments: phase gates (mandatory — the project cannot advance until the reconciliation report shows all checks passing or the Operator has explicitly acknowledged each failure), periodic mid-phase checks (recommended — every N deliverables or N hours during Phase 2, calibrated by the Operator), and on-demand (the Operator's instinct says something is not right). Phase-gate audits are not optional.

The Auditor does not communicate with the Coordinator or the Integrator. It receives documents from the Operator and delivers its report to the Operator. This structural isolation prevents the entity being checked from influencing the check.

The Design
A persistent Auditor is tempting because it could watch in real time. But real-time watching requires persistent state, and persistent state requires trust, and trust requires verification — which is the problem the Auditor was created to solve. The intermittent design breaks the recursion. Each session is a clean room. It trusts nothing except the documents it is handed and the checks it was built to run.
Section 15
Scope Boundary

Loop MMT is a data-flow architecture for business applications where the cost of incorrect data exceeds the cost of latency. It handles CRUD operations, multi-step workflows, event-driven communication, real-time sync, continuous data stream intake, hierarchical data propagation, geospatial queries, financial transactions, and multi-tenant access control.

It does not handle real-time signal processing — audio synthesis, video encoding, physics simulation, or any domain where sub-millisecond latency is required and the cost of latency exceeds the cost of correctness. The bus's contract validation and priority queue add overhead that is negligible for business logic but potentially fatal for sample-accurate audio.

The two can coexist in the same application. A browser-based DAW's project management layer (tracks, arrangements, presets, file management) is a Loop MMT constellation. Its audio engine is a direct Web Audio API implementation that bypasses the bus architecture entirely. The boundary is clear: data management goes through the constellation; signal processing does not.

The Rule
A bus that can optionally bypass its own safety guarantees is a bus that will bypass its safety guarantees at the worst possible moment. Loop MMT does not offer a "fast mode" that skips validation. If your traffic cannot tolerate validation overhead, it belongs outside the constellation. The boundary is explicit, not optional.
Performance Budget

The architecture's safety layers — contract validation on every packet, Auth Gate on every action, transaction logging on every workflow step, Observatory tap on every bus transfer — each add overhead. For any single operation the cost is negligible. On hot paths that fire on every user interaction, the costs compound. The Standard does not prescribe specific performance targets because they are application-specific: a real-time dashboard has different latency requirements than a batch reconciliation job.

The Constellation Spec should declare performance budgets for latency-sensitive paths. The Gantry should include a performance verification tier that measures actual overhead against those budgets. The first build of a constellation should measure the real cost of the safety layers so that performance decisions are informed by data, not by assumption. If the measured overhead is unacceptable for a specific path, the answer is to redesign the path — not to bypass the safety layers.

Noted Future Extensions

These are areas where the architecture is well-positioned but a formal pattern has not yet been specified. They are noted here so that when a real project needs them, the design is deliberate rather than improvised. The standard evolves by discovering what is missing when real problems hit real constraints — not by imagining what might be useful.

Tenant Scoping Pattern. The standard handles multi-tenant access control through the Auth Gate (stamps packets with tenantId), scoped subscriptions (Presentation Loops receive only their tenant's data), and RBAC (contextual access rules can include tenant predicates). These are logic-layer controls. What is not yet formalized is a standard pattern for data-layer tenant isolation — ensuring that Tenant A's records are physically or logically separated from Tenant B's at the storage level. For constellations using Postgres (via Supabase or similar), the mechanisms are well-understood: Row Level Security policies, schema-per-tenant, or separate databases. The Vault Loop architecture is well-suited to this — all database access is concentrated in Vault Loops, so adding tenant scoping affects exactly one loop type. The formal pattern — how tenant scoping is declared in the Constellation Spec, how Vault Loops enforce it, how it interacts with the Dependency Gate's buffered mode — will be specified when a real multi-tenant project requires it.

Section 16
Development Methodology

Parallel AI conversations coordinated by a human Operator. The development process is loop-shaped: conversations are loops, documents are the bus, the Operator routes between them. Six conversation types, each with defined responsibilities and structural boundaries, manage the flow of work from specification through deployment. An append-only event ledger records every significant action, making the entire process auditable and replayable.

The Ten Practices: (1) Read before you write. (2) Talk first, code second. (3) Plan in phases, deliver in builds. (4) The Verification Gantry — tests are a fixed gate, not a suggestion. (5) Code as persistence layer — architecture in the code, not the chat. (6) The Operator owns decisions; pushback is a deliverable. (7) Documentation is part of the product. (8) Verify against source, not memory. (9) Design around the memory boundary. (10) This method is not a shortcut.

Parallel model: Phase 0 (spec finalization). Phase 1 (infrastructure: bus, gates, workflow engine, Clock, Auth Gate, initializer with self-verification). Phase 2 (parallel loop development — multiple conversations, each with a loading pack). Phase 3 (integration — assemble, test scenarios, inject failures). Phase 4 (verification — gantry green). Phase 5 (deployment and final archive).

Conversation loading protocol: Each conversation receives: coding standards, bus API reference, relevant slice of routing table and contracts, capability injection spec for its loop type. Not loaded: full routing table, other loops' implementations, full Standard, Action Plan. Context is finite. Give each conversation exactly what it needs.

The Coordinator's Boundaries
The Coordinator is expected to push back when it detects conflicts — routing to a blocked module, spec changes that break dependencies, decisions that contradict earlier decisions. The Operator can always override. The override is always recorded in the event ledger. Pushback is not disobedience; it is the Coordinator doing its job. But the Coordinator never writes code, never verifies code against the spec, and never applies a spec change without explicit Operator approval. These boundaries have the same structural status as the closure wall that constrains loop capabilities.
Session Deliverables and Design Decisions

Each conversation type produces defined deliverables. Specification conversations produce the Constellation Spec and the Action Plan. Module conversations produce code, tests, and handoffs. Technical Design Documents are the expected default output of Specification conversations and Module conversations when design decisions involving non-trivial alternatives are made. A decision has non-trivial alternatives when the rejected options are reasonable — not when the chosen option is complex. The Operator may decline to produce a TDD when the decision is trivial, fully captured by a spec update, or reversible at near-zero cost. The talk-first protocol (Practice 2) is the trigger. If a talk-first conversation surfaces a decision with non-trivial alternatives, it should produce a TDD. The TDD records the reasoning; the Constellation Spec records the result. Both may be produced in the same conversation. When a TDD's decision affects the spec, both must be updated — a decision that lives only in a TDD is a decision the Integrator will never verify against.

The Event Ledger

The event ledger is an append-only chronological record of everything that happens in the project. It is the source of truth. All other project documents — the progress dashboard, the decision log, the session log, the dependency graph — are materialized views derived from the ledger. They are convenient snapshots, but the ledger is canonical.

Nothing in the ledger is edited. If a status was wrong and gets corrected, that is two events — the original and the correction. Both remain. Each event carries a sequential ID, a timestamp, an event type, the session ID that produced it, a payload, and a triggered-by reference tracing the causal chain. The Coordinator produces an updated ledger artifact after every event; if the Coordinator crashes, the most recent artifact is the recovery point.

Document governance rides on the ledger. Every document change is a ledger event. Every dependency relationship is tracked. When a spec change is applied, the Coordinator identifies all downstream documents that need review — loading packs, module status, dependency graph — and flags them automatically. The Document Registry (§17) is a materialized view of the ledger, not a separately maintained artifact. This replaces manual dependency tracking and propagation with structural tracking that cannot be forgotten.

The specific event types, entry formats, compaction strategy, and context management mechanics are defined in the Coordinator Operating Instructions. The standard establishes the principle: the ledger is the source of truth, everything else is a view.

The L21 Parallel
In L21, the session recording captures every tick — every state change, every signal propagation. The current state is just the latest frame. Any prior state is reachable by replaying the log to that point. The event ledger is the same idea applied to project management. The dashboard is not the state. The ledger is the state. The dashboard is a view.
Coordinator–Integrator Interface

The Coordinator and Integrator coordinate through defined document schemas, not direct communication. The Operator carries documents between them. The Integrator produces tagged verification reports; the Coordinator routes findings based on the tags. Each finding carries a category (mechanical, judgment, or architectural) and a confidence level (high or low). The Coordinator routes based on these tags without interpreting the substance. The Integrator maintains its own verification log, independent of the Coordinator's ledger — the overlap at interface events is what the Auditor checks for reconciliation. The report schema and routing rules are defined in the Coordinator Operating Instructions.

Auto-Routing and Decision Queue

The Coordinator can auto-route Integrator findings tagged mechanical with high confidence directly to module conversations. All other findings go to the Operator. The threshold is calibrated per project. Every auto-routed finding is logged. When uncertain, the Coordinator escalates — false escalation costs seconds, false auto-routing costs a module conversation building against wrong instructions.

Decisions requiring the Operator are collected into a priority-ordered queue: P1 (blocking other work), P2 (spec-affecting), P3 (cross-module), P4 (module-scoped). The Coordinator presents decisions at minimal verbosity; the Operator requests more context on any item. The routing matrix, decision queue format, and escalation rules are defined in the Coordinator Operating Instructions.

The Five-Layer Safety Architecture

No single layer catches every failure. Each layer catches a different category at a different point in the process. Together, they provide defense-in-depth for the development process itself.

Layer 1: Coordinator Self-Checks. After every event, the Coordinator runs mechanical verification: index integrity (module counts match), ledger continuity (no sequence gaps), open item consistency (pending counts match unresolved events). Self-checks catch immediate errors but not accumulated cross-document drift.

Layer 2: The Auditor. Intermittent reconciliation comparing documents that should agree. Catches cross-document inconsistencies that accumulate between Coordinator operations — stale loading packs, undocumented spec changes, missing dependency entries, deliverables received but never verified. See §14 for full specification.

Layer 3: The Integrator as Incidental Cross-Check. When a module builds code against a stale loading pack, the resulting code will fail Integrator verification because it implements the wrong contract. The Integrator does not know the loading pack was stale — it knows the code does not match the current spec. The effect is the same: the error is caught indirectly.

Layer 4: Document Fingerprinting. Every document the Coordinator maintains carries a version number and a content fingerprint (hash). If the Coordinator reports "dashboard updated" but the fingerprint did not change, the update did not happen. Fingerprints detect silent failures — operations that were reported as completed but did not actually change the document.

Layer 5: Operator Touch Points. Every spec change requires Operator approval. Every phase transition requires Operator review of the Auditor's reconciliation report. Every circuit breaker trip requires Operator resolution. These are mandatory and cannot be delegated. The Operator also periodically spot-checks the Coordinator's mechanical routing decisions — not to approve them, but to verify the Coordinator's patterns are calibrated correctly.

Defense in Depth
Self-checks catch immediate errors but not accumulated drift. The Auditor catches drift but not quality. The Integrator catches code defects but not process defects. Fingerprints catch silent failures but not semantic errors. The Operator catches everything but cannot look at everything. Remove any layer and a category of failure becomes invisible until it is expensive. The safety architecture does not assume AI is untrustworthy. It assumes that multi-role systems with evolving state will develop inconsistencies, and that catching them early and mechanically is cheaper than catching them late and manually.
The Divergence Circuit Breaker

If a single Auditor check cycle produces failures at or above a defined threshold (default: three), routing pauses — no new deliverables go to the Integrator, no findings are routed, no spec changes propagate, no new conversations start. Module conversations actively building continue; the Coordinator queues incoming events rather than processing them. The Operator reviews the Auditor report, diagnoses the issue, and releases the pause. Below the threshold, failures are reported for Operator review but work continues. The threshold is calibrated per project.

Document rank
The Standard is the highest-ranked document. When any document conflicts with the Standard, the Standard wins. When project documents conflict with each other, the Constellation Spec outranks the Action Plan, which outranks the Loading Packs. Conflicts are resolved by updating the lower-ranked document, not by exception.
Document Registry

The Document Registry is a materialized view of the event ledger showing the current state of every project document: name, version, last updated date, dependencies, and status. The Coordinator maintains it automatically as it processes events — the Operator does not need to update it manually.

Four status values: Current (approved, consistent with dependencies), Review Required (a dependency has changed since last review), Draft (produced but not yet approved), Archived (retired). When a document's dependency changes version, the Coordinator flags the downstream document as Review Required in the ledger. The flag is automatic and cannot be forgotten — it is a property of the event, not a manual step.

Loading packs pin exact document versions. If a pinned document has advanced, the pack is stale and must be updated before use. The Coordinator tracks this through the ledger; the Operator spot-checks.

Version Pinning in Loading Packs

Each loading pack declares the exact version of every document it includes. "Load the Glossary" is not sufficient — the pack must specify which version. When a pinned document advances, the Coordinator flags the pack as stale in the ledger. A session that opens with a stale loading pack will produce work built on shifted ground — the Coordinator and the Integrator (incidentally, through contract mismatches) both catch this.

Version numbers appear in exactly two places — a document's own header and the Document Registry. Prose cross-references use document names and section numbers only. Version numbers are never included in prose cross-references. "See the Interaction Guide §5" — not "see the Interaction Guide v4 §5." The Document Registry is the sole authority on version state. Loading packs are the one exception: they pin specific versions because the session needs to know exactly what it loaded.

Document Authorship

The AI produces content; the Operator reviews and approves it. A version increment equals Operator approval — the AI does not unilaterally advance version numbers. A document the Operator has not reviewed is a Draft. This is logged in the ledger like everything else.

Document Hygiene

When a Coordinator is active, document hygiene is automatic: every document change is a ledger event, dependency propagation is structural, and the registry updates itself. The Operator's role is oversight — spot-checking that the Coordinator's picture matches reality, not manually running a protocol.

When working without a Coordinator (small projects, Phase 0), the AI enumerates document changes at session close, flags downstream dependencies as Review Required, and records the state in the handoff. The Operator confirms completeness. This is the same logic the Coordinator automates — just executed manually.

The Principle
Document governance is the same problem as data governance: track changes, propagate impacts, surface inconsistencies. The event ledger solves both. A document dependency that changed is a ledger event. A stale loading pack is a query against the ledger. The Coordinator makes it structural. Without the Coordinator, the Operator makes it manual. Either way, the discipline is the same.
Stale Documents

A Review Required document must be reviewed or explicitly acknowledged before loading into a conversation. The Operator either updates it (returns to Current) or acknowledges the known risk on the record. There is no silent loading of stale documents — an AI operating on stale information cannot detect that it is stale.
