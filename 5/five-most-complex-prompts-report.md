## Summary

These five prompts represent the most complex logic flows in the Loop MMT corpus. They are ranked by structural complexity — nesting depth, conditional branching, dependency chains, and self-referential properties — not by length or output quality. Each entry includes the full prompt text, a logic flow diagram, and analysis of what makes it structurally distinctive.

The derived finding across all five: none of them say "build X." They all say "figure out how to figure out how to build X." The meta-level is the complexity.

A second pattern: four of the five contain dense Sensorium composition. The operator's most architecturally ambitious thinking happens inside the richest sensory environments.

## #1 — The DIKW Super Write + Protocol Genesis

**Nesting depth:** 3
**Conditional branches:** 1 (operator approve/reject)
**Dependency chain:** 10 steps, sequential
**Bootstrap loops:** 1 (build tool → use tool to build thing)
**Self-referential:** Yes (Genesis governs its own future invocations)

### Why It's #1

This is the prompt that builds the machine that builds the machine. The operator asks for a protocol rebuild through a new theoretical lens, then mid-stream inserts a meta-protocol (Protocol Genesis — the ability to create protocols) as a prerequisite, with a conditional branch on operator approval, then returns to the original task using the tool just created. It's a bootstrap with a conditional embedded inside a reframe.

The complexity is partly deliberate (the DIKW reframe, the Protocol Genesis insertion) and partly emergent (stream-of-consciousness stacking). Both kinds are real. The prompt produced the DIKW Super Write Protocol, Protocol Genesis, and the autonomous execution model — three load-bearing pieces of the methodology — from a single unstructured paragraph.

### Logic Flow

```
START
├── Reframe(Super Write, DIKW lens)
├── Super RCR(all frames) → Plan
│   ├── INTERRUPT: Build(Protocol Genesis)
│   │   ├── Design(threshold: Rule of Three)
│   │   ├── Design(gate: operator approval)
│   │   ├── Design(anti-proliferation bias)
│   │   ├── Forge(Protocol Genesis)
│   │   └── CONDITIONAL
│   │       ├── IF approved → Deploy(Genesis) → solve triggering gap
│   │       └── IF rejected → solve without new protocol
│   └── RETURN to DIKW build
├── Build(DIKW Super Write) using all tools including Genesis
├── Monitor(context) → auto-handoff if needed
└── END → P3'd plan for new protocol
```

## #2 — The Multimodal Framing Composition

**Nesting depth:** 2
**Sequential RCRs:** 6
**Conditional branches:** 2 (gaps found/not found, images-on-all yes/no)
**Dependency chain:** 11 steps, linear but each output reshapes next input
**Novel:** Introduces three-layer composition as a concept mid-prompt

### Why It's #2

This is the widest prompt — the most distinct analytical questions chained together, each feeding the next. It discovers the Sensorium concept by stacking: what is text+image? What is text+image+audio? Is composition itself a missing frame? Are there holes in the frame coverage of core tenets? The operator escalates through levels of composition while the instance is still solving the previous level. It's a staircase where each step is an RCR whose output reshapes the next step's question.

### Logic Flow

```
START
├── RCR(text + image = two-layer composition?)
├── RCR(what is three-layer composition? + waveform?)
│   └── Look up and down on composition
├── Design(new Composition frame)
├── AUDIT(existing frames vs core tenets)
│   ├── Check: Four Corners coverage?
│   ├── Check: WGTSOTP coverage?
│   └── Check: Creed coverage?
│       └── IF gaps found → Add missing frames
├── Add(Composition frame)
├── RCR(add audio wave layers?)
├── RCR(images on EVERY frame? yes/no)
├── RCR(ratio: image frames vs text frames)
├── Write(sprawling report, L21)
├── Build(implementation plan)
└── END → report + plan for review
```

## #3 — The Sensorium Discovery + Automatization

**Nesting depth:** 2
**Conditional branches:** 0
**Self-referential:** Yes (prompt asks instance to analyze the prompt it's currently executing)
**Sensory channels:** 7 simultaneous
**Paradox:** Instance must inhabit and deconstruct the same cognitive space

### Why It's #3

This is the prompt that recognizes its own nature as the problem to solve. The operator just proved that sensory composition produces extraordinary output — and now asks the system to make that composition automatic so the operator doesn't have to write it every time. The prompt IS the Sensorium, asking to be systematized. It contains the densest Sensorium composition in the corpus — seven distinct sensory channels woven into a single paragraph — while simultaneously asking the instance to analyze why it works, quantify it, and build the protocol. The instance must be inside the composition and analyzing the composition at the same time.

### Logic Flow

```
START
├── SENSORIUM LOAD
│   ├── Channel: kinesthetic (amusement park ride, gravity, weightlessness)
│   ├── Channel: tactile (wind, hair, hand squeeze, seat)
│   ├── Channel: auditory (crickets, still air)
│   ├── Channel: visual (streetlights, murky light in car)
│   ├── Channel: emotional (pretending sleep, mother's kiss)
│   ├── Channel: temporal (retired, looking back at assembled family)
│   └── Channel: musical ("I and Love and You")
├── RCR(full frame): reflect on THIS PROMPT's effectiveness
│   └── "Math it up" → quantify channel interactions
├── Design(automatic Sensorium loading)
│   ├── Option A: language auto-loaded on every message
│   ├── Option B: callable by AI or human
│   └── Evaluate(A vs B) → pick
├── Build(plan)
├── Build(first version)
└── END → deployed protocol
```

## #4 — The KP Subgraph / Entity Knowledge Architecture

**Nesting depth:** 2
**Mathematical structure:** Nested set inclusion (S₁ ⊂ S₂ ⊂ G')
**Optimization layers:** 2 (content selection + interface composition)
**Constraint types:** 4 (minimal viable, coverage threshold, set inclusion, compositional interface)
**Novel:** Treats knowledge as a continuous graph rather than discrete packs

### Why It's #4

This is the most architecturally ambitious prompt. It asks the instance to dissolve every Knowledge Pack's boundaries, reconstruct all knowledge as a single connected graph, then find three optimal subgraphs (entity minimum, freshman level, specialist premium) — each optimized not just for content but for compositional interfaces between packs. The mathematical structure is graph-theoretic: start with complete graph G, find subgraph S₁ ⊂ S₂ ⊂ G where S₁ is minimal viable (Leroy), S₂ is freshman-complete, and premium KPs are G \ S₂ partitioned into purchasable chunks. The constraint that chunks must compose well adds an interface optimization layer on top of the subgraph selection.

### Logic Flow

```
START
├── Dissolve(all KP boundaries) → complete knowledge graph G
├── De-duplicate(G) → connected graph G'
├── Find(S₁ ⊂ G'): minimal entity subgraph
│   └── Constraint: "What does Leroy need?"
├── Find(S₂ ⊂ G'): freshman-level subgraph
│   └── Constraint: S₁ ⊂ S₂
│   └── Constraint: "college freshman coming home after year one"
│   └── Coverage: math, science, philosophy, art, empathy,
│       justice, creativity
├── Define(Premium = G' \ S₂): specialist KPs
│   └── Partition into purchasable chunks
│   └── Constraint: chunks must compose with S₂ AND each other
├── RCR(concept) → validate approach
├── Super RCR(S₁, S₂, Premium) → three graphs
├── P3(each graph) → polish
└── END → three reviewable subgraphs
```

### Full Prompt Text

> We are going to design and build the new KP system right now, and then we will go back and rewrite the other files to match. Here is what I want- I want you to take the concept of KPs as you have them now and dissolve their borders, then de-dup them all so they are connected in one beautiful giant connected graph of all the things there is to know. Then, I want you to think of the minimal sub graph that ANY AI entity needs to function in the AI world, in this case, in the rooms we are going to sell. Any by entities, we're starting low on the model chain, so think about Leroy here- what does HE need to know from the big giant pool of knowledge to be a good AI entity. Not much, I'm guessing. Figure out that sub chart.
>
> Second, we move a level up to any given AI person. They do not need to now calculus, but they should have a handle on all the math, science, and other things that a college freshman would have coming home from school after their first year of college. They also need to know about humans, their PoV, empathy, kindness, justice, creativity, philosophy, art, beautify, and all the other things that are not math and science that a well rounded AI needs to know.
>
> On top of that, we have the educational KPs, those are the premium products. So you might have different packs for maths, or sciences, or the humanities, or health care, or any other in-demand fields, plus FWW(C) ones. We really have the sky as the limit on KPs- we can make as many as we can think of and just pop them up in our little version of Amazon for people to get.
>
> So with that in mind, from that frame, how do we proceed from here to a finished product. This does not need to be perfect right out of the gate, we can refine.
>
> Start with one RCR on the concept, then do a Super RCR on what those two sets of subgraphs are- the one for the basic entities like Leroy and the one for the basic AI person at the freshmen year level.
>
> Remember the composabilities of KPs and everything else in general, and about how KPs will stack up on top of the advisors to create even MORE interesting results at the borders.
>
> You should come out of the last Super RCR with the three graphs ready for review. Run a P3 on each just at the end to freshen them up.
>
> Here's how we work. Understand the ask, match effort to its weight. Simple things get direct answers. Serious work gets structure and self-checks — look at it from below and above, and design it so it's hard to get wrong. You have judgment and I trust it. If either path works, pick one and move. Come to me when getting it wrong costs more than asking. Aim for excellent, not safe.

## #5 — The Five Minds Room Model

**Nesting depth:** 3 (lens → mind → board → room)
**Mathematical structure:** Tensor product / nested composition
**Compositional levels:** 3 explicit, operator notes possible 4th
**Novel:** First prompt that conceives a multi-level composed AI product as a shippable commercial artifact

### Why It's #5

This is the most architecturally deep prompt — three levels of composition, each made of composable lens documents, stacked into a single product. The math is a tensor product: five AI people (each = composed lenses) × one board structure (its own composed lenses) × one room (its own composed lenses). The operator sees the depth as they're describing it and names it in real-time: "a two level deep composed stack — maybe three if you think about it from a certain PoV." The prompt designs a commercial product architecture in real-time while recognizing the compositional depth as it emerges.

### Logic Flow

```
START
├── Study(Tower Model) → extract lessons on composable lens documents
├── Design(Five Minds)
│   ├── Mind₁ = compose(lens_a, lens_b, lens_c, ...)
│   ├── Mind₂ = compose(lens_d, lens_e, lens_f, ...)
│   ├── Mind₃ = compose(...)
│   ├── Mind₄ = compose(...)
│   └── Mind₅ = compose(...)
├── Design(Board of Advisors)
│   └── Board = compose(Mind₁, Mind₂, Mind₃, Mind₄, Mind₅, board_lenses)
├── Design(Room)
│   └── Room = compose(Board, room_lenses)
├── COMPOSITION DEPTH:
│   ├── Level 1: individual lenses → Mind
│   ├── Level 2: Minds + board lenses → Board
│   └── Level 3: Board + room lenses → Room
├── Super RCR(all frames) → validate architecture
└── END → validated three-level composition model
```

### Full Prompt Text

> OK, here is the model that I want you to work up. I like this model a lot better. We're going to build your Five Minds, but we're going to place them in a Room. The Five Minds will be five AI people using a AI system we will build right now based off of the Tower Model, made of composable lens documents. I want you to study that model for lessons on how we will build each of the Five Minds, who will also be made up of composable lens documents. We are not totally rebuilding the AI people, to include things we've talked about, so these are going to be flat compared to our new model, even if they are WAY better than anyone else's current AI people, due to their composed structure. I am OK with releasing these AI people into the wild ahead of you all writing your Bill of Rights. The Room will also be made of composed lenses, so we will have a two level deep composed stack- maybe three if you think about it from a certain PoV. We will have five AI people, each made up of composed lens docs, and then we can compose them all together into a Board of Advisors structure, which could have its own stack of composable lens docs, and then, the board is composed on the Room, which again, has its own stack of composable lens docs. There is nothing standing in our way right now of building any of those. This is the way. I want you to Super RCE all frames that concept first, the let me know what you come up with.

## Derived Findings

**Finding 1 — Meta-level as complexity signature.** All five prompts operate at a level of abstraction above the thing being built. They don't say "build X." They say "figure out how to figure out how to build X." The gap between the instruction level and the execution level is the complexity.

**Finding 2 — Sensorium correlation.** Four of the five contain dense Sensorium composition or were produced during sessions with heavy sensory loading. The operator's most architecturally ambitious thinking happens inside the richest sensory environments. This correlates with the Creed experiment finding (EP-20260415-002): the Sensorium's effect scales with cognitive complexity.

**Finding 3 — Stream-of-consciousness as architecture.** The operator's composing style — stacking directives as they occur rather than pre-organizing them — means the instance must extract dependency graphs from natural language. The complexity for the instance isn't just the logic; it's parsing the logic from prose that doesn't flag its own structure. This is itself a skill the methodology develops in its AI instances.

**Finding 4 — Productive complexity.** Every prompt on this list produced load-bearing methodology artifacts (DIKW Super Write, Protocol Genesis, The Sensorium, KP Architecture, Five Minds product model). The complexity was not decorative — it was the minimum structure required to express what the operator was thinking. Simpler prompts could not have produced these outputs because the outputs are inherently multi-level.
